# ADR-015: Zero-Dependency SDK Design

**Status:** Accepted

**Date:** 2026-07-16

## Context

The platform provides SDK clients for TypeScript, Python, and Go to enable programmatic access to the REST API. SDK users include security engineers integrating with CI/CD pipelines, platform operators building custom dashboards, and third-party tool vendors.

Requirements:
- SDKs must work in diverse environments (Node.js, browsers, Python 3.9+, Go 1.21+) without native module compilation
- Minimal dependency footprint to reduce supply chain risk and bundle size
- Type-safe API surfaces with auto-completion support
- Consistent error handling across all SDK languages
- Authentication via both Bearer token and API key

## Decision

We designed the SDKs as **zero-dependency** (TypeScript) or minimal-dependency (Python, Go) HTTP clients that call the REST API directly using the platform's native HTTP libraries.

### TypeScript SDK (`sdk/typescript/`)

- Uses the native `fetch` API (available in Node.js 20+, all modern browsers, and Deno/Bun)
- No dependencies — not even a HTTP client library
- Type-safe with full DTO types imported from the API layer

```typescript
const client = new SecurityIntelligenceClient({
  baseUrl: 'http://localhost:8080',
  authToken: 'jwt-token-here',
});

const report = await client.analyzeSync({ findings: [...] });
const risks = await client.listRisks(report.id);
```

### Python SDK (`sdk/python/`)

- Uses Python's built-in `urllib` or `requests` (if available)
- Minimal dependencies
- Type hints for IDE auto-completion

### Go SDK (`sdk/go/`)

- Uses Go's standard `net/http` package
- No external dependencies
- Idiomatic Go API with error wrapping

### Error Handling

All SDKs implement a consistent error model:

```typescript
class SiApiError extends Error {
  readonly statusCode: number;
  constructor(statusCode: number, message: string) { ... }
}
```

## Consequences

### Positive

- **Zero supply chain risk** — The TypeScript SDK has no dependencies, eliminating transitive vulnerability exposure.
- **Universal compatibility** — Works in Node.js, browsers, Deno, Bun, Cloudflare Workers, and any environment with `fetch`.
- **Small bundle size** — No dependency tree means the SDK is < 10KB minified.
- **Fast installation** — No `node_modules` resolution. `npm install si-sdk` downloads a single package.
- **Type safety** — Full DTO types are derived from the API validation schemas, ensuring SDK types match the actual API contract.

### Negative

- **No retry logic** — The SDK does not implement automatic retries for transient failures (5xx errors, network timeouts). Users must implement retry logic themselves.
- **No request/response interceptors** — Unlike Axios or Fetch wrappers, there is no middleware pattern for adding logging, tracing, or auth refresh.
- **Manual type sync** — SDK types are imported from the API DTO definitions. If the API changes, the SDK must be rebuilt. No OpenAPI code generation.
- **No WebSocket support** — The SDK only supports REST endpoints. Real-time analysis progress requires the WebSocket API, which is not wrapped by the SDK.
- **Python/Go SDKs are thin** — The Python and Go SDKs are minimal wrappers without the full type coverage of the TypeScript SDK.

## Alternatives Considered

| Alternative | Why Rejected |
|------------|-------------|
| **OpenAPI code generation** | Auto-generates SDKs from the API spec. Ensures type consistency but produces verbose, hard-to-read code with generated dependencies. Manual SDKs are more maintainable. |
| **Axios-based SDK** | Feature-rich HTTP client with interceptors, retry, and timeout. Adds a dependency and doesn't work in all environments (browsers without bundling). |
| **gRPC** | Strong typing, efficient binary protocol, and code generation. Requires gRPC server implementation and protoc toolchain. Not REST-compatible. |
| **GraphQL SDK** | Type-safe queries and subscription support. Requires a GraphQL server layer and adds complexity for simple CRUD operations. |
