# ADR-003: 9-Stage Pipeline Architecture

**Status:** Accepted

**Date:** 2026-07-12

## Context

The Security Intelligence Engine must transform raw vulnerability findings from diverse scan sources into actionable intelligence — including correlated findings, risk assessments, attack paths, impact analyses, remediation recommendations, and full explanations. This processing requires multiple sequential transformations, each building on the outputs of prior stages.

Requirements:
- Deterministic, reproducible results suitable for audit
- Clear separation of concerns between analysis stages
- Ability to skip optional stages (attack paths, impact, explanations)
- Cancellation support for long-running analyses
- Progress tracking and event emission per stage
- Measurable per-stage performance metrics

## Decision

We implemented a **9-stage sequential pipeline** in `SecurityIntelligenceEngine`:

```
normalize → correlate → knowledge-graph → risk → attack-path → impact → recommendation → explain → report
```

Each stage is a discrete processing step with defined inputs and outputs:

| Stage | Component | Input | Output |
|-------|-----------|-------|--------|
| 1. Normalize | `NormalizationEngine` | Raw findings | Normalized `SecurityFinding[]` |
| 2. Correlate | `CorrelationEngine` | Findings | Correlations + groups |
| 3. Knowledge Graph | `KnowledgeGraphBuilder` | Findings + correlations | Graph with nodes/edges |
| 4. Risk | `RiskEngine` | Findings + groups | Risk assessments |
| 5. Attack Path | `AttackPathBuilder` | Findings + risks + graph | Attack graphs |
| 6. Impact | `ImpactEngine` | Findings + risks | Impact assessments |
| 7. Recommendation | `RecommendationEngine` | Findings + risks | Recommendations + remediation plan |
| 8. Explain | `ExplainabilityEngine` | Findings + risks | Explanations + analysis trace |
| 9. Report | (aggregation) | All prior outputs | `SecurityIntelligenceReport` |

The pipeline emits events at each stage boundary (`stage:started`, `stage:completed`, `stage:failed`), supports `AbortSignal` for cancellation, and tracks per-stage duration in `PipelineMetrics`.

## Consequences

### Positive

- **Deterministic execution** — Stages run in fixed order with defined data contracts, producing reproducible results.
- **Testability** — Each stage component is independently testable with well-defined input/output types.
- **Observability** — Per-stage progress events enable real-time CLI progress bars and API streaming.
- **Flexibility** — Optional stages (attack-path, impact, explain) can be skipped via `AnalysisOptions`.
- **Metrics** — Per-stage timing enables performance bottleneck identification.

### Negative

- **Sequential bottleneck** — Stages cannot be parallelized because each depends on prior outputs. The distributed pipeline (ADR-009) mitigates this for multi-service deployments.
- **Memory pressure** — All findings, correlations, and intermediate results are held in memory for the pipeline's duration. Large scan results (>10,000 findings) may require streaming approaches.
- **Stage coupling** — Risk engine requires correlation groups; attack path builder requires the knowledge graph. Changing stage interfaces has downstream ripple effects.

## Alternatives Considered

| Alternative | Why Rejected |
|------------|-------------|
| **DAG-based pipeline** | Allows parallel execution of independent stages, but introduces complexity in data synchronization and makes deterministic ordering harder to guarantee. Overkill for the current dependency chain. |
| **Single-pass analysis** | Combining all logic into one pass would be faster but eliminates modularity, testability, and per-stage observability. |
| **Actor model** | Each stage as an actor with message passing. Adds significant infrastructure complexity for marginal benefit in the current sequential model. |
