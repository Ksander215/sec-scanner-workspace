# ADR-001: TypeScript as Primary Language

**Status:** Accepted

**Date:** 2026-07-10

## Context

The Security Intelligence Platform requires a programming language for building a multi-layer system that includes a REST API server, CLI tool, domain analysis engine, and SDK clients. The system must handle concurrent pipeline processing, real-time event streaming, and integrations with diverse external services (LLM providers, cloud APIs, databases, enterprise systems).

Key requirements influencing language choice:
- Type safety for a complex domain model with 18+ node types, 14+ edge types, and 15+ permission types
- Ecosystem support for HTTP frameworks, database drivers, and CLI tooling
- Performance adequate for security analysis workloads (pipeline processing, correlation, risk scoring)
- Developer productivity for rapid iteration across CLI, API, and engine components
- Shared type definitions between server, SDK, and validation layers

## Decision

We selected **TypeScript** (targeting Node.js 20+) as the primary language for the platform.

The entire core platform — including the REST API (`src/api/`), CLI (`src/cli/`), domain engine (`src/domain/`), and infrastructure layer (`src/infrastructure/`) — is implemented in TypeScript. SDK clients for Python (`sdk/python/`) and Go (`sdk/go/`) are provided as thin wrappers over the REST API.

## Consequences

### Positive

- **End-to-end type safety** — Types flow from domain models through API validation (Zod schemas), DTO mappers, persistence layers, and SDK clients. The `Permission`, `Role`, `PipelineStage`, `CorrelationType`, `LlmProviderType`, and `SecretsBackend` types are all compile-time enforced.
- **Single type system** — API request/response types, Zod validation schemas, and persistence types share the same definitions, eliminating serialization mismatches.
- **Rich ecosystem** — Fastify, Zod, Commander, better-sqlite3, ioredis, neo4j-driver, and prom-client provide production-grade implementations for every platform layer.
- **Async-native** — `async/await` and `Promise` are first-class, mapping naturally to the pipeline's sequential stage execution and event-driven architecture.
- **Rapid development** — Single-language stack eliminates context switching between server, CLI, and engine code.

### Negative

- **Single-threaded event loop** — CPU-bound analysis (correlation, risk scoring) blocks the event loop. Mitigated by worker threads for distributed pipeline stages.
- **Type erasure at runtime** — Zod schemas must duplicate type constraints that TypeScript already defines. Mitigated by deriving types from schemas where possible.
- **Larger dependency tree** — Node.js ecosystem tends toward many small packages. Mitigated by strict dependency review and minimal `package.json`.

## Alternatives Considered

| Alternative | Why Rejected |
|------------|-------------|
| **Go** | Strong concurrency, but lacks the ecosystem depth for API frameworks, CLI tooling, and rapid prototyping that Node.js provides. Graph/network libraries are less mature. |
| **Python** | Excellent for ML/AI integration, but weaker type enforcement at runtime, slower for CPU-bound analysis, and GIL limits true parallelism. |
| **Rust** | Maximum performance and safety, but significantly slower development velocity for a platform with 20+ modules. Ecosystem gaps in API frameworks and CLI tooling. |
| **Java/Kotlin** | Mature ecosystem, but heavier operational footprint, slower startup (relevant for CLI), and more ceremony for domain modeling. |
