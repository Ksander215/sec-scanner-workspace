# ADR-009: Circuit Breaker for Distributed Pipeline

**Status:** Accepted

**Date:** 2026-07-14

## Context

When the platform's pipeline stages are distributed across multiple services (normalization service, correlation service, risk service, etc.), any individual service may become unavailable due to network issues, resource exhaustion, or deployment failures. Without protection, a failing service can cascade failures to upstream stages and eventually the entire pipeline.

Requirements:
- Prevent cascade failures when a downstream service is unavailable
- Automatically detect service health degradation
- Provide a recovery mechanism that tests service readiness before resuming traffic
- Allow partial pipeline completion when non-critical stages fail

## Decision

We implemented a **Circuit Breaker pattern** in `DistributedPipeline` (`src/infrastructure/distributed-pipeline/engine.ts`).

### State Machine

```
         failure count >= threshold
CLOSED ─────────────────────────────► OPEN
  ▲                                    │
  │                                    │ reset timeout elapsed
  │       probe success                │
  │  HALF-OPEN ◄──────────────────────┘
  │      │
  │      │ probe failure
  │      └────────────────────────────► OPEN
  └────── (success resets failure count)
```

### Configuration

```typescript
circuitBreaker: {
  failureThreshold: 5,       // failures before opening
  resetTimeoutMs: 30000,     // time before half-open probe
  halfOpenRequests: 1,       // probe requests in half-open state
}
```

### Behavior

- **Closed** — Normal operation. Requests pass through. Failure count tracked.
- **Open** — All requests are skipped with `status: 'skipped'`. Pipeline continues with `overallStatus: 'partial'`.
- **Half-Open** — After `resetTimeoutMs`, one request is allowed through. Success resets the breaker to closed; failure reopens it.

## Consequences

### Positive

- **Cascade prevention** — A failing normalization service won't block the entire distributed pipeline. Subsequent stages are skipped, and the pipeline reports `partial` status.
- **Automatic recovery** — The half-open state allows the system to self-heal without manual intervention.
- **Observability** — Circuit breaker state is included in pipeline results and health checks.
- **Graceful degradation** — The pipeline can produce partial results even when some services are down.

### Negative

- **State in memory** — Circuit breaker state is per-process. In a multi-instance deployment, each instance tracks state independently, which may lead to inconsistent behavior.
- **No request queuing** — When the breaker is open, requests are skipped, not queued. This means data may be lost during outages if no fallback is configured.
- **Threshold tuning** — The `failureThreshold` of 5 may be too aggressive (opening on transient errors) or too lenient (allowing prolonged failures). Requires monitoring and tuning per deployment.

## Alternatives Considered

| Alternative | Why Rejected |
|------------|-------------|
| **Retry only** | Simple retry without a breaker doesn't prevent cascade failures. It may amplify load on a struggling service. |
| **Bulkhead pattern** | Isolates resources per service but doesn't prevent calling a known-failing service. Complementary, not a replacement. |
| **Service mesh (Istio/Linkerd)** | Provides circuit breaking at the infrastructure level but adds operational complexity and doesn't give application-level control over partial pipeline results. |
| **Timeout only** | Prevents hanging but doesn't prevent repeated calls to a failing service. Wastes resources on repeated timeouts. |
