# ADR-005: Builder Pattern for Engine Configuration

**Status:** Accepted

**Date:** 2026-07-12

## Context

The `SecurityIntelligenceEngine` and `PersistenceEngine` both require flexible configuration — custom normalization rules, correlation rules, risk parameters, event handlers, and data directories. Direct constructor parameter injection becomes unwieldy as configuration options grow, and the platform needs to support both programmatic configuration (via SDK/API) and declarative configuration (via config files).

Requirements:
- Fluent API for step-by-step engine configuration
- Optional configuration — all parameters have sensible defaults
- Immutable construction — the engine is fully configured before first use
- Type-safe configuration with compile-time checking

## Decision

We implemented the **Builder Pattern** for both engines:

### SecurityIntelligenceBuilder (`src/domain/security-intelligence/orchestrator/builder.ts`)

```typescript
const engine = new SecurityIntelligenceBuilder()
  .withRiskParameters({ severityWeight: 0.4 })
  .addCorrelationRule(customRule)
  .addNormalizationRule(customNormRule)
  .onEvent((event) => console.log(event))
  .build();
```

### PersistenceBuilder (`src/domain/security-intelligence/persistence/builder.ts`)

```typescript
const persistence = new PersistenceBuilder()
  .withDataDir('/data/si-platform')
  .onEvent((event) => auditLog(event))
  .build();
```

Each builder method returns `this`, enabling method chaining. The `build()` method creates the fully configured engine instance.

## Consequences

### Positive

- **Fluent API** — Method chaining produces readable, declarative configuration code.
- **Defaults** — The `build()` method creates a working engine with zero configuration.
- **Extensibility** — New configuration options are added as builder methods without breaking existing callers.
- **Type safety** — TypeScript enforces correct parameter types at compile time.
- **Testability** — Test fixtures can create minimally configured engines for unit testing.

### Negative

- **No runtime validation** — The builder does not validate parameter combinations until `build()` is called. Invalid configurations (e.g., negative weights) may produce incorrect results silently.
- **Partial application** — There is no "freeze" mechanism to prevent calling `build()` multiple times with different configurations.
- **Indirect construction** — The builder adds an extra object allocation and indirection compared to direct constructor calls.

## Alternatives Considered

| Alternative | Why Rejected |
|------------|-------------|
| **Constructor with config object** | `new Engine({ riskParams, rules, handlers })` — simpler but doesn't enforce optional parameters, and the config object type grows unmanageably. |
| **Factory function** | `createEngine(options)` — equivalent to builder but loses the fluent API and makes conditional configuration less readable. |
| **Dependency injection container** | Full DI (InversifyJS, tsyringe) — adds significant infrastructure complexity and runtime overhead for a platform that prefers explicit, compile-time configuration. |
