# ADR-010: Saga Pattern for Long-Running Pipelines

**Status:** Accepted

**Date:** 2026-07-14

## Context

Long-running analysis pipelines may involve multiple distributed steps, each with side effects (persistence, notification, SIEM forwarding, ticketing creation). If a step fails midway, the system must either:

1. Retry the failed step (with exponential backoff), or
2. Compensate (undo) previously completed steps to maintain consistency

The platform needs a mechanism that:
- Executes multi-step workflows with compensation support
- Tracks saga state across steps (running, completed, compensating, compensated, failed)
- Persists completed saga history for audit
- Supports per-step timeout and retry policies
- Handles tenant context propagation across steps

## Decision

We implemented the **Saga Orchestrator pattern** in `src/infrastructure/saga/engine.ts`.

### Design

Each saga consists of ordered steps, where each step defines:
- `execute(input, context)` — The forward action
- `compensate(output, context)` — The undo action (using the step's output)

If step N fails, the orchestrator compensates steps N-1 through 1 in reverse order.

### State Machine

```
running → completed          (all steps succeed)
running → compensating → compensated  (step fails, compensation succeeds)
running → compensating → failed       (step fails, compensation fails)
running → failed             (unexpected error, compensation fails)
```

### Configuration

```typescript
const config: SagaConfig = {
  maxConcurrent: 10,
  defaultTimeoutMs: 300000,     // 5 minutes per step
  defaultRetryPolicy: {
    maxRetries: 3,
    backoffMs: 1000,
    backoffStrategy: 'exponential',
  },
  persistence: true,            // store completed sagas for audit
};
```

### Usage Example

```typescript
const result = await sagaOrchestrator.execute('full-analysis', [
  { name: 'normalize', execute: ..., compensate: ... },
  { name: 'correlate', execute: ..., compensate: ... },
  { name: 'persist', execute: ..., compensate: ... },
  { name: 'notify', execute: ..., compensate: ... },
], input, { tenantId: 'tenant-123' });
```

## Consequences

### Positive

- **Data consistency** — Compensating actions ensure that partial pipeline executions don't leave the system in an inconsistent state (e.g., a persisted report without corresponding notifications).
- **Audit trail** — Completed sagas are stored with per-step timing, enabling performance analysis and compliance reporting.
- **Retry resilience** — Each step has its own retry policy with exponential backoff, handling transient failures before triggering compensation.
- **Tenant isolation** — Saga context carries `tenantId`, ensuring compensation actions are scoped to the correct tenant.
- **Observability** — `getStatistics()` provides active/completed counts and breakdown by status.

### Negative

- **Compensation complexity** — Writing correct compensation logic is hard. Some actions (e.g., sending a notification) cannot be truly undone. Compensations must be idempotent.
- **Memory usage** — Completed sagas are stored in memory. Long-running services with many sagas may need periodic cleanup.
- **No parallel steps** — Steps execute sequentially. Parallel step execution would require a more complex coordination model.
- **Orchestrator bottleneck** — The orchestrator is a single point of coordination. If it crashes mid-saga, compensation state may be lost.

## Alternatives Considered

| Alternative | Why Rejected |
|------------|-------------|
| **Choreography (event-based)** | Each step publishes an event triggering the next. Simpler but harder to debug, no centralized state, and compensation requires event-based undo chains that are fragile. |
| **Two-Phase Commit (2PC)** | ACID guarantee but requires all participants to hold locks until commit. Unsuitable for long-running operations and external service calls. |
| **Simple retry** | No compensation means partial side effects remain after failure. Not acceptable for enterprise deployments. |
| **Workflow engine (Temporal, Camunda)** | Production-grade but adds significant infrastructure dependency. Suitable for large-scale deployments but overkill for the core platform. |
