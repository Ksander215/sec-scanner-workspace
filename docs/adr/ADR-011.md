# ADR-011: Multi-Provider LLM Router

**Status:** Accepted

**Date:** 2026-07-15

## Context

The platform's AI layer (risk assistant, threat hunting, copilot, remediation) requires LLM capabilities for natural language analysis, threat interpretation, and remediation guidance. Different deployment scenarios have different requirements:

- **Cloud deployments** — Use OpenAI, Azure OpenAI, or Anthropic via API
- **Air-gapped / on-premises** — Use local models via Ollama, vLLM, or GGUF files
- **Hybrid** — Route different tasks to different providers based on cost, latency, and capability requirements

Requirements:
- Abstract away provider-specific APIs behind a common interface
- Support 6+ LLM providers with different authentication and request formats
- Enable runtime provider selection per request
- Track per-provider health, latency, and model availability
- Support chat, completion, and embedding operations

## Decision

We implemented a **Multi-Provider LLM Router** in `src/infrastructure/ai/llm/`.

### Architecture

```
LlmRouter
  ├── OpenAiLlmProvider      (GPT-4o, GPT-4o Mini)
  ├── AzureLlmProvider       (Azure OpenAI deployments)
  ├── AnthropicLlmProvider   (Claude 3.5 Sonnet, Claude 3 Haiku)
  ├── OllamaLlmProvider      (Local models via Ollama)
  ├── VllmLlmProvider        (Self-hosted vLLM inference)
  └── LocalGgufLlmProvider   (Direct GGUF file loading)
```

### LlmProvider Interface

```typescript
interface LlmProvider {
  readonly type: LlmProviderType;
  readonly name: string;
  chat(request: LlmChatRequest): Promise<LlmChatResponse>;
  complete(request: LlmCompletionRequest): Promise<LlmCompletionResponse>;
  embed(text: string): Promise<number[]>;
  health(): Promise<LlmHealth>;
  getModels(): LlmModel[];
}
```

### Router Behavior

- `chat(request, providerType?)` — Routes to the specified provider, or the default if none specified.
- `registerProvider(provider)` — Registers a provider by its type.
- `listProviders()` — Lists all registered providers with availability status.

## Consequences

### Positive

- **Deployment flexibility** — The same platform code works with cloud APIs (OpenAI, Anthropic), enterprise deployments (Azure), and fully air-gapped environments (Ollama, vLLM, local GGUF).
- **Cost optimization** — Different providers have different pricing. The router enables routing cheap tasks (embedding) to cost-effective providers and complex tasks (threat analysis) to high-capability models.
- **Resilience** — If one provider is unavailable, requests can be routed to an alternative (application-level failover).
- **Model transparency** — `getModels()` returns context windows, tool support, and pricing, enabling informed routing decisions.
- **Latency tracking** — Each response includes `latencyMs`, enabling provider performance comparison.

### Negative

- **API abstraction leakage** — Different providers support different features (tool calling, JSON mode, context windows). The common interface represents the lowest common denominator. Provider-specific features are inaccessible.
- **No automatic failover** — The router requires explicit `providerType` selection. It does not automatically fall back on provider failure.
- **Mock implementations** — Current provider implementations return placeholder responses. Production integration requires actual HTTP calls to provider APIs.
- **No streaming** — The `chat()` and `complete()` methods return complete responses, not streams. Real-time use cases (copilot, threat hunting) need streaming support.

## Alternatives Considered

| Alternative | Why Rejected |
|------------|-------------|
| **LangChain** | Popular abstraction but heavy dependency chain, frequent breaking changes, and introduces framework lock-in. Our interface is simpler and more stable. |
| **LiteLLM** | Good provider coverage but adds a Python dependency. Our TypeScript-native approach avoids cross-language overhead. |
| **Single provider (OpenAI)** | Simplest but violates the air-gapped deployment requirement and creates vendor lock-in. |
| **Direct API calls** | No abstraction. Every AI module would implement provider-specific logic, leading to duplication and inconsistency. |
