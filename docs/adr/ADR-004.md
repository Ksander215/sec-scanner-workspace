# ADR-004: Union-Find for Correlation Grouping

**Status:** Accepted

**Date:** 2026-07-12

## Context

The `CorrelationEngine` identifies relationships between security findings through pair-wise rule evaluation. When findings A and B are correlated, and B and C are correlated, they must be grouped together into a single `CorrelationGroup` — even if A and C have no direct correlation. This transitive grouping is essential for risk multiplier calculation (groups with more correlated findings receive higher risk multipliers: `1 + memberCount * 0.1`).

Requirements:
- Efficient grouping of findings into connected components based on correlation relationships
- Support for O(n²) pair-wise correlation (where n = number of findings)
- Deterministic group assignment
- Minimal overhead — correlation grouping should not dominate pipeline runtime

## Decision

We implemented **Union-Find (Disjoint Set Union)** in `CorrelationEngine.buildGroups()`:

```typescript
const parent = new Map<string, string>();
const find = (x: string): string => {
  if (parent.get(x) !== x) parent.set(x, find(parent.get(x)!));
  return parent.get(x) ?? x;
};
const union = (a: string, b: string) => {
  const ra = find(a), rb = find(b);
  if (ra !== rb) parent.set(ra, rb);
};
```

For each finding, the parent is initialized to itself. For each correlation, the two findings are unioned. After processing all correlations, findings sharing the same root belong to the same group. Groups with fewer than 2 members are discarded.

## Consequences

### Positive

- **Near-linear complexity** — With path compression, `find` operations are effectively O(α(n)) where α is the inverse Ackermann function. The overall grouping step is essentially O(n + m) where m is the number of correlations.
- **Transitivity for free** — Union-Find naturally handles transitive relationships without explicitly computing transitive closures.
- **Deterministic** — Given the same input findings and correlations, the grouping output is identical.
- **Simple implementation** — The algorithm is ~15 lines of code with no external dependencies.

### Negative

- **Path compression mutation** — The `find` function mutates the parent map as a side effect, which could be surprising. This is an inherent property of the algorithm.
- **No weighted union** — The current implementation does not use union by rank, meaning the tree could degenerate in pathological cases. Path compression mitigates this in practice.
- **Group naming** — Groups are assigned sequential names (`Correlation Group 1`, `Correlation Group 2`), which are not semantically meaningful. Enrichment would require additional analysis.

## Alternatives Considered

| Alternative | Why Rejected |
|------------|-------------|
| **Graph DFS/BFS** | Compute connected components via graph traversal. Works but requires building an explicit adjacency list and is O(n + m) with higher constant factor and more code. |
| **SQL GROUP BY** | Offload grouping to a database query. Requires persistence layer dependency and is slower for in-memory processing. |
| **Clustering algorithms** | DBSCAN or similar density-based clustering. Overkill for the simple connected-component requirement and adds a dependency. |
