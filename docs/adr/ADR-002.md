# ADR-002: Fastify over Express for REST API

**Status:** Accepted

**Date:** 2026-07-10

## Context

The platform requires an HTTP framework for its REST API that serves analysis requests, report retrieval, finding searches, risk assessments, attack path queries, recommendation management, and explainability endpoints. The API must support:

- Schema-based request validation
- High throughput for concurrent analysis pipelines
- Swagger/OpenAPI documentation generation
- CORS, compression, rate limiting, and WebSocket support
- Plugin architecture for extending endpoints
- Security hardening (headers, payload limits, request ID validation)

## Decision

We selected **Fastify v5** as the HTTP framework, with the following plugin stack:

| Plugin | Purpose |
|--------|---------|
| `@fastify/cors` | Cross-origin request handling |
| `@fastify/compress` | Response compression (gzip/deflate) |
| `@fastify/swagger` + `@fastify/swagger-ui` | Auto-generated API documentation |
| `@fastify/rate-limit` | Request rate limiting |
| `@fastify/websocket` | WebSocket support for streaming analysis progress |

## Consequences

### Positive

- **Performance** — Fastify's routing and serialization are significantly faster than Express. Benchmarks show ~2x throughput for JSON APIs.
- **Schema-based validation** — Native JSON Schema validation on routes eliminates the need for separate validation middleware. Combined with Zod for domain-level validation, this provides defense in depth.
- **Plugin system** — Fastify's encapsulated plugin system (`register`) maps well to our modular API route structure (`src/api/routes/`), where each domain (findings, risk, attack-paths, etc.) is a separate route file.
- **Lifecycle hooks** — `onRequest`, `preHandler`, `onSend` hooks enable security header injection, payload size validation, and request ID sanitization without manual middleware chains.
- **TypeScript support** — First-class TypeScript types for `FastifyInstance`, `FastifyRequest`, and `FastifyReply`.

### Negative

- **Smaller ecosystem** — Fewer community plugins than Express, though all required functionality is covered.
- **Plugin encapsulation** — Fastify's scoped plugin model requires explicit decoration to share state across scopes. Some routes need access to shared services (engine, persistence, auth).
- **Migration cost** — Teams familiar with Express need to learn Fastify's plugin model and lifecycle hooks.

## Alternatives Considered

| Alternative | Why Rejected |
|------------|-------------|
| **Express** | Slower performance, no native schema validation, middleware ordering issues, callback-based error handling. No native OpenAPI generation. |
| **Koa** | Lighter weight, but requires significant middleware assembly. Less ecosystem support for Swagger, rate limiting, and WebSocket. |
| **Hono** | Excellent performance and edge compatibility, but less mature plugin ecosystem and limited WebSocket support at the time of decision. |
| **NestJS** | Full-featured but introduces heavy abstraction (decorators, dependency injection container) that conflicts with our explicit builder pattern for engine configuration. |
