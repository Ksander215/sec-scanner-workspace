# ADR-012: Plugin-Based Extension System

**Status:** Accepted

**Date:** 2026-07-15

## Context

The platform must be extensible without modifying core code. Different deployments require different correlation rules, risk factors, recommendation logic, output formats, and persistence backends. A plugin system enables:

- Third-party extensions for domain-specific analysis
- Custom correlation rules for organization-specific finding patterns
- Additional REST endpoints for integration with internal tools
- Custom output formatters for reporting pipelines
- New persistence backends without core changes

Requirements:
- Well-defined extension points (not a free-for-all)
- Sandboxed plugin execution (no direct access to internals)
- Plugin lifecycle management (load, initialize, destroy)
- Plugin manifest for metadata and capability declaration
- Auto-load support with opt-in (disabled by default for security)

## Decision

We implemented a **Plugin Engine** in `src/infrastructure/plugins/plugin-engine.ts` with 8 extension points:

| Extension Point | Purpose | Example |
|----------------|---------|---------|
| `correlation-rule` | Add custom finding correlation logic | Same-CVE correlation across hosts |
| `risk-factor` | Add custom risk scoring factors | Business criticality weighting |
| `recommendation-rule` | Add custom remediation rules | Organization-specific patching SLAs |
| `cli-command` | Add custom CLI commands | `si compliance check` |
| `rest-endpoint` | Add custom REST endpoints | `/api/v1/custom/export` |
| `persistence-provider` | Add custom storage backends | MongoDB persistence |
| `analysis-stage` | Add custom pipeline stages | Compliance checking stage |
| `output-formatter` | Add custom output formats | SARIF format export |

### Plugin Interface

```typescript
interface SiPlugin {
  readonly manifest: PluginManifest;
  initialize(context: PluginContext): Promise<void>;
  destroy(): Promise<void>;
}
```

### Plugin Context (Sandboxed API)

```typescript
interface PluginContext {
  logger: PluginLogger;
  config: Record<string, unknown>;
  registerCorrelationRule(rule): void;
  registerRiskFactor(factor): void;
  registerRecommendationRule(rule): void;
  registerCliCommand(command): void;
  registerRestEndpoint(endpoint): void;
  registerPersistenceProvider(provider): void;
  registerAnalysisStage(stage): void;
  registerOutputFormatter(formatter): void;
}
```

Plugins receive a controlled context during initialization. They can only interact with the platform through the registered extension points. No direct access to the database, event bus, or other plugins.

## Consequences

### Positive

- **Extensibility without forking** — Organizations can add custom analysis logic without modifying core code.
- **Sandboxed execution** — Plugins cannot access platform internals directly. All interaction is through the `PluginContext` API.
- **Discoverable capabilities** — The manifest declares which extension points a plugin uses, enabling validation before loading.
- **Lifecycle management** — `loadPlugin()` and `unloadPlugin()` enable runtime plugin management.
- **Ordered analysis stages** — `AnalysisStageExtension` includes an `order` field, enabling deterministic stage insertion.

### Negative

- **No type safety for extensions** — Extension point handlers use `Record<string, unknown>` for finding/risk objects rather than the actual domain types. This prevents compile-time type checking within plugins.
- **No dependency between plugins** — Plugins cannot depend on or communicate with other plugins. Complex extensions must be implemented as a single plugin.
- **No hot reload** — Unloading and reloading a plugin creates a new instance. State from the previous instance is lost.
- **Security surface** — Malicious plugins can still execute arbitrary JavaScript (the sandbox is API-level, not process-level). For untrusted plugins, a separate process with IPC is needed.
- **Auto-load risk** — If `plugins.autoLoad: true` is enabled, any plugin in the directory is loaded. This is disabled by default.

## Alternatives Considered

| Alternative | Why Rejected |
|------------|-------------|
| **Microservices** | Each extension as a separate service. Maximum isolation but adds operational complexity (deployment, networking, monitoring). |
| **WebAssembly plugins** | True sandboxing and language agnostism, but the WASM ecosystem for server-side TypeScript is immature, and the performance overhead of boundary crossing is significant for data-heavy extensions. |
| **Hook/callback system** | Simple but lacks the structure for manifest declaration, lifecycle management, and ordered extensions. |
| **No plugin system** | All extensions via core code changes. Violates the open/closed principle and makes the codebase unmaintainable. |
