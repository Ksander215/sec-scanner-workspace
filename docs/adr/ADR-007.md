# ADR-007: Zod for API Validation

**Status:** Accepted

**Date:** 2026-07-13

## Context

The REST API must validate all incoming requests to ensure data integrity, prevent injection attacks, and provide clear error messages. Validation requirements include:

- Request body schema validation for analysis requests, report queries, and search parameters
- Type coercion for query parameters (e.g., string `"100"` → number `100`)
- Range constraints (pagination limits, severity scores)
- Enum validation (severity levels, finding categories)
- Composable schemas for shared structures (pagination, search)
- Integration with Fastify's request lifecycle

## Decision

We selected **Zod v3** as the API validation library, with schemas defined in `src/api/validation/schemas.ts`.

Key schemas:

```typescript
export const SeveritySchema = z.enum(['critical', 'high', 'medium', 'low', 'info', 'none']);
export const FindingCategorySchema = z.enum(['vulnerability', 'misconfiguration', 'exposure', 'secret', 'outdated', 'policy-violation', 'anomaly']);
export const RawFindingSchema = z.object({ ... });
export const AnalyzeRequestSchema = z.object({ findings: z.array(RawFindingSchema).min(1), options: ... });
export const PaginationSchema = z.object({ limit: z.coerce.number().int().min(1).max(1000).default(100), offset: ... });
export const SearchSchema = z.object({ q: z.string().min(1), ... });
```

## Consequences

### Positive

- **Type inference** — `z.infer<typeof AnalyzeRequestSchema>` produces TypeScript types that stay in sync with validation rules. No duplicate type definitions.
- **Rich validation** — `.min()`, `.max()`, `.enum()`, `.coerce()`, `.default()`, `.optional()` cover all validation needs without custom code.
- **Error messages** — Zod produces structured, human-readable validation errors with path information, enabling precise API error responses.
- **Composability** — Schemas compose via `.array()`, `.object()`, and `.union()`, reducing duplication.
- **Runtime + compile-time** — Unlike TypeScript interfaces, Zod schemas enforce constraints at runtime (where attacks happen) while also informing compile-time types.

### Negative

- **Schema duplication** — Some types are defined both as TypeScript interfaces (in `types.ts`) and Zod schemas (in `schemas.ts`). Changes must be synchronized manually.
- **Performance** — Zod validation adds ~1-5ms per request. Acceptable for the platform's throughput but measurable.
- **Bundle size** — Zod adds ~13KB gzipped to the bundle. Relevant for the SDK but not the server.

## Alternatives Considered

| Alternative | Why Rejected |
|------------|-------------|
| **JSON Schema (Fastify native)** | Fastify supports JSON Schema natively, but it lacks type inference, composable schemas, and the ergonomics of Zod's chainable API. |
| **Joi** | Mature and feature-rich, but lacks TypeScript type inference and has a larger bundle size. |
| **class-validator** | Decorator-based validation tied to class definitions. Conflicts with the platform's preference for plain objects and functional patterns. |
| **Yup** | Similar to Zod but slower performance, less maintained, and weaker TypeScript inference. |
