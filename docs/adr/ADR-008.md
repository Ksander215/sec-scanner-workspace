# ADR-008: Event-Driven Architecture with Event Bus

**Status:** Accepted

**Date:** 2026-07-14

## Context

The platform consists of numerous loosely coupled modules (pipeline stages, persistence, audit, analytics, notifications, SIEM integration, threat intelligence) that need to communicate without direct dependencies. Requirements include:

- Decoupled communication between platform modules
- Event replay for debugging and audit
- Consumer group isolation for parallel processing
- Dead-letter handling for failed event processing
- Correlation and causation ID tracking for distributed tracing
- Tenant-aware event routing
- Configurable retry policies with exponential backoff
- Event persistence with retention and size limits

## Decision

We implemented a **custom Event Bus** in `src/infrastructure/event-bus/event-bus.ts`.

### Key Design Decisions

1. **Pub/Sub with pattern matching** — Subscribers match on event types with wildcard support (`pipeline.*` matches `pipeline.started`, `pipeline.completed`, etc.).
2. **Event store** — Events are persisted in-memory with a 100MB limit and 7-day retention. Supports replay by time range, event type, correlation ID, source, and tenant ID.
3. **Consumer groups** — Each subscription has a consumer group for isolation. Statistics track per-group latency and failure rates.
4. **Retry with backoff** — Configurable retry policy with fixed, linear, or exponential backoff and jitter.
5. **Dead-letter handling** — Subscribers can register a `deadLetterHandler` for events that exceed max retries.
6. **Batch publishing** — `publishBatch()` for efficient multi-event publication.

### Event Envelope

```typescript
interface EventEnvelope<T> {
  eventId: string;
  eventType: string;
  version: string;
  timestamp: Date;
  source: string;
  correlationId: string;
  causationId?: string;
  payload: T;
  metadata: {
    tenantId?: string;
    priority: 'low' | 'normal' | 'high' | 'critical';
    persistent: boolean;
    headers: Record<string, string>;
    retryCount: number;
    maxRetries: number;
    traceId?: string;
  };
}
```

## Consequences

### Positive

- **Decoupling** — Modules communicate through events without import dependencies. The pipeline emits events; audit, analytics, and notification modules consume them independently.
- **Observability** — Every event carries correlation and causation IDs, enabling end-to-end trace reconstruction.
- **Resilience** — Retry policies and dead-letter queues prevent transient failures from losing events.
- **Testability** — The in-memory event store enables deterministic testing of event-driven workflows.
- **Replay** — Events can be replayed for debugging, audit, or reprocessing after bug fixes.

### Negative

- **In-memory store** — Events are lost on process restart. For production durability, a persistent event store (Kafka, Redis Streams) is needed as a backend.
- **Sequential delivery** — Events are delivered to subscribers sequentially within a single Node.js process. High-throughput scenarios need the distributed pipeline.
- **No ordering guarantees** — Events from different sources may arrive out of order. Consumers must be idempotent.
- **Memory pressure** — The 100MB event store limit is a rough guard. Precise memory management requires monitoring.

## Alternatives Considered

| Alternative | Why Rejected |
|------------|-------------|
| **Kafka / Redis Streams** | Production-grade durability and scaling, but adds infrastructure dependency. Suitable for enterprise deployment but not for the default/development configuration. |
| **Node.js EventEmitter** | Too simplistic — no persistence, no retry, no consumer groups, no pattern matching. |
| **RxJS** | Powerful reactive streams, but the Observable paradigm is overkill for our fire-and-forget event pattern. |
| **No event bus (direct calls)** | Simple but creates tight coupling between modules. Makes testing and extension difficult. |
