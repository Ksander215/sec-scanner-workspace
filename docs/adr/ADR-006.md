# ADR-006: JSON as Default Persistence Provider

**Status:** Accepted

**Date:** 2026-07-13

## Context

The platform requires a persistence layer to store analysis reports, findings, risk assessments, correlations, attack paths, recommendations, explanations, and snapshots. The persistence system must support:

- Multiple storage backends for different deployment scales (single-user, team, enterprise)
- A repository pattern for each domain entity
- Snapshot creation and restoration
- Event emission for persistence operations
- Zero-configuration startup for development and small deployments

The configuration schema defines five backend options: `json`, `sqlite`, `postgres`, `neo4j`, `redis`.

## Decision

We selected **JSON file-based persistence** as the default provider, implemented in `JsonPersistenceProvider`.

**Storage layout:**

```
data/si-platform/
  reports/
    {reportId}.json       # Full SecurityIntelligenceReport
  snapshots/
    {snapshotId}.json     # Snapshot metadata + base64-encoded report data
```

The `PersistenceProvider` interface defines the contract, with repository implementations for each domain entity (ReportRepository, FindingRepository, RiskRepository, etc.). The `JsonPersistenceProvider` implements this interface using Node.js `fs/promises` for file I/O.

The `PersistenceEngine` wraps the provider, and the `PersistenceBuilder` enables fluent configuration.

## Consequences

### Positive

- **Zero configuration** — Works out of the box with `dataDir: ./data/si-platform`. No database server required.
- **Human-readable** — JSON files can be inspected, diffed, and manually edited for debugging.
- **Portability** — Files can be copied, versioned, and transferred between environments.
- **Simplicity** — No connection pooling, schema migrations, or ORM overhead for the default case.
- **Full report storage** — Each report is stored as a self-contained JSON file, enabling simple snapshot creation (base64 encode the file) and restoration.

### Negative

- **No query optimization** — Finding searches scan all report files linearly. Performance degrades with report count.
- **No concurrent writes** — File-based storage does not support concurrent writes from multiple processes. Single-writer assumption.
- **No ACID transactions** — Write operations are not transactional. A crash during write can corrupt a report file.
- **Storage growth** — Each snapshot duplicates the full report data (base64-encoded). Large deployments need retention policies.

### Migration Path

For deployments that outgrow JSON:

| Backend | When to Use |
|---------|------------|
| `sqlite` | Single-server, better query performance, concurrent reads |
| `postgres` | Multi-server, ACID transactions, connection pooling |
| `neo4j` | Graph queries for knowledge graph traversal |
| `redis` | Caching, session storage, distributed state |

## Alternatives Considered

| Alternative | Why Rejected (as default) |
|------------|--------------------------|
| **SQLite as default** | Better query performance but requires native compilation (`better-sqlite3`), which complicates Docker builds and may fail on some platforms. |
| **In-memory only** | Fastest but no durability. Unsuitable for any production use. |
| **PostgreSQL required** | Too heavy for development/small deployments. Requires external service. |
