# ADR-013: Multi-Tenancy with Row/Schema/Database Isolation

**Status:** Accepted

**Date:** 2026-07-15

## Context

The platform serves multiple organizations (tenants) from a single deployment. Each tenant's data must be isolated from other tenants, and resource usage must be bounded by plan-specific quotas. Requirements include:

- Three levels of data isolation depending on security requirements
- Per-tenant RBAC with tenant-scoped roles and permissions
- Quota enforcement (scans, findings, reports, storage, API calls, concurrent pipelines)
- Plan-based feature gating (free, starter, professional, enterprise)
- Tenant lifecycle management (provisioning, active, suspended, deleting)
- Tenant-specific configuration (SSO, audit logging, encryption, retention)

## Decision

We implemented **three-tier data isolation** in `MultiTenancyEngine` (`src/infrastructure/multi-tenancy/engine.ts`), with the isolation level determined by the tenant's plan:

| Plan | Isolation Level | Description |
|------|----------------|-------------|
| **Free** | `row` | Shared tables with `tenant_id` column filtering. Lowest cost, least isolation. |
| **Starter / Professional** | `schema` | Separate database schema per tenant. Moderate isolation, shared database server. |
| **Enterprise** | `database` | Dedicated database per tenant. Maximum isolation, independent scaling. |

### Tenant Configuration

```typescript
interface TenantConfig {
  dataIsolation: 'schema' | 'database' | 'row';
  encryptionEnabled: boolean;
  auditLogging: boolean;
  ssoEnabled: boolean;
  customBranding: boolean;
  retentionDays: number;
  maxUsers: number;
  features: string[];
}
```

### Quota System

```typescript
interface TenantQuotas {
  maxScans: number;
  maxFindings: number;
  maxReports: number;
  maxStorageMb: number;
  maxApiCalls: number;
  maxConcurrentPipelines: number;
}
```

Enterprise plans have unlimited quotas (`-1`). Free plans are heavily restricted (10 scans, 1000 findings, 100MB storage).

### Per-Tenant RBAC

Each tenant gets default roles (viewer, analyst, admin) with tenant-specific permissions (scan.execute, finding.read, admin.users, admin.billing, etc.).

## Consequences

### Positive

- **Flexible isolation** — Different tenants get different isolation levels based on their security requirements and budget. Row isolation is cost-effective for free users; database isolation satisfies enterprise compliance requirements.
- **Plan-gated features** — The `features` array enables fine-grained feature gating. Free users get `basic-scan`; enterprise users get `*` (all features).
- **Quota enforcement** — Resource limits prevent noisy-neighbor problems and ensure fair resource allocation.
- **Tenant lifecycle** — Clear state machine (provisioning → active → suspended → deleting) enables automated tenant management.

### Negative

- **Row isolation risks** — The `row` isolation level relies on application-level `tenant_id` filtering. A bug in data access code could leak cross-tenant data. Requires rigorous testing.
- **Schema migration complexity** — `schema` isolation requires running migrations per-tenant, which complicates deployment and rollback.
- **Database isolation cost** — `database` isolation requires a separate database instance per enterprise tenant, increasing infrastructure cost and operational complexity.
- **In-memory tenant state** — Tenant data, roles, and users are stored in memory. For production use, this must be backed by a persistent store.
- **No tenant-specific persistence** — The current `JsonPersistenceProvider` does not implement tenant isolation. A tenant-aware persistence provider is needed.

## Alternatives Considered

| Alternative | Why Rejected |
|------------|-------------|
| **Single isolation level (database)** | Maximum security but too expensive for free/starter tenants. |
| **Single isolation level (row)** | Cheapest but insufficient for enterprise compliance requirements. |
| **Namespace isolation** | Kubernetes namespace-per-tenant. Maximum isolation but extreme operational complexity and cost. |
| **No multi-tenancy** | Separate deployment per tenant. Eliminates data isolation concerns but multiplies operational burden. |
