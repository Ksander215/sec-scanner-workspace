# Архитектура SIP — Security Intelligence Platform

## Обзор

SIP представляет собой распределённую платформу безопасности с модульной архитектурой, построенную по принципам микросервисного дизайна и ориентированную на горизонтальное масштабирование. Платформа спроектирована с учётом требований Enterprise-заказчиков: отказоустойчивость, изолированность данных, расширяемость через систему плагинов и поддержка многоарендного (multi-tenant) режима. Каждая компонента системы может быть развёрнута и масштабирована независимо, что обеспечивает гибкость при адаптации под различные размеры организации — от стартапа до крупной корпорации с тысячами хостов.

## Стек фронтенда

Фронтенд реализован на базе **Next.js 15** с использованием **React 19** и **TypeScript** в строгом режиме. Выбор Next.js обусловлен поддержкой серверного рендеринга (SSR), статической генерации (SSG) и инкрементальной регенерации (ISR), что обеспечивает оптимальную производительность как для посадочной страницы, так и для динамических дашбордов. Стилизация реализована через **Tailwind CSS 4** с собственной дизайн-системой на базе CSS-переменных, обеспечивающей тёмную и светлую темы, а также согласованность визуальных компонентов.

Состояние приложения управляется через комбинацию **Zustand** (глобальное состояние), **TanStack Query** (серверное состояние, кэширование и синхронизация) и **React Context** (локализованные данные — i18n, тема, пользовательские предпочтения). Компоненты UI построены на библиотеке **shadcn/ui**, расширенной кастомными компонентами для визуализации данных безопасности: графики уязвимостей (Recharts), интерактивные графы знаний (D3.js + WebGL для рендеринга больших графов), карты атак и таймлайны инцидентов.

Маршрутизация использует файловую систему Next.js App Router с динамическими сегментами для локализации (`/ru/dashboard`, `/en/dashboard`). Локализация реализована через кастомный провайдер `I18nProvider` с поддержкой плюрализации, интерполяции и ленивой загрузки словарей.

## Стек бэкенда

Бэкенд построен на **Node.js 20 LTS** с фреймворком **Express.js** и **TypeScript**. Архитектура следует паттерну Controller-Service-Repository: контроллеры обрабатывают HTTP-запросы и валидацию, сервисы содержат бизнес-логику, репозитории инкапсулируют доступ к данным. Все слои связаны через dependency injection с использованием кастомного DI-контейнера.

В качестве основной базы данных используется **PostgreSQL 16** с расширением **PostGIS** для геопространственных данных. ORM-слой реализован на **Prisma** с миграциями и генерацией типов TypeScript. Для кэширования и хранения сессий применяется **Redis 7**. Очередь задач сканирования реализована на **BullMQ** с поддержкой приоритизации, повторных попыток и dead-letter очередей.

Аутентификация построена на **JWT** (access + refresh tokens) с поддержкой OAuth 2.0 для SSO-интеграции. Авторизация реализована через middleware с ролевой моделью (RBAC). Каждое API-сообщение валидируется через **Zod**-схемы с автоматической генерацией OpenAPI-документации.

Real-time уведомления и обновления статуса сканирования доставляются через **WebSocket** (библиотека `ws`) с fallback на Server-Sent Events (SSE) для клиентов за строгими прокси.

## Система плагинов

Архитектура плагинов — один из ключевых конструктивных элементов SIP, обеспечивающий расширяемость без изменения ядра. Каждый плагин описывается манифестом (`sip.plugin.json`) с метаданными, декларацией точек расширения (extension points) и требованиями к правам доступа. Система поддерживает следующие типы расширений:

- **Scanner Plugins** — добавляют новые типы сканирования (например, сканер Kubernetes, облачный аудит AWS).
- **Parser Plugins** — парсеры отчётов сторонних инструментов (Burp Suite, Nessus, Qualys).
- **Report Plugins** — генераторы отчётов в произвольных форматах.
- **Integration Plugins** — коннекторы к внешним системам (Jira, Slack, ServiceNow).
- **Visualization Plugins** — кастомные виджеты для дашборда.

Плагины выполняются в изолированном sandbox-окружении (VM2 / isolated-vm) с ограниченным доступом к файловой системе, сети и API ядра. Права плагина определяются декларацией в манифесте и утверждаются пользователем при установке. Хотрелоад плагинов обеспечивает обновление без перезапуска сервера.

Регистрация расширений происходит через `PluginRegistry` — централизованный реестр, управляющий жизненным циклом плагинов, хуками и межплагинным взаимодействием. Поддерживается система хуков `beforeScan`, `afterScan`, `onFinding`, `onReport`, позволяющая плагинам реагировать на события ядра.

## Поток данных

Типичный поток данных при запуске сканирования выглядит следующим образом:

1. **Инициация**: пользователь создаёт задачу сканирования через UI или API, указывая цели, тип сканера и параметры. Запрос проходит через API Gateway, валидируется и помещается в очередь BullMQ.

2. **Оркестрация**: Scanner Service извлекает задачу из очереди, определяет подходящий сканер (встроенный или плагин), подготавливает окружение и запускает процесс сканирования. Статус задачи обновляется в реальном времени через WebSocket.

3. **Сбор результатов**: по завершении сканирования сырые результаты парсятся соответствующим Parser'ом (встроенным или плагинным), нормализуются в единую модель `Finding` с CVSS-оценкой, привязкой к активу и маппингом на MITRE ATT&CK.

4. **Аналитика**: нормализованные находки поступают в аналитический конвейер — Knowledge Graph обновляет связи, Attack Paths перестраивает графы атак, AI Recommendations формирует персонализированные советы. Каждый модуль работает асинхронно и независимо.

5. **Хранение**: результаты и аналитика сохраняются в PostgreSQL, связи графа — в специализированном хранилище, кэш — в Redis. Все изменения версонируются для обеспечения аудиторского следа.

6. **Уведомление**: система отправляет уведомления через настроенные каналы (WebSocket, email, Slack, Jira) и обновляет дашборд в реальном времени.

## Инфраструктура и деплой

Все компоненты контейнеризированы (Docker) и оркестрируются через Docker Compose для разработки и Kubernetes для production. CI/CD-пайплайн на базе GitHub Actions обеспечивает автоматическую сборку, тестирование, сканирование безопасности и деплой. Мониторинг реализован через Prometheus + Grafana, логирование — ELK Stack (Elasticsearch, Logstash, Kibana). Трассировка запросов — OpenTelemetry + Jaeger.
