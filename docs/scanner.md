# Сканер безопасности SIP

## Обзор

Модуль сканирования является ядром платформы SIP и отвечает за обнаружение уязвимостей, неправильных конфигураций и потенциальных векторов атак в целевой инфраструктуре. Сканер спроектирован как оркестратор, координирующий работу нескольких типов сканеров — встроенных и плагинных. Результаты всех сканеров нормализуются в единую модель данных `Finding`, что обеспечивает консистентность и возможность кросс-анализа независимо от источника обнаружения. Архитектура модуля поддерживает асинхронное выполнение, приоритизацию задач, ограничение параллельности и автоматический повтор при временных сбоях.

## Как работает сканирование

### Жизненный цикл задачи сканирования

1. **Создание задачи** — пользователь указывает целевые хосты/IP-диапазоны, выбирает тип сканера и настраивает параметры (порты, таймауты, интенсивность). Задача валидируется и помещается в очередь `BullMQ` с указанием приоритета.

2. **Оркестрация** — `ScannerService` извлекает задачу из очереди с учётом приоритета и текущей загрузки воркеров. Определяется подходящий сканер (встроенный или плагинный), подготавливается изолированное окружение выполнения.

3. **Исполнение** — сканер запускается в отдельном процессе (child process) с ограничениями по памяти и CPU. Прогресс обновляется в реальном времени через WebSocket-канал. Система отслеживает состояние процесса и автоматически завершает зависшие задачи по таймауту.

4. **Сбор и парсинг** — по завершении сканирования сырые результаты (XML, JSON, текстовый вывод) передаются соответствующему `Parser`у. Парсер нормализует данные в единую модель `Finding` со следующими полями: уникальный идентификатор, CVSS-оценка, тип уязвимости, затронутый актив, описание, рекомендации по устранению, маппинг на MITRE ATT&CK, доказательство (proof of concept), статус (open/confirmed/false_positive/fixed).

5. **Дедупликация** — новые находки сравниваются с существующими в проекте. Если аналогичная уязвимость уже зарегистрирована на том же активе, обновляется её статус и timestamp последнего обнаружения, а не создаётся дубликат. Это обеспечивает чистоту данных и корректную трендовую аналитику.

6. **Аналитический конвейер** — нормализованные находки поступают в систему аналитики: Knowledge Graph обновляет связи, Attack Paths пересчитывает маршруты атак, AI Recommendations формирует персонализированные советы. Все модули работают асинхронно и не блокируют друг друга.

7. **Уведомление** — по завершении всего конвейера система отправляет уведомления через настроенные каналы: WebSocket (реальное время), email (дайджест), Slack/Jira (интеграционные плагины).

## Поддерживаемые инструменты сканирования

### Сетевое сканирование (Network Scanner)

Основано на **Nmap** с дополнительными NSE-скриптами для обнаружения уязвимостей. Поддерживаемые возможности:

- Обнаружение хостов (host discovery) в указанных IP-диапазонах
- Сканирование портов (TCP SYN, TCP Connect, UDP) с настраиваемым диапазоном
- Определение сервисов и версий (service/version detection)
- Определение операционной системы (OS fingerprinting)
- NSE-скрипты для обнаружения известных уязвимостей (vuln category)
- Поддержка кастомных NSE-скриптов через конфигурацию

Результаты Nmap XML парсятся и преобразуются в модель `Finding`. Каждый открытый порт с определённым сервисом становится узлом в Knowledge Graph, а обнаруженные уязвимости — находками с CVSS-оценкой.

### Сканирование веб-приложений (Web Scanner)

Интеграция с **OWASP ZAP** для динамического анализа безопасности веб-приложений. Выполняет:

- Обход (spidering) целевого веб-приложения для построения карты ресурсов
- Активное сканирование на наличие уязвимостей из OWASP Top 10
- Проверка на SQL-инъекции, XSS, CSRF, обход аутентификации
- Анализ заголовков безопасности (CSP, HSTS, X-Frame-Options)
- Поддержка аутентифицированного сканирования через форму входа или токен

### Аудит зависимостей (Dependency Scanner)

Основано на **Trivy** и **npm audit** для анализа зависимостей программного обеспечения:

- Сканирование `package.json`, `package-lock.json`, `yarn.lock` для npm-проектов
- Анализ `requirements.txt`, `Pipfile.lock` для Python-проектов
- Сканирование `go.sum` для Go-проектов
- Проверка известных уязвимостей (CVE) в транзитивных зависимостях
- Рекомендации по обновлению до безопасных версий

### Сканирование контейнеров (Container Scanner)

Использует **Trivy** для анализа Docker-образов и Kubernetes-манифестов:

- Обнаружение уязвимостей в базовых образах (Alpine, Ubuntu, Debian и др.)
- Анализ установленных пакетов и системных библиотек
- Проверка Kubernetes-манифестов на небезопасные конфигурации (привилегированные контейнеры, монтирование docker.sock, запуск от root)
- Интеграция с CI/CD для блокировки сборки образов с критическими уязвимостями

### Кастомные сканеры (Plugin Scanners)

Через систему плагинов можно добавить любой кастомный сканер. Достаточно реализовать интерфейс `ScannerExtension` из SIP SDK и зарегистрировать плагин с типом `scanner`. Примеры доступных плагинных сканеров в Marketplace: Kubernetes CIS Benchmark Scanner, AWS Security Audit, Azure Compliance Check.

## Рабочий процесс (Workflow)

### Однократное сканирование

```bash
# Через API
curl -X POST https://api.sip-platform.dev/v1/scans \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"projectId":"prj_01","type":"network","targets":["192.168.1.0/24"]}'

# Через CLI
sip scan start --project prj_01 --type network --target 192.168.1.0/24
```

### Плановое сканирование

Поддержка расписания через cron-выражения:

```json
{
  "projectId": "prj_01",
  "type": "network",
  "targets": ["192.168.1.0/24"],
  "schedule": "0 2 * * 1",
  "scheduleLabel": "Каждый понедельник в 02:00"
}
```

### Сравнительное сканирование (Delta Scan)

При повторном сканировании того же проекта SIP автоматически формирует дельта-отчёт: новые находки, исправленные уязвимости, изменения статуса. Это позволяет отслеживать прогресс программы безопасности и оценивать эффективность мер по устранению.

### Параллельное сканирование

Поддержка запуска нескольких типов сканеров одновременно для одного проекта (например, network + web + dependency). Результаты агрегируются в единую карту уязвимостей. Конфигурация параллельности задаётся на уровне проекта и может ограничиваться для предотвращения перегрузки целевой инфраструктуры.
