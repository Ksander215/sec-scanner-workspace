# TECHNICAL_DEBT — INT-004 Attack Path Builder

Дата: 2026-07-16
Модуль: `src/domain/security-intelligence/attack-path/`

## Осознанные компромиссы

### TD-001: Risk Assessment → KG Adapter связь через внутренний Map

**Проблема:** `updateKGAdapterWithRisks()` использует внутренний `_riskScoreMap` вместо перестроения `KnowledgeGraphAdapter`. Это означает, что risk scores, переданные через `DiscoveryInput.riskAssessments`, обогащают только новые node ID, но не обновляют уже существующие attack nodes, созданные адаптером.

**Почему:** Перестроение `KnowledgeGraphAdapter` при каждом вызове `discover()` потребовало бы полного пересоздания внутренних структур данных, что негативно сказалось бы на производительности. Текущее решение — компромисс между точностью и скоростью.

**Решение (будущее):** Внедрить Observer-паттерн между RiskEngine и AttackPathEngine для push-обновлений risk scores.

---

### TD-002: ID генерация через Math.random()

**Проблема:** Генерация ID (`generateAttackPathId`, и др.) использует `Date.now()` + `Math.random()`, что не является криптографически стойким (CSPRNG). Для домена безопасности это может быть проблемой, если ID используются в security-sensitive контекстах.

**Почему:** Внутренняя генерация ID не требует криптографической стойкости — ID используются только для идентификации в рамках движка. Внешние системы должны генерировать собственные CSPRNG-идентификаторы.

**Решение (будущее):** Заменить `Math.random()` на `crypto.randomUUID()` или аналогичный CSPRNG.

---

### TD-003: Batch обработка последовательная

**Проблема:** `discoverBatch()` обрабатывает запросы последовательно, а не параллельно. Для enterprise-нагрузок (10K+) это может быть узким местом.

**Почему:** Параллельная обработка потребовала бы управления конкурентным доступом к GraphRepository и гарантию консистентности графа. Текущая реализация проще и детерминирована.

**Решение (будущее):** Внедрить Worker Pool с partitioned graph snapshots.

---

### TD-004: Cumulative Probability предполагает независимость шагов

**Проблема:** `computeCumulativeProbability()` вычисляет кумулятивную вероятность как произведение вероятностей отдельных шагов, предполагая их независимость. В реальных атаках шаги часто коррелированы.

**Почему:** Учёт корреляций между шагами требует значительно более сложной модели (например, Bayesian Network), что нарушает требование детерминированности. Текущая модель — аппроксимация первого порядка.

**Решение (будущее):** Добавить опциональный correlation factor между смежными шагами.

---

### TD-005: Detection Point Logic упрощена

**Проблема:** `identifyDetectionPoints()` определяет точки обнаружения только по двум критериям: `isDetectionPoint` flag и `nodeType === Vulnerability`. В реальных сценариях detection points зависят от развёрнутых security controls, SIEM правил, и других факторов.

**Почему:** Полная модель detection потребовала бы интеграции с external security tools, что выходит за рамки текущего модуля.

**Решение (будущее):** Добавить DetectionProfile — расширяемую модель detection capabilities.

---

### TD-006: `findEdgeTo()` возвращает только первое ребро

**Проблема:** В `PathDiscoveryEngine.buildStepsFromNodes()` метод `findEdgeTo()` возвращает только первое подходящее ребро, отбрасывая параллельные attack vectors между теми же узлами.

**Почему:** Обработка параллельных рёбер значительно усложнила бы структуру AttackStep. Текущая модель фокусируется на одном векторе между узлами.

**Решение (будущее):** Изменить `incomingEdge` на `incomingEdges: readonly AttackEdge[]` для поддержки нескольких параллельных векторов.

---

### TD-007: Ranking weights не валидируются на сумму 1.0

**Проблема:** `RankingConfig` не валидирует, что сумма весов равна 1.0. Это может привести к неожиданным ranking scores, если пользователь укажет некорректные веса.

**Почему:** Добавление валидации потребовало бы логику нормализации весов, что может быть неожиданным поведением. Текущий подход — документировать ожидание, что веса должны суммироваться до 1.0.

**Решение (будущее):** Добавить валидацию в конструктор `PathRankingEngine` с предупреждением (не ошибкой) при отклонении от 1.0.

---

### TD-008: Async API без реальной асинхронности в ranking/simulation

**Проблема:** Методы `rank()`, `simulate()`, `project()` синхронны, но `discover()` асинхронный. Это создаёт несогласованность в API.

**Почему:** Ranking и simulation — чисто вычислительные операции без I/O. Обёртывание их в Promise добавило бы overhead без пользы. Discovery асинхронен из-за GraphRepository I/O.

**Решение (будущее):** Рассмотреть uniform async API для всех методов, если появится need for concurrent execution.
