# TECHNICAL_DEBT_INT-004.5 — Attack Impact Analysis Engine

## Осознанные компромиссы

### TD-01: Захардкоженные факторы митигации
**Severity:** Medium  
**Описание:** Каждый сценарный обработчик возвращает фиксированные значения `riskReductionFactor`, `confidenceFactor`, `exploitabilityFactor` и т.д. Эти значения не вычисляются динамически на основе реальных характеристик finding-ов.  
**Причина:** На данном этапе pipeline ещё нет достаточных данных для точной оценки эффективности каждого типа митигации. Фиксированные значения обеспечивают детерминированность и предсказуемость.  
**Решение:** В INT-005 (Recommendation Engine) добавить механизм калибровки факторов на основе исторических данных об эффективности ремедиаций.

### TD-02: Graph Delta не использует Knowledge Graph напрямую
**Severity:** Low  
**Описание:** `computeGraphDelta` принимает `allNodeIds` и `allEdgeIds` как опциональные параметры, но не подключается к Knowledge Graph для получения полной структуры графа. Анализ ограничен данными из Attack Paths.  
**Причина:** Knowledge Graph API в текущей реализации не предоставляет методов `removeNodes`/`removeEdges` — он read-only. Мутации через Graph Repository потребовали бы создания транзакций.  
**Решение:** При реализации INT-005 добавить Graph Mutation API или отдельный Immutable Graph Snapshot для полноты анализа.

### TD-03: Ограничение на 20 зависимостей в RemoveAsset
**Severity:** Low  
**Описание:** Обработчик `removeAsset` ограничивает список каскадных зависимостей до 20 (`deps.slice(0, 20)`), чтобы избежать экспоненциального роста при больших графах.  
**Причина:** Удаление актива может затронуть сотни узлов, и генерация зависимостей для каждого из них приведёт к O(N²) по памяти.  
**Решение:** Добавить пагинацию или ленивую генерацию зависимостей в будущих итерациях.

### TD-04: Отсутствие параллельной обработки батчей
**Severity:** Low  
**Описание:** `analyzeBatch` обрабатывает сценарии последовательно, без параллелизма.  
**Причина:** Параллелизм потребовал бы Worker Threads или async pipeline, что усложнило бы реализацию без значительного выигрыша при текущих объёмах.  
**Решение:** При переходе к production нагрузкам добавить async batch processing с Worker Pool.

### TD-05: Security Score — линейная формула
**Severity:** Low  
**Описание:** Security Score вычисляется как `(1 - overallRisk) × 100`, что является линейной формулой. В реальности зависимость между риском и security posture нелинейна.  
**Причина:** Линейная формула обеспечивает простоту, детерминированность и прозрачность.  
**Решение:** В INT-007 (Security Score Engine) добавить нелинейную формулу с конфигурируемыми порогами.

### TD-06: Risk Delta не использует существующий Risk Engine для перерасчёта
**Severity:** Medium  
**Описание:** `computeRiskDelta` вычисляет after-risk путём умножения на `(1 - riskReductionFactor)`, а не путём реального перерасчёта через Risk Engine.  
**Причина:** Перерасчёт через Risk Engine потребовал бы модификации canonical findings и повторного запуска всего pipeline, что нарушает принцип "no mutations".  
**Решение:** Добавить отдельный метод `recalculate()` в INT-005, который создаст модифицированную копию findings и запустит Risk Engine заново.
