## 1. Введение

### 1.1 Контекст

Sec Scanner — это развивающаяся DAST SaaS-платформа (Next.js 16 + Prisma + SQLite) с уже реализованными ключевыми доменными модулями:

- **Security State Engine** — чистый доменный слой, вычисляющий Security Score (0-100), Risk Score, Trend, Confidence, SeverityBreakdown. Нулевая связка с фреймворками. Расширяем через `ScoreStrategy` и `TrendStrategy`.
- **Explainability Layer** — детерминированный слой объяснений: причины текущего состояния, анализ изменений, ROI-отсортированные рекомендации, executive summary. Расширяем через `RecommendationStrategy` и `SummaryStrategy`.
- **API Layer** — Next.js API Routes с авторизацией через next-auth session + Bearer API ключи (scans:read, scans:write, teams:read, teams:write, billing:read).
- **Real-time** — SSE для push-уведомлений о завершении сканирования.
- **Collaboration** — Teams с ролями (owner/admin/member), приглашениями.
- **Dashboard** — 6 вкладок: Security (ScoreGauge + Explainability), Scan, History, Teams, API Keys, Billing.

### 1.2 Проблема

Появилась идея добавить Telegram Mini App. Перед реализацией необходимо определить:
- Действительно ли это увеличит ценность продукта?
- Как это вписывается в общую экосистему?
- Какую роль должен играть каждый канал (Web, Telegram Bot, Telegram Mini App, Public API, будущий Mobile App)?
- Что нужно изменить в архитектуре уже сейчас, чтобы все каналы использовали единый доменный слой?

### 1.3 Принцип анализа

Каждый канал оценивается по трём критериям:

1. **Ценность для пользователя** — решает ли канал задачи, которые другие каналы не решают или решают хуже?
2. **Влияние на метрики продукта** — удержание, ежедневное использование, продажи.
3. **Стоимость реализации и поддержки** — разработка, infrastructure, поддержка.

**Ключевой вопрос не «можно ли это сделать?», а «стоит ли это делать именно сейчас, а не что-то другое?»**

### 2.3 Ключевые наблюдения из матрицы

**1. Web — универсальный, но не всегда лучший.**
Web решает 100% сценариев, но для 6 из 22 сценариев (уведомления, быстрый просмотр score, executive summary, подтверждение действий) есть более подходящие каналы.

**2. Telegram Bot имеет чёткую нишу: push-уведомления.**
Сценарии 7-9 (критические алерты, завершение скана, регрессии) — это задачи **доставки**, а не **взаимодействия**. Telegram Bot решает их лучше всего: мгновенный push, не нужно открывать браузер, работает на телефоне.

**3. Telegram Mini App — «быстрый просмотр + быстрые действия».**
Сценарии 2, 5, 6, 18, 19, 21 — это задачи, где нужно **быстро посмотреть** состояние или **быстро подтвердить** действие, не открывая полноценный web-интерфейс. Mini App даёт UI внутри Telegram (кнопки, карточки), чего обычный бот не может.

**4. API — для автоматизации, не для людей.**
Сценарии 1, 10, 15, 16 — это задачи, которые выполняет код, а не человек. API уже частично реализован (scans:read, scans:write, teams:read, teams:write, billing:read), но нужен Public API с документацией.

**5. Mobile App — на данном этапе не нужен.**
Ни один из 22 сценариев не требует **нативных** мобильных возможностей (камера, GPS, push без Telegram, офлайн). Web PWA + Telegram закрывают 100% мобильных потребностей.

### 3.2 Web Application — «Командный центр»

**Роль:** Полноценный интерфейс для всех задач, требующих глубокой работы с данными.

**Ответственность:**

| Область | Обоснование |
|---------|------------|
| Настройка целей (Target management) | Требует сложных форм, валидации, preview — не подходит для мобильных/чат-интерфейсов |
| Запуск сканирования с параметрами | Выбор depth, scope, authentication, excluded paths — слишком много параметров для Telegram |
| Глубокий анализ Findings | Evidence (HTTP requests/responses), code references, remediation steps — нужен широкий экран и возможность копировать |
| Настройка Workspace/Teams | CRUD-операции с ролями, приглашениями, slug-ами — сложные формы |
| Генерация и просмотр отчётов | PDF с графиками, SARIF export, diff-сравнение сканов — тяжёлый рендеринг |
| Управление API ключами | Создание, настройка scopes, просмотр lastUsedAt — админская функция |
| Администрирование (Admin Panel) | CMS, продукты, audit log — внутренняя функция |
| Security Dashboard (полная версия) | ScoreGauge + RiskTrendCards + ExecutiveSummary + TopImpacts + Reasons + Recommendations + ChangeSummary — слишком много данных для мини-интерфейса |

**Что Web НЕ должен делать:** Push-уведомления (для этого есть Telegram и email), быстрые одношаговые действия без контекста.

### 3.3 Telegram Bot — «Сигнальная система»

**Роль:** Мгновенная доставка критических событий. Бот — это канал **односторонней коммуникации** (server → user), не интерактивный интерфейс.

**Ответственность:**

| Событие | Формат | Обоснование |
|---------|--------|------------|
| Критический finding (Critical severity) | Push-уведомление + ссылка на Dashboard | CTO и Founder должны узнать мгновенно, не открывая browser |
| Регрессия (ранее закрытый finding вернулся) | Push-уведомление + ссылка | Это самая опасная ситуация — исправленная уязвимость вернулась |
| Завершение скана | Краткий результат: score + количество findings + ссылка | Developer не должен refresh-ить страницу |
| Падение Security Score > 10 пунктов | Alert + ссылка на Dashboard | Тренд ухудшается — нужно принять меры |
| Еженедельный дайджест (по расписанию) | Executive summary + Top 3 рекомендации | Для CTO/Founder, которые заходят раз в неделю |

**Формат сообщений Bot:**

```
🚨 Критическая уязвимость обнаружена

Цель: api.example.com
Finding: SQL Injection в /api/users/list
CWE: CWE-89 | OWASP: A03:2021
Security Score: 72 → 47 (-25)

👉 Просмотреть детали: https://secscanner.com/dashboard?scan=abc123
```

**Чего Bot НЕ должен делать:** Сложный анализ, настройку параметров, управление командой. Это не интерактивный интерфейс.

**Команды Bot (минимальный набор):**

| Команда | Действие |
|---------|----------|
| `/status [target]` | Показать текущий Security Score, тренд, количество open findings |
| `/last [target]` | Показать результат последнего скана |
| `/critical` | Список всех открытых Critical/High findings |
| `/help` | Список команд и ссылка на Dashboard |
| `/settings` | Ссылка на настройки уведомлений в Web |

**Обоснование минимального набора:** Команды Bot — это «фаст-фуд» интерфейса. Пользователь должен получить ответ за 1-2 секунды, не думая о навигации. Более сложные запросы — это задача Web или Mini App.

### 3.4 Telegram Mini App — «Карточка состояния»

**Роль:** Быстрый просмотр Security State и быстрые действия **без выхода из Telegram**. Это не дубликат Web, а **специализированное окно** для 3-4 задач.

**Ответственность (строго ограниченный набор):**

| Функция | Обоснование включения |
|---------|----------------------|
| Security Score + Trend + Confidence | Самый частый запрос «как у нас дела?» — не нужно открывать browser |
| Топ-3 критических findings (title + severity) | CTO/Founder нужно знать «что горит» за 5 секунд |
| Последний скан: score, количество findings, дата | Быстрый статус-чек |
| Executive Summary (из Explainability Layer) | 1-5 предложений — идеально для маленького экрана |
| Топ-3 рекомендации (из Explainability Layer) | «Что мне делать дальше?» — ответ за 5 секунд |
| Кнопка «Открыть Dashboard» для деталей | Фолбэк на полный интерфейс |

**Чего Mini App НЕ должен делать:**

- Полный список findings с evidence (нужен широкий экран)
- Настройка параметров сканирования
- Управление командой и ролями
- Генерация отчётов
- Администрирование

**Почему Mini App, а не просто Bot-команды?**

Bot-команды (`/status`) возвращают текст. Mini App возвращает **интерактивные карточки** с цветовой индикацией, прогресс-барами, кнопками действий. Для Security Score (0-100 с цветовой кодировкой) визуальное представление в 10 раз информативнее текста «Score: 72, Trend: stable».

### 3.5 Public API — «Интеграционный слой»

**Роль:** Программный доступ для CI/CD pipelines, внешних систем и автоматизации.

**Текущее состояние:**
- API ключи уже реализованы (ssk_ формат, SHA-256 хранение, scopes)
- Middleware поддерживает Authorization: Bearer header
- Существующие scopes: scans:read, scans:write, teams:read, teams:write, billing:read

**Что нужно добавить:**

| Endpoint | Метод | Описание | Обоснование |
|----------|-------|----------|------------|
| `/api/v1/scans` | GET | Список сканов с фильтрами | Уже есть, нужно версионировать |
| `/api/v1/scans` | POST | Запуск скана | Уже есть, нужно версионировать |
| `/api/v1/scans/:id` | GET | Детали скана с findings | Уже есть |
| `/api/v1/scans/:id/pdf` | GET | PDF-отчёт | Уже есть |
| `/api/v1/security-state/:targetId` | GET | Security State (Score, Risk, Trend, Confidence) | **НОВОЕ** — нужен для интеграций |
| `/api/v1/security-state/:targetId/explain` | GET | Explainability Layer result | **НОВОЕ** — нужен для дашбордов клиентов |
| `/api/v1/targets` | GET/POST | Управление целями | **НОВОЕ** — нужно для CI/CD |
| `/api/v1/webhooks` | GET/POST/DELETE | Настройка webhooks | **НОВОЕ** — Slack, Jira, PagerDuty |

**Ключевое требование:** Все новые endpoints должны использовать единый доменный слой (Security State Engine, Explainability Layer) — без дублирования бизнес-логики.

### 3.6 Mobile App — «Отложен»

**Роль:** Нативное мобильное приложение.

**Решение: НЕ реализовывать в ближайшие 12 месяцев.**

**Обоснование:**
1. **Web PWA** закрывает 95% мобильных потребностей (responsive dashboard, touch-friendly UI).
2. **Telegram Bot + Mini App** закрывает оставшиеся 5% (push-уведомления, быстрый просмотр на ходу).
3. **Ни один сценарий** из матрицы (раздел 2.2) не требует нативных возможностей (камера, GPS, Bluetooth, offline-first).
4. **Стоимость поддержки** двух мобильных платформ (iOS + Android) непропорциональна ценности на текущем этапе.
5. **PWA как промежуточный шаг**: если через 12 месяцев данные покажут потребность в native, PWA-код (React) переносится в React Native с минимальными изменениями.

**Условие пересмотра:** Если > 30% активных пользователей заходят через мобильный browser > 3 раз в неделю — рассмотреть React Native обёртку.

### 3.7 Email — «Асинхронный отчёт»

**Роль:** Асинхронная доставка отчётов и алертов. Уже частично реализовано (nodemailer).

| Тип | Частота | Получатели |
|-----|---------|-----------|
| Алерт о критическом finding | По событию | Developer, Security Engineer |
| Отчёт о завершении скана | По событию | Все роли |
| Еженедельный дайджест | Раз в неделю | Team Lead, CTO, Founder |
| Месячный отчёт (PDF) | Раз в месяц | CTO, Founder |

## 5. Архитектурные требования

### 5.1 Проблема

Текущая архитектура имеет доменный слой (Security State Engine, Explainability Layer) с нулевой связкой с фреймворками. Это **правильная** архитектура. Но API routes и UI компоненты напрямую вызывают Prisma и domain modules, создавая тесную связку:

```
Текущее:
  API Route → Prisma → Domain Module → Response

Целевое:
  API Route → Adapter → Domain Module → Response
                  ↑
  Telegram Bot → Adapter → Domain Module → Response
                  ↑
  Mini App   → Adapter → Domain Module → Response
```

### 5.2 Требуемые изменения (без-breaking)

**1. Вынести адаптеры данных в отдельный слой.**

Сейчас данные для Security State Engine собираются прямо в UI (demo-adapter.ts) или в API routes. Нужно создать **port/adapter** слой, который:

```typescript
// src/adapters/types.ts
interface SecurityStatePort {
  getFindings(targetId: string): Promise<Finding[]>;
  getScanSummaries(targetId: string): Promise<ScanSummary[]>;
  getPreviousSnapshots(targetId: string): Promise<SecurityStateSnapshot[]>;
}

// Адаптер для Prisma
class PrismaSecurityStateAdapter implements SecurityStatePort {
  constructor(private db: PrismaClient) {}
  async getFindings(targetId: string) { /* Prisma query */ }
  // ...
}

// Адаптер для API (возвращает JSON, а не Prisma objects)
class ApiSecurityStateAdapter implements SecurityStatePort {
  constructor(private baseUrl: string, private apiKey: string) {}
  async getFindings(targetId: string) {
    const res = await fetch(`${this.baseUrl}/api/v1/targets/${targetId}/findings`, {
      headers: { Authorization: `Bearer ${this.apiKey}` }
    });
    return res.json();
  }
}
```

**Важное замечание:** Это НЕ рефакторинг всего кода. Это **добавление** нового слоя параллельно с существующим кодом. Существующие API routes продолжают работать.

**2. Создать Notification Port.**

```typescript
// src/domain/notifications/types.ts
interface NotificationPort {
  sendCriticalAlert(userId: string, event: CriticalEvent): Promise<void>;
  sendScanCompleted(userId: string, scan: ScanSummary): Promise<void>;
  sendRegressionAlert(userId: string, finding: Finding): Promise<void>;
  sendWeeklyDigest(userId: string, digest: WeeklyDigest): Promise<void>;
}

// Реализация: Email (уже есть через nodemailer)
class EmailNotificationAdapter implements NotificationPort { ... }

// Реализация: Telegram Bot (будущая)
class TelegramNotificationAdapter implements NotificationPort { ... }
```

**3. Добавить API версионирование.**

Не переписывать существующие /api/scans, а добавить /api/v1/ параллельно:

```
/api/scans          — текущие (оставить для обратной совместимости)
/api/v1/scans       — версионированные (новый код)
/api/v1/security-state/:targetId       — новый endpoint
/api/v1/security-state/:targetId/explain — новый endpoint
/api/v1/webhooks                        — новый endpoint
```

**4. Вынести общие TypeScript типы для API response.**

```typescript
// src/api/types.ts
interface ApiResponse<T> {
  data: T;
  meta?: { page: number; perPage: number; total: number };
}

interface SecurityStateResponse {
  targetId: string;
  computedAt: string; // ISO 8601
  securityScore: number;
  riskScore: number;
  trend: TrendDirection;
  confidence: number;
  openFindings: SeverityBreakdown;
  // ...полный SecurityState, сериализованный для JSON
}

interface ExplanationResponse {
  summary: string;
  reasons: Reason[];
  recommendations: Recommendation[];
  topImpacts: TopImpactFactor[];
  improvements: ChangeFactor[];
  regressions: ChangeFactor[];
  scoreChangeReasons: ScoreChangeReason[];
}
```

**Priority Score = (User Value × 0.3) + (Retention × 0.25) + (DAU × 0.2) + (Sales × 0.15) + (Technical Ease × 0.1)**

| Telegram блокирует бота (спам, complaints) | Низкая | Высокое | Правильная настройка rate limiting, opt-in модель (пользователь сам подключает бота), unsubscribe за один клик |

| Пользователи не подключают бота | Средняя | Среднее | Встроенный onboarding в Web Dashboard: «Подключите Telegram Bot для мгновенных алертов» + QR-код |
| Утечка данных через Bot (wrong user получает чужие алерты) | Низкая | Критическое | Строгая верификация Telegram user ID ↔ Sec Scanner user ID, no broadcast |

### 7.2 Риски Telegram Mini App

| Риск | Вероятность | Влияние | Митигация |
|------|------------|---------|-----------|
| Telegram WebView баги (iOS, keyboard, scroll) | Высокая | Среднее | Минимальный UI (карточки, не сложные формы), тестирование на реальных устройствах, fallback на Bot-команды |
| Telegram изменяет Mini App API | Средняя | Высокое | Mini App — это обычный React-бандл. Если Telegram изменит API, перепаковка займёт 1-2 дня. Не хранить бизнес-логику в Mini App. |
| Низкая конверсия «Bot → Mini App» | Средняя | Среднее | Mini App открывается по кнопке в сообщении бота. Если конверсия < 20%, значит ценности нет — не развивать. |
| Дублирование Web Dashboard | Высокая | Среднее | Строго ограничить функционал Mini App (только чтение + быстрые действия). Ни одной формы настройки. |

### 7.3 Риски Public API

| Риск | Вероятность | Влияние | Митигация |
|------|------------|---------|-----------|
| Breaking changes при обновлении | Средняя | Высокое | Версионирование (/api/v1/, /api/v2/), deprecation период 6 месяцев, changelog |

### 7.4 Общие риски многоканальности

## 8. Самопроверка (CTO + PM Review)

### 8.1 Ревью как CTO

**Архитектурная целостность: 8/10**

Сильные стороны:
- Доменный слой (Security State Engine, Explainability Layer) уже спроектирован правильно — чистые функции, нулевая связка. Это делает добавление каналов тривиальным с точки зрения бизнес-логики.
- Adapter pattern — правильное решение для многоканальности. Не нужно переписывать существующий код, нужно только добавить адаптеры.
- Telegram Bot как отдельный Node.js процесс — правильное решение для изоляции.

Слабые стороны:
- SQLite не поддерживает соединения из нескольких процессов. Если Telegram Bot будет отдельным процессом, ему нужен доступ к DB. Решения: (1) Bot общается с Web через HTTP API (не напрямую с DB), (2) миграция на PostgreSQL (ROADMAP_V2 уже планирует это).
- Отсутствие message queue (Redis/RabbitMQ). Текущие SSE-уведомления — in-memory. При нескольких экземплярах или при перезапуске уведомления теряются.

**Рекомендация CTO:** Реализовать Фазу 1 (адаптеры, порты) + Фазу 2 (Telegram Bot). Bot должен общаться с Web через HTTP API, не напрямую с SQLite. Это одновременно решает проблему multi-process DB access и создаёт чистый API contract.

### 8.2 Ревью как Product Manager

**Product-Market Fit каналалов: 7/10**

Сильные стороны:
- Telegram Bot — это feature request #1 для B2B security-продуктов. Enterprise-клиенты ожидают real-time alerts. Реализация Bot за ~40 часов — excellent ROI.
- Public API — это необходимое условие для перехода от «инструмент» к «платформа». Без API нет CI/CD интеграции, а CI/CD интеграция = ежедневное использование = retention.
- Честная оценка Telegram Mini App: 4.3/10 ROI — это ниже порога принятия решений (наш порог: 6.0). Правильное решение — отложить.

Слабые стороны:
- Нет данных о реальных пользователях. Все оценки основаны на анализе сценариев, а не на данных о поведении. Рекомендация: добавить аналитику (mixpanel/posthog) для валидации гипотез.
- Email-канал недооценён в документе. Еженедельный email-дайджест может иметь больший ROI, чем Telegram Bot, потому что: (1) не требует от пользователя установки ничего, (2) email = universal channel, (3) CTO/Founder читают email ежедневно.

**Рекомендация PM:** Перед запуском Telegram Bot реализовать Email Digest (4-8 часов). Это даст данные о том, какие уведомления пользователи реально открывают и кликают.

### 8.3 Ответ на ключевой вопрос

> **«Если бы бюджет был ограничен, какой канал дал бы максимальный эффект в ближайшие 6 месяцев?»**

**Ответ: Telegram Bot + Email Digest.**

**Обоснование:**

1. **Telegram Bot (28-40h)** даёт:
   - Мгновенные push-уведомления о критических событиях — это #1 feature request для security-продуктов
   - Повышение retention: пользователь не «забывает» зайти в dashboard, потому что dashboard приходит к нему
   - Upsell-аргумент для paid-планов: «Real-time Telegram alerts» — звучит enterprise-grade

2. **Email Digest (4-8h)** даёт:
   - Еженедельный автоматический отчёт для CTO/Founder — они не заходят в dashboard сами, но читают email
   - Данные о том, какие метрики пользователи кликают (для принятия решений о развитии)
   - Нулевая стоимость поддержки (email уже работает)

3. **Public API (36-52h)** — второй приоритет, потому что:
   - Ценность огромная (CI/CD интеграция), но
   - Пользователи, которые будут использовать API — это разработчики, и они сами могут использовать Web UI для запуска сканов
   - API становится критически важным только при > 10 active teams

**Итого: максимальный эффект за минимальный бюджет = Telegram Bot (40h) + Email Digest (8h) = 48 часов.**

### 8.4 Честная оценка Telegram Mini App

> **«Если Telegram Mini App не оправдывает вложения на текущем этапе, честно укажи это и предложи более выгодную альтернативу.»**

**Telegram Mini App на текущем этапе НЕ оправдывает вложения. ROI 4.3/10 — ниже порога.**

**Причины:**

1. **Ценностное пересечение с Web.** Mini App показывает те же данные (Security Score, findings, recommendations), что и Web Dashboard, только в урезанном виде. Пользователь, который открывает Mini App, — это пользователь, который мог бы открыть Web Dashboard. Мы не создаём новую ценность, мы перемещаем существующую ценность в другой интерфейс.

2. **Стоимость не пропорциональна пользе.** 80-120 часов на Mini App vs 40 часов на Bot. При этом Bot решает уникальную задачу (push-уведомления), а Mini App — дублирует Web.

3. **Зависимость от платформы.** Telegram Mini Apps — проприетарная технология. В отличие от Web (открытый стандарт) или API (контролируемый нами), мы зависим от решений Telegram. Если Telegram изменит WebView, ограничения, или monetization — мы не сможем ничего сделать.

4. **Нет данных о спросе.** Мы предполагаем, что пользователям нужен «быстрый просмотр score в Telegram». Но мы не знаем это точно. Правильный подход: реализовать Bot, измерить, какие команды/ссылки пользователи кликают чаще всего. Если > 30% кликов приходится на «просмотр score» — это сигнал для Mini App.

**Более выгодная альтернатива: Email Weekly Digest.**

| Критерий | Telegram Mini App | Email Weekly Digest |
|----------|------------------|-------------------|
| Время разработки | 80-120h | 4-8h |
| Покрытие аудитории | Только пользователи Telegram | 100% пользователей (у всех есть email) |
| Уникальная ценность | Нет (дублирует Web) | Да (автоматический отчёт, который приходит сам) |
| CTO/Founder использование | Низкое (не все используют Telegram для work) | Высокое (email — основной рабочий инструмент) |
| Стоимость поддержки | 8-16h/мес | 1-2h/мес |
| Данные для product decisions | Нет | Да (open rate, click rate) |

**Рекомендация:** Заменить Mini App в roadmap на Email Weekly Digest. Если через 3 месяца данные покажут, что пользователи активно кликают на «просмотреть score» из email — рассмотреть Mini App как следующую итерацию.

## 9. Итоговое решение

### 9.1 Роли каналов (финальные)

| Канал | Роль | Статус | ROI |
|-------|------|--------|-----|
| **Web Application** | Командный центр: полный интерфейс для настройки, анализа, отчётов | ✅ Существует, развивается | — |
| **Telegram Bot** | Сигнальная система: push-уведомления о критических событиях + быстрые команды | 🔨 Следующий (Фаза 2) | **7.7/10** |
| **Telegram Mini App** | Отложен: «карточка состояния» без уникальной ценности на текущем этапе | ⏸️ Через 4-6 месяцев (если данные justify) | **4.3/10** |
| **Public API** | Интеграционный слой: CI/CD, автоматизация, внешние системы | 🔨 После Bot (Фаза 3) | **7.8/10** |
| **Mobile App** | Не нужен: Web PWA + Telegram закрывают 100% сценариев | ❌ Не реализовывать | **1.8/10** |
| **Email Digest** | Асинхронный отчёт: еженедельный автоматический дайджест для CTO/Founder | 🔨 Перед Bot (пре-фаза) | **~8/10** |

### 9.2 Архитектурные принципы

1. **Единый доменный слой.** Security State Engine и Explainability Layer — единственный источник бизнес-логики. Все каналы = тонкие адаптеры.
2. **Adapter pattern.** Каждый канал реализует свой адаптер к domain layer. Никакой бизнес-логики в адаптерах.
3. **Additive changes.** Новые каналы добавляются параллельно существующему коду, без breaking changes.
4. **HTTP API как точка интеграции.** Telegram Bot общается с Web через HTTP API, не напрямую с SQLite.
5. **Версионирование.** Все новые endpoints = /api/v1/. Существующие endpoints не меняются.

### 9.3 Ответ на главный вопрос задачи

> **Действительно ли Telegram Mini App увеличит ценность продукта?**

**Нет, на текущем этапе — нет.**

Telegram Mini App решает те же задачи, что и Web Dashboard (просмотр score, findings, recommendations), только в урезанном виде. Это не создаёт новую ценность, а перемещает существующую в другой интерфейс.

**Что действительно увеличит ценность:**

1. **Telegram Bot (push-уведомления)** — это уникальная ценность, которую Web не может предоставить. Мгновенное знание о критическом событии = ускорение реакции = меньше ущерба.
2. **Public API (CI/CD интеграция)** — это уникальная ценность, которая превращает Sec Scanner из ручного инструмента в платформу. Автоматическое сканирование при каждом деплое = непрерывный мониторинг.
3. **Email Weekly Digest** — это уникальная ценность для CTO/Founder, которые не заходят в dashboard сами, но обязаны знать состояние безопасности.

**Telegram Mini App может стать ценным в будущем**, когда:
- Есть данные, подтверждающие спрос на «быстрый просмотр в Telegram»
- Telegram WebView стабилизировался
- Web Dashboard достаточно развит, чтобы Mini App стал удобным «окном» в него, а не его урезанной копией
