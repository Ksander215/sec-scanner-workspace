# PRODUCT_PRINCIPLES.md — Продуктовые принципы

> Каждый принцип обязателен к соблюдению. Нарушение любого из них — баг.

---

## 1. Business First

Интерфейс говорит на языке бизнеса, а не техники.

**Примеры**:
- ❌ «CVE-2024-1234: Buffer overflow in Redis 7.2.1»
- ✅ «На вашем сервере открыт Redis без пароля — любой может прочитать данные»

**Проверка**: Если владелец бизнеса не понимает сообщение без словаря — переписать.

---

## 2. Explain Everything

Каждый элемент интерфейса объясняет себя. Пользователь не должен гуглить, что означает термин, метрика или статус.

**Примеры**:
- Рядом с «Risk Score» — кнопка «Что это?» с объяснением
- Технические термины — всплывающие подсказки (TermTooltip)
- Графики — подписи, а не просто числа

**Проверка**: Если пользователь задал вопрос «а что это значит?» — интерфейс не объяснил.

---

## 3. Zero Anxiety UX

Никаких красных алертов без объяснения. Никаких сообщений об ошибке без плана действий.

**Правила**:
- Критические проблемы показывают с объяснением последствий и шагами решения
- Вместо «Ошибка» → «Есть возможность улучшить»
- Вместо «Опасность» → «Рекомендуем обратить внимание»
- Вместо «Провал» → «Следующий шаг поможет повысить уровень контроля»

**Проверка**: Если сообщение вызывает панику вместо понимания — переписать.

---

## 4. Self-Explaining Platform

Платформа объясняет сама себя. Пользователь не должен обращаться в поддержку, читать документацию или смотреть туториалы.

**Реализация**:
- ContextualHelp на каждой странице
- TermTooltip для технических терминов
- SmartNextStep подсказывает следующее действие
- AIS даёт контекстные советы

**Проверка**: Если новый пользователь не понял, что делать, без подсказок извне — платформа не объясняет себя.

---

## 5. Adaptive Intelligence

Платформа подстраивается под пользователя: его роль, скорость работы, предпочтения.

**Правила**:
- CEO видит бизнес-информацию
- DevOps видит техническую информацию
- Новичок видит объяснения
- Эксперт видит минимум текста
- Быстрый пользователь получает короткие уведомления
- Медленный — длинные

**Проверка**: Если AIS показывает одно и то же всем — не адаптивный.

---

## 6. Event Driven UI

Любое сообщение появляется только после реального события.

**Правила**:
- ❌ «Настройка завершена» при первом открытии страницы
- ✅ Пользователь нажал «Сохранить» → сервер ответил OK → появилось сообщение
- ❌ «Найдено 5 проблем» при загрузке дашборда
- ✅ Сканирование завершилось → показаны результаты

**Проверка**: Если сообщение появилось без действия пользователя или системы — удалить.

---

## 7. Zero Fake State

Запрещается показывать состояния, которые не соответствуют реальности.

**Правила**:
- Нельзя показывать «Настройка завершена», если настройки не производились
- Нельзя показывать прогресс, если действий не было
- Нельзя показывать найденные проблемы, если анализ не запускался
- Demo-данные обязаны иметь пометку Demo (DemoBadge)

**Проверка**: Если состояние отображается без реального основания — это фейк, исправить.

---

## 8. Zero Duplicate Text

Запрещается повторять одну мысль дважды.

**Примеры нарушений**:
- «ТАРИФЫ» + «Прозрачные тарифы» — подзаголовок удаляется
- «ВОЗМОЖНОСТИ» + «Все возможности платформы» — подзаголовок удаляется
- Большой заголовок уже объясняет раздел — маленький удаляется

**Проверка**: Перед коммитом — можно ли убрать строку? Если можно — убрать.

---

## 9. Progressive Disclosure

Не показывать всё сразу. Информация раскрывается по мере необходимости.

**Реализация**:
- Дашборд показывает 3 ключевых метрики, детали — по клику
- Настройки сгруппированы, продвинутые — скрыты
- AIS показывает один совет, остальные — по запросу

**Проверка**: Если пользователь перегружен информацией — раскрыть меньше.

---

## 10. Business Language

Все тексты написаны на языке бизнеса, а не техники.

**Таблица перевода**:

| Технический | Бизнес |
|-------------|--------|
| CVE-2024-1234 | Уязвимость в вашем сервере |
| CVSS 9.8 | Критический риск |
| SQL Injection | Риск утечки данных |
| Open port 6379 | Сервер доступен без пароля |
| Remediation | Что сделать |
| Vulnerability | Проблема |
| Exploit | Способ атаки |
| Attack vector | Путь атаки |

**Проверка**: Если владелец бизнеса не понимает текст без технического бэкграунда — перевести.

---

## 11. Trust Before Features

Доверие важнее функциональности. Лучше честный простой интерфейс, чем красивый с фейковыми данными.

**Правила**:
- Если данных нет — показать «Нет данных», а не выдуманные
- Если функция не работает — не показывать кнопку
- Если анализ не запускался — не показывать результаты
- Честность > Впечатление

**Проверка**: Если пользователь может усомниться в реальности данных — исправить.
