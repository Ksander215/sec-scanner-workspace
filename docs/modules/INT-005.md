# INT-005 — Attack Path Builder

## Overview

The Attack Path Builder discovers potential multi-step attack chains by combining security findings, risk assessments, and — optionally — the knowledge graph. It traces how an adversary might chain vulnerabilities to progress from an initial foothold to a high-value target, producing `AttackPath` objects that enumerate each step along with prerequisites, outcomes, and cumulative risk.

Key responsibilities:

- **Path discovery** — Enumerate viable attack sequences from findings and risks.
- **Graph-aware routing** — Leverage the knowledge graph (when available) to find structurally valid paths between assets.
- **Attack graph construction** — Build a directed `AttackGraph` with `AttackGraphNode` and `AttackGraphEdge` representations.
- **Statistics** — Provide metrics on discovered paths, graph density, and step distributions.

When a knowledge graph is supplied, the chainer uses graph edges to validate that steps are structurally reachable (e.g. a finding on Host A can chain to a finding on Host B only if A communicates with B). Without a graph, paths are inferred purely from finding metadata and risk scores.

---

## Data Flow

```
1.  discoverPaths(findings, risks, graph?)
    a.  Identify entry points — findings with high exposure or on publicly accessible hosts.
    b.  For each entry point, trace forward:
        i.   Use graph traversal (if available) to find reachable subsequent findings.
        ii.  Use metadata heuristics (shared host, service, CVE chains) otherwise.
        iii. Each hop becomes an AttackStep.
    c.  Rank paths by cumulative risk score.
    d.  Build AttackGraph from all discovered paths.
2.  Return AttackPath[], AttackGraph, and AttackGraphStatistics.
```

## Public API

### Class: `AttackPathBuilder`

| Method | Signature | Description |
|--------|-----------|-------------|
| `discoverPaths` | `discoverPaths(findings: SecurityFinding[], risks: RiskAssessment[], graph?: KnowledgeGraph): { paths: AttackPath[]; graph: AttackGraph; statistics: AttackGraphStatistics }` | Discover attack paths and build the attack graph. |

### Types

#### `AttackStep`

```typescript
interface AttackStep {
  id: string;
  findingId: string;
  description: string;
  riskScore: number;
  prerequisites: string[];    // IDs of prior steps required
  outcome: string;            // what the attacker gains at this step
  technique?: string;         // MITRE ATT&CK technique ID
  order: number;              // step position in the path
}
```

#### `AttackPath`

```typescript
interface AttackPath {
  id: string;
  steps: AttackStep[];
  totalRisk: number;          // cumulative risk across all steps
  entryPoint: string;         // first step ID
  target: string;             // last step ID (highest-impact finding)
  length: number;             // number of steps
  feasibility: number;        // 0.0 – 1.0, estimated likelihood of success
}
```

#### `AttackGraph`

```typescript
interface AttackGraph {
  nodes: AttackGraphNode[];
  edges: AttackGraphEdge[];
}
```

#### `AttackGraphNode`

```typescript
interface AttackGraphNode {
  id: string;
  findingId: string;
  label: string;
  riskLevel: RiskLevel;
  riskScore: number;
  properties: Record<string, unknown>;
}
```

#### `AttackGraphEdge`

```typescript
interface AttackGraphEdge {
  id: string;
  source: string;
  target: string;
  type: string;               // e.g. "enables", "prerequisite", "escalation"
  weight: number;
  properties: Record<string, unknown>;
}
```

#### `AttackGraphStatistics`

```typescript
interface AttackGraphStatistics {
  totalPaths: number;
  totalNodes: number;
  totalEdges: number;
  averagePathLength: number;
  maxPathLength: number;
  criticalPaths: number;       // paths containing a Critical risk step
  entryPoints: number;
  techniqueDistribution: Record<string, number>;
  processingTimeMs: number;
}
```

## Extension Points

1. **Custom entry-point criteria** — Override the entry-point finder by subclassing or by pre-filtering findings before calling `discoverPaths()`.
2. **Chaining heuristics** — The step chainer uses a default set of heuristics. Domain-specific chains (e.g. cloud-specific escalation paths) can be enabled by enriching the knowledge graph with custom edges.
3. **Technique mapping** — Attack steps support MITRE ATT&CK technique IDs. Map findings to techniques via the `technique` field.
4. **Path ranking** — Paths are ranked by cumulative risk by default. Custom ranking functions can be applied by sorting the returned `paths` array.

## Examples

### Discovering Attack Paths

```typescript
import { AttackPathBuilder } from './attack-path';

const builder = new AttackPathBuilder();

const result = builder.discoverPaths(normalizedFindings, riskAssessments);

console.log(`Discovered ${result.paths.length} attack paths`);
console.log(`Attack graph: ${result.statistics.totalNodes} nodes, ${result.statistics.totalEdges} edges`);

// Top 5 most dangerous paths
const topPaths = result.paths.slice(0, 5);
for (const path of topPaths) {
  console.log(
    `Path ${path.id}: ${path.length} steps, ` +
    `totalRisk=${path.totalRisk.toFixed(3)}, feasibility=${path.feasibility.toFixed(2)}`
  );
}
```

### With Knowledge Graph

```typescript
// Knowledge graph enables structurally-aware path discovery
const result = builder.discoverPaths(
  normalizedFindings,
  riskAssessments,
  knowledgeGraph   // optional KnowledgeGraph from INT-003
);

// Graph-aware paths are more realistic — they only chain findings
// between hosts/services that actually communicate
console.log(`Critical paths: ${result.statistics.criticalPaths}`);
```

### Inspecting a Path Step-by-Step

```typescript
const path = result.paths[0];
console.log(`Attack path: ${path.entryPoint} → ${path.target}`);
console.log(`Feasibility: ${(path.feasibility * 100).toFixed(1)}%`);
console.log();

for (const step of path.steps) {
  console.log(`  Step ${step.order}: ${step.description}`);
  console.log(`    Finding: ${step.findingId}`);
  console.log(`    Risk: ${step.riskScore.toFixed(3)}`);
  console.log(`    Outcome: ${step.outcome}`);
  if (step.technique) {
    console.log(`    MITRE ATT&CK: ${step.technique}`);
  }
  if (step.prerequisites.length > 0) {
    console.log(`    Prerequisites: ${step.prerequisites.join(", ")}`);
  }
}
```

### Working with the Attack Graph

```typescript
const { graph, statistics } = result;

// Find the most-connected nodes (high-value targets or pivot points)
const nodeDegree = new Map<string, number>();
for (const edge of graph.edges) {
  nodeDegree.set(edge.source, (nodeDegree.get(edge.source) ?? 0) + 1);
  nodeDegree.set(edge.target, (nodeDegree.get(edge.target) ?? 0) + 1);
}

const topNodes = [...nodeDegree.entries()]
  .sort((a, b) => b[1] - a[1])
  .slice(0, 10);

for (const [nodeId, degree] of topNodes) {
  const node = graph.nodes.find(n => n.id === nodeId);
  console.log(`${node?.label} (${node?.riskLevel}): degree=${degree}`);
}
```

## Performance Notes

| Aspect | Detail |
|--------|--------|
| **Time complexity** | O(n² × d) where n = findings and d = max path depth. With knowledge graph: O(V + E) per entry point for graph traversal. |
| **Path explosion** | In highly connected graphs, the number of paths can grow combinatorially. The builder limits path length and prunes low-feasibility paths. |
| **Throughput** | ~5 000 findings in < 3 s without graph. With graph: depends on graph density; sparse graphs are fast, dense graphs require more pruning. |
| **Memory** | Each `AttackPath` stores full step arrays. For large path sets (> 10 000 paths), memory usage can reach ~100 MB. |
| **Graph supply** | Providing a knowledge graph is optional but strongly recommended for accuracy and for constraining the search space. |
