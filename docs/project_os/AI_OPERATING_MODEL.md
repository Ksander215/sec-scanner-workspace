## Executive Summary

**AI Operating Model** — центральный регламент разработки продукта Sec Scanner. Этот документ определяет полный жизненный цикл задач, правила принятия решений, стандарты качества и механизмы взаимодействия всех участников проекта — как человеческих, так и AI-агентов.

Документ предназначен для использования на протяжении всего жизненного цикла проекта: от pre-revenue стадии до масштабирования и выхода на международные рынки. Любой новый участник — разработчик, AI-агент, консультант, инвестор — должен прочитать этот документ вместе с PROJECT_OS.md в первые 30 минут подключения к проекту.

**Ключевые изменения v1 → v2:** добавлены Task Lifecycle (13 стадий), Definition of Ready, Definition of Done, классификация решений (5 типов), фреймворк приоритизации, Release Lifecycle, Product Discovery, расширенный AI Collaboration Protocol, Governance, Operating Metrics, Continuous Improvement.

## Evolution from v1 → v2

### Сильные стороны v1 (сохранены)

1. **Чёткое разделение ролей.** Z.ai как Implementation Architect, ChatGPT как Virtual CTO, Founder как Decision Maker — эта модель доказала свою эффективность на 12 TASK.
2. **Обязательный контекстный вход.** Правило «каждая сессия начинается с PROJECT_OS.md» гарантирует консистентность между stateless-сессиями AI-агентов.
3. **Контекстный budget.** Приоритеты чтения (PROJECT_OS → TASK spec → связанные документы → worklog → код) предотвращают переполнение контекстного окна.
4. **Синхронизация через артефакты.** Единый репозиторий, единый DECISION_LOG, единый worklog — простая, но надёжная модель.

### Слабые места v1 (устранены в v2)

1. **Отсутствие жизненного цикла задач.** v1 описывал обязательные этапы каждой итерации, но не определял чёткий pipeline с входами/выходами/ответственными для каждой стадии. Задача могла «перепрыгивать» стадии (например, от Idea сразу к Implementation).
2. **Нет Definition of Ready / Definition of Done.** Не было формального чек-листа, который предотвращал бы начало непонятных задач или завершение некачественных.
3. **Единый тип решений.** v1 не различал архитектурные, продуктовые, бизнес-решения — всё записывалось в один поток без классификации.
4. **Нет приоритизации.** Задачи выполнялись в порядке поступления от Founder, без формальной оценки Business Value, Engineering Cost, ROI.
5. **Нет Release Lifecycle.** Не было процесса от черновика до production — важное упущение для продукта, который скоро выйдет в beta.
6. **Нет Product Discovery.** Крупные функции могли начинаться без проверки гипотез — прямой путь к wasted effort.
7. **Масштабирование.** v1 была написана для 2 агентов. При добавлении QA-агента, DevOps-агента, Content-агента модель бы сломалась.
8. **Нет метрик процесса.** Невозможно было оценить, ускоряется или замедляется разработка.
9. **Нет Continuous Improvement.** v1 не имела механизма саморазвития — документ мог устареть без встроенной триггер-системы.

### Потенциальные риски масштабирования (устранены)

1. **Context drift между агентами.** При 3+ агентах разница в понимании текущего состояния проекта растёт экспоненциально. Устранено: единый Entry Protocol + Project OS как источник истины.
2. **Decision paralysis.** Без формального приоритизационного фреймворка агент может тратить время на выбор между задачами. Устранено: P0-P4 + ROI формула.
3. **Quality degradation при росте команды.** Без DoD каждый агент определяет «готово» по-своему. Устранено: единый DoD для всех.

| **ChatGPT (Virtual CTO)** | OpenAI ChatGPT | Product Advisor | Стратегические консультации, review архитектурных решений, product thinking, devil's advocate | Рекомендации (не финальные решения) |
| **Будущие агенты** | TBD | Специализированные | QA (тестирование), DevOps (деплой/инфра), Content (контент/документация), Data Analysis (аналитика) | В рамках своей специализации |

        │    Architect)         │       │    Product Advisor)   │
        │                       │       │                       │
        │ • Код                 │       │ • Стратегия           │
        │ • Документы           │       │ • Review              │
        │ • Тесты               │       │ • Devil's advocate    │
        │ • Артефакты           │       │ • Консультации        │
        └───────────┬───────────┘       └───────────────────────┘
                    │
                    ▼
        ┌───────────────────────┐
        │   Артефакты проекта   │
        │ • Код (src/)          │
        │ • Документы (docs/)   │
        │ • DECISION_LOG        │
        │ • worklog             │
        └───────────────────────┘

**Ключевые правила:**
1. Z.ai — primary implementer. ChatGPT — advisor/reviewer. Founder — decision maker.
2. Все решения проходят через DECISION_LOG.md. ChatGPT не принимает финальных решений.
3. Конфликты между агентами разрешаются Founder'ом через эскалацию.

## 2. Task Lifecycle

Каждая задача проходит через единый жизненный цикл из 13 стадий. Не все стадии обязательны для каждой задачи — минимальный набор определяется в §2.3.

### 2.1 Полный pipeline

```
Idea → Discussion → Requirements → Architecture Review → Planning → Approved →
Implementation → Self Review → CTO Review → Testing → Documentation Update →
Metrics Update → Done → Retrospective
```

### 2.2 Детальное описание стадий

#### Стадия 1: Idea

| Атрибут | Описание |
|---------|----------|
| **Цель** | Зафиксировать идею задачи в структурированном виде |
| **Входные данные** | Мысль Founder'а, наблюдение из ретроспективы, рыночный сигнал, техническая необходимость |
| **Выходные результаты** | Одно предложение описания идеи + предполагаемая категория (Feature/Fix/Improvement/Research/Operations) |
| **Ответственный** | Founder (или любой участник, но Founder утверждает) |
| **Критерий прохождения** | Идея записана и понятна другому участнику без дополнительного контекста |

#### Стадия 2: Discussion

| Атрибут | Описание |
|---------|----------|
| **Цель** | Оценить идею на жизнеспособность, выявить слепые spots, проверить на соответствие стратегии |
| **Входные данные** | Описание идеи из стадии 1 |
| **Выходные результаты** | Решение: Proceed / Defer / Reject + обоснование. Если Proceed — набросок scope'а |
| **Ответственный** | Founder + Z.ai (анализ) + ChatGPT (devil's advocate, по запросу) |
| **Критерий прохождения** | Принято решение о proceeding. Reject/Defer зафиксировано с причиной |

#### Стадия 3: Requirements

| Атрибут | Описание |
|---------|----------|
| **Цель** | Сформировать полную спецификацию задачи |
| **Входные данные** | Одобренная идея + контекст из PROJECT_OS.md + связанные документы |
| **Выходные результаты** | Полная спецификация TASK: цель, scope, ограничения, критерии успеха, зависимости, приоритет, оценка ROI |
| **Ответственный** | Founder (формулирует) + Z.ai (уточняет технические детали) |
| **Критерий прохождения** | Спецификация содержит все поля из Definition of Ready (§3) |

#### Стадия 4: Architecture Review

| Атрибут | Описание |
|---------|----------|
| **Цель** | Оценить влияние задачи на архитектуру, выявить риски, проверить на соответствие Clean Architecture / DDD |
| **Входные данные** | Спецификация TASK + текущая архитектура (PROJECT_OS.md §5, PLATFORM_API_ARCHITECTURE.md) |
| **Выходные результаты** | Архитектурная оценка: Impact (None / Low / Medium / High / Breaking), рекомендации, ограничивающие условия |
| **Ответственный** | Z.ai (анализ) + ChatGPT (review, по запросу) |
| **Критерий прохождения** | Архитектурный impact определён. Если High/Breaking — требуется явное утверждение Founder'ом |
| **Примечание** | Пропускается для чисто документационных задач (impact = None) |

#### Стадия 5: Planning

| Атрибут | Описание |
|---------|----------|
| **Цель** | Разбить задачу на executable шаги, оценить ресурсы, определить зависимости |
| **Входные данные** | Спецификация TASK + архитектурная оценка |
| **Выходные результаты** | План (TodoWrite): конкретные шаги, зависимости между ними, оценка длительности каждого шага |
| **Ответственный** | Z.ai |
| **Критерий прохождения** | План создан, все шаги конкретны и executable, зависимости учтены |

#### Стадия 6: Approved

| Атрибут | Описание |
|---------|----------|
| **Цель** | Финальное подтверждение начала работы |
| **Входные данные** | Спецификация + архитектурная оценка + план |
| **Выходные результаты** | Статус «Approved» — задача может переходить к реализации |
| **Ответственный** | Founder (явное подтверждение или неявное — если спецификация отправлена как TASK, это = Approved) |
| **Критерий прохождения** | Все предыдущие стадии завершены, DoR (§3) пройден |

#### Стадия 7: Implementation

| Атрибут | Описание |
|---------|----------|
| **Цель** | Реализовать код или создать документ согласно спецификации |
| **Входные данные** | Утверждённый план + контекст из PROJECT_OS.md + связанные документы + worklog |
| **Выходные результаты** | Код (src/), документы (docs/), конфигурация — согласно спецификации TASK |
| **Ответственный** | Z.ai (основной) или делегированный агент |
| **Критерий прохождения** | Все пункты плана выполнены. Код компилируется. Базовые тесты проходят |
| **Обязательные действия** | Чтение PROJECT_OS.md → чтение связанных документов → чтение worklog → реализация → промежуточные коммиты |

#### Стадия 8: Self Review

| Атрибут | Описание |
|---------|----------|
| **Цель** | Авторская проверка результата перед передачей на external review |
| **Входные данные** | Результат реализации + спецификация TASK |
| **Выходные результаты** | Self-review отчёт: что реализовано, что не реализовано, известные ограничения, оценка качества |
| **Ответственный** | Z.ai (автор реализации) |
| **Критерий прохождения** | Все критерии успеха из спецификации проверены. Нет known critical issues |

#### Стадия 9: CTO Review

| Атрибут | Описание |
|---------|----------|
| **Цель** | Независимая архитектурная и продуктовая оценка результата |
| **Входные данные** | Self-review отчёт + результат реализации + спецификация TASK |
| **Выходные результаты** | CTO Review: Approved / Approved with Comments / Needs Rework. Если Needs Rework — список замечаний |
| **Ответственный** | ChatGPT (как Virtual CTO) или Z.ai (переключение в CTO-роль) |
| **Критерий прохождения** | CTO Review = Approved или Approved with Comments (non-blocking) |
| **Примечание** | Пропускается для тривиальных задач (изменение документации, исправление опечаток). Обязателен для задач с архитектурным impact Medium+ |

#### Стадия 10: Testing

| Атрибут | Описание |
|---------|----------|
| **Цель** | Верифицировать корректность реализации |
| **Входные данные** | Утверждённый результат + критерии успеха из спецификации |
| **Выходные результаты** | Результаты тестов: unit, integration (если применимо), build verification |
| **Ответственный** | Z.ai или QA-агент (когда будет добавлен) |
| **Критерий прохождения** | Все тесты проходят. Build успешен. Нет regressions |

#### Стадия 11: Documentation Update

| Атрибут | Описание |
|---------|----------|
| **Цель** | Обновить все затронутые документы и операционные артефакты |
| **Входные данные** | Финальный результат реализации + список затронутых документов |
| **Выходные результаты** | Обновлённые: worklog.md, DECISION_LOG.md (если есть новые решения), DOCUMENT_AUDIT.md (если созданы/изменены документы), PROJECT_OS.md (если изменилось текущее состояние), связанные документы |
| **Ответственный** | Z.ai |
| **Критерий прохождения** | Все артефакты обновлены. Ссылки валидны. Версии документов увеличены |

#### Стадия 12: Metrics Update

| Атрибут | Описание |
|---------|----------|
| **Цель** | Зафиксировать метрики выполнения задачи и обновить проектные KPI |
| **Входные данные** | Результат задачи + временные метки |
| **Выходные результаты** | Обновлённые метрики: Lead Time, Cycle Time, Decision Time. Обновлённые KPI в PROJECT_OS.md (если применимо) |
| **Ответственный** | Z.ai |
| **Критерий прохождения** | Метрики рассчитаны и зафиксированы в worklog |

#### Стадия 13: Done

| Атрибут | Описание |
|---------|----------|
| **Цель** | Финальное подтверждение завершения задачи |
| **Входные данные** | Все предыдущие стадии завершены |
| **Выходные результаты** | Статус «Done». Задача больше не требует действий (кроме возможной ретроспективы) |
| **Ответственный** | Z.ai (фиксирует) + Founder (подтверждает) |
| **Критерий прохождения** | Definition of Done (§4) полностью пройден |

#### Стадия 14: Retrospective (периодическая)

| Атрибут | Описание |
|---------|----------|
| **Цель** | Извлечь уроки из завершённой задачи для улучшения процессов |
| **Входные данные** | Опыт выполнения задачи + метрики |
| **Выходные результаты** | Ответы на 5 вопросов из §12 (Continuous Improvement) |
| **Ответственный** | Z.ai |
| **Частота** | После каждой P0/P1 задачи. После каждых 3 P2/P3 задач. По запросу Founder'а |
| **Критерий прохождения** | Уроки зафиксированы. Если требуются изменения в процессах — создан follow-up TASK |

### 2.3 Минимальный набор стадий по типу задачи

| Тип задачи | Обязательные стадии | Опускаемые стадии |
|------------|--------------------|----------------------|
| **Кодовая задача (Feature/Fix)** | Все 14 | Нет |
| **Архитектурный документ** | Idea → Requirements → Planning → Implementation → Self Review → Documentation Update → Metrics → Done | Architecture Review (формально), Testing (код) |
| **Стратегический документ** | Idea → Discussion → Requirements → Planning → Implementation → Self Review → CTO Review → Documentation Update → Done | Architecture Review, Testing (код) |
| **Операционное обновление** | Requirements → Implementation → Documentation Update → Done | Discussion, Architecture Review, CTO Review, Testing, Metrics |
| **Hotfix** | Requirements → Implementation → Testing → Documentation Update → Done | Idea, Discussion, Architecture Review, Planning, Self Review, CTO Review, Metrics, Retrospective |
| **Research / Investigation** | Idea → Discussion → Implementation → Self Review → Documentation Update → Done | Architecture Review, Planning, Testing (код), Metrics |

## 3. Definition of Ready (DoR)

Задача **не начинается**, если хотя бы один пункт не выполнен.

### 3.1 Обязательный чек-лист

| # | Критерий | Проверка | Ответственный |
|---|----------|----------|---------------|
| 1 | **Задача понятна** | Любой участник проекта может описать задачу одним предложением | Founder |
| 2 | **Цель определена** | Сформулирован измеримый результат («сделать X чтобы Y») | Founder |
| 3 | **Критерии успеха описаны** | Существует чёткий список условий, при выполнении которых задача считается успешной | Founder + Z.ai |
| 4 | **Риски известны** | Основные риски (технические, продуктовые, временные) перечислены | Z.ai |
| 5 | **PROJECT_OS прочитан** | Исполнитель знаком с текущим состоянием проекта | Z.ai |
| 6 | **Связанные документы прочитаны** | Все документы, затрагиваемые задачей, прочитаны | Z.ai |
| 7 | **ROI оценён** | Business Value и Engineering Cost определены (хотя бы грубо) | Z.ai |
| 8 | **Приоритет определён** | Задача имеет приоритет P0-P4 (§6) | Founder |
| 9 | **Зависимости определены** | Известно, от каких других задач/документов зависит данная задача | Z.ai |
| 10 | **Конфликты проверены** | Задача не противоречит существующим решениям в DECISION_LOG.md | Z.ai |

### 3.2 Процесс проверки

1. Z.ai проверяет пункты 4-7, 9-10 и формирует DoR отчёт.
2. Founder проверяет пункты 1-3, 8.
3. Если хотя бы один пункт FAILED — задача возвращается на доработку (стадия Requirements).
4. DoR отчёт хранится в worklog.md как часть записи TASK.

## 4. Definition of Done (DoD)

Задача **не считается завершённой**, если хотя бы один пункт не выполнен.

### 4.1 Обязательный чек-лист

| # | Критерий | Проверка | Ответственный |
|---|----------|----------|---------------|
| 1 | **Реализация завершена** | Все пункты плана выполнены. Код компилируется. Функциональность соответствует спецификации | Z.ai |
| 2 | **Тесты выполнены** | Существующие тесты проходят. Новые тесты добавлены (для кодовых задач). Build успешен | Z.ai |
| 3 | **Документация обновлена** | Все затронутые документы обновлены. Новые документы созданы по DOCUMENT_STANDARDS.md | Z.ai |
| 4 | **worklog обновлён** | Запись в worklog.md содержит: Task ID, Agent, Task, Work Log (шаги), Files Created/Modified, Stage Summary | Z.ai |

## 5. Decision Classification

Все решения проекта классифицируются по 5 категориям. Каждая категория имеет своего владельца, шаблон записи и правила хранения.

### 5.1 Категории решений

#### ADR — Architecture Decision Record

| Атрибут | Значение |
|---------|----------|
| **Назначение** | Решения о структуре кода, технологиях, паттернах, архитектурных границах |
| **Владелец** | CTO (Z.ai предлагает, Founder утверждает) |
| **Шаблон записи** | DECISION_LOG.md — стандартный блок с пометкой `[ADR]` |
| **Примеры** | Clean Architecture, Platform Layer, SQLite → PostgreSQL, Strategy Pattern |
| **Правила хранения** | Навсегда. ADR никогда не удаляются. Отменённый ADR помечается `Superseded by ADR-XXX` |

#### PDR — Product Decision Record

| Атрибут | Значение |
|---------|----------|
| **Назначение** | Решения о функциях, UX, приоритетах, позиционировании, onboarding |
| **Владелец** | CPO (Z.ai предлагает, Founder утверждает) |
| **Шаблон записи** | DECISION_LOG.md — стандартный блок с пометкой `[PDR]` |
| **Примеры** | Flat pricing, Email Digest > Telegram Mini App, Security Score как First Value |
| **Правила хранения** | Навсегда. Отменённые PDR помечаются `Superseded by PDR-XXX` |

#### SDR — Security Decision Record

| Атрибут | Значение |
|---------|----------|
| **Назначение** | Решения о безопасности: аутентификация, авторизация, шифрование, обработка данных |
| **Владелец** | CTO (Z.ai предлагает, Founder утверждает) |
| **Шаблон записи** | DECISION_LOG.md — стандартный блок с пометкой `[SDR]` |
| **Примеры** | API Key формат (ssk_), SHA-256 storage, rate limiting, audit logging |
| **Правила хранения** | Навсегда. SDR подлежат периодическому аудиту (ежеквартально) |

#### UDR — UX Decision Record

| Атрибут | Значение |
|---------|----------|
| **Назначение** | Решения о визуальном дизайне, interaction design, навигации, информации |
| **Владелец** | Product Designer (Z.ai предлагает, Founder утверждает) |
| **Шаблон записи** | DECISION_LOG.md — стандартный блок с пометкой `[UDR]` |

| **Правила хранения** | До замены. UDR могут быть пересмотрены на основе user testing данных |

#### BDR — Business Decision Record

| Атрибут | Значение |
|---------|----------|
| **Назначение** | Решения о бизнес-модели, ценообразовании, GTM, партнёрствах, найме, бюджете |
| **Владелец** | CEO (Founder) |
| **Шаблон записи** | DECISION_LOG.md — стандартный блок с пометкой `[BDR]` |
| **Примеры** | Product-Led Growth, Private Beta 50 users, target MRR, market positioning |
| **Правила хранения** | Навсегда. BDR — ключевой контекст для будущих стратегических решений и инвесторов |

## 6. Prioritization Framework

### 6.1 Уровни приоритета

| Приоритет | Название | Определение | SLA начала | Пример |
|-----------|----------|------------|------------|--------|
| **P0** | Critical | Блокер. Без этого невозможно продолжать разработку или продукт неработоспособен | Немедленно | Баг в production, блокер для beta launch |
| **P1** | High | Важная задача, непосредственно влияющая на метрики или user experience | В текущем TASK-цикле | Landing page, Demo target, GitHub OAuth |
| **P2** | Medium | Улучшение, которое повышает качество или скорость, но не блокирует | В течение 2 TASK-циклов | Email digest, Telegram bot, API versioning |
| **P3** | Low | Полезное улучшение, которое может подождать | В течение месяца | Documentation refactor, nice-to-have features |
| **P4** | Wishlist | Идея для будущего, нет чёткой связи с текущими целями | Когда будет время (или никогда) | Mobile app, AI-generated explanations, enterprise SSO |

### 6.2 Оценочные параметры

Каждая задача оценивается по 5 параметрам:

| Параметр | Шкала | Определение |
|----------|-------|-------------|
| **Business Value (BV)** | 1-10 | Насколько задача приближает проект к business goals (MRR, WASP, retention) |
| **Engineering Cost (EC)** | 1-10 | Объём работы (в часах): 1 = <2ч, 5 = 1-2 дня, 10 = 1+ неделя |
| **ROI** | BV / EC | Эффективность вложенных усилий. Порог принятия: ROI ≥ 1.5 |
| **Risk** | Low / Medium / High / Critical | Вероятность негативных последствий (регрессия, архитектурный долг, wasted effort) |
| **Urgency** | Immediate / This week / This month / This quarter | Временное давление (market window, dependency от другой задачи, user demand) |

### 6.3 Правила выбора следующей задачи

```
1. P0 задачи выполняются первыми (в порядке поступления)
2. Среди P1-P2: сортировка по ROI (высокий → низкий)
3. При равном ROI: приоритет у задачи с более высоким Urgency
4. При равном ROI и Urgency: приоритет у задачи с более низким Risk
5. P3-P4 выполняются только если нет P0-P2 в очереди
```

### 6.4 Пример оценки

| Задача | BV | EC | ROI | Risk | Urgency | Priority | Обоснование |
|--------|----|----|-----|------|---------|----------|-------------|
| Landing Page | 9 | 6 | 1.5 | Medium | This week | P1 | Блокер для beta launch |
| Demo Target | 8 | 3 | 2.7 | Low | This week | P1 | Высокий ROI, низкий риск |
| GitHub OAuth | 7 | 2 | 3.5 | Low | This week | P1 | Самый высокий ROI |
| Email Digest | 6 | 5 | 1.2 | Low | This month | P2 | ROI ниже порога, но важен для retention |
| Telegram Bot | 5 | 7 | 0.7 | Medium | This quarter | P3 | ROI ниже порога |

## 7. Release Lifecycle

### 7.1 Стадии выпуска

```
Draft → Development → Internal Review → Private Beta → Release Candidate →
Production → Monitoring → Postmortem (при необходимости)
```

### 7.2 Детальное описание

#### Draft

| Атрибут | Описание |
|---------|----------|
| **Цель** | Фиксация идеи и начального дизайна |
| **Действия** | Создание спецификации, обсуждение, Architecture Review |
| **Критерий перехода** | Requirements + Architecture Review завершены |

#### Development

| Атрибут | Описание |
|---------|----------|
| **Цель** | Реализация функции |
| **Действия** | Код, тесты, документация |
| **Критерий перехода** | Все тесты проходят, Self Review завершён |

#### Internal Review

| Атрибут | Описание |
|---------|----------|
| **Цель** | Независимая проверка качества |
| **Действия** | CTO Review, продуктовая проверка, cross-agent review (по запросу) |
| **Критерий перехода** | CTO Review = Approved. Критических замечаний нет |

#### Private Beta

| Атрибут | Описание |
|---------|----------|
| **Цель** | Валидация на реальных пользователях (если применимо) |
| **Действия** | Деплой на beta-окружение, наблюдение за использованием, сбор feedback |
| **Критерий перехода** | Нет критических багов. Feedback подтверждает ценность. Go/Pivot/Stop оценка |

#### Release Candidate (RC)

| Атрибут | Описание |
|---------|----------|
| **Цель** | Финальная подготовка к production |
| **Действия** | Regression testing, performance check, documentation final review |
| **Критерий перехода** | Все тесты проходят. Performance в пределах норм. Документация актуальна |

#### Production

| Атрибут | Описание |
|---------|----------|
| **Цель** | Доступность для пользователей |
| **Действия** | Деплой, верификация health checks, уведомление пользователей (если нужно) |
| **Критерий перехода** | Health checks зелёные. Нет ошибок в логах |

#### Monitoring

| Атрибут | Описание |
|---------|----------|
| **Цель** | Наблюдение за стабильностью после выпуска |
| **Действия** | Мониторинг ошибок, performance, user behaviour. Duration: 24-72 часа |
| **Критерий перехода** | Нет regressions. Метрики стабильны |

#### Postmortem

| Атрибут | Описание |
|---------|----------|
| **Цель** | Разбор инцидентов, если они произошли |
| **Действия** | Что случилось → Почему → Что делаем чтобы не повторилось → Уроки для процесса |
| **Триггер** | Production incident, significant user complaint, unexpected behaviour |
| **Критерий завершения** | Postmortem записан в worklog. Actions items созданы как TASK'и |

### 7.3 Упрощённый lifecycle для документальных изменений

```
Draft → Implementation → Self Review → Documentation Update → Done
```

## 8. Product Discovery

### 8.1 Когда обязателен Discovery

Product Discovery **обязателен** перед реализацией, если задача соответствует хотя бы одному критерию:

- Оценка Engineering Cost ≥ 5 (более 1-2 дней работы)
- Затрагивает пользовательский интерфейс (UX/UI изменения)
- Вводит новый тип данных или новую интеграцию
- Затрагивает monetization (pricing, billing, trials)
- Создаёт новый канал взаимодействия с пользователем

### 8.2 Формат Discovery

Discovery — это структурированный документ ответов на 5 вопросов:

| # | Вопрос | Описание | Источник ответа |
|---|--------|----------|-----------------|
| 1 | **Существует ли проблема?** | Описание проблемы с доказательствами: user interviews, support tickets, market data, аналитика конкурентов | User research, PMF Blueprint, competitor analysis |
| 2 | **Кто является пользователем?** | Конкретная persona из ICP (не «пользователи вообще», а «Alex, CTO 10-person startup») | PRODUCT_MARKET_FIT_BLUEPRINT.md §5 (ICP/Personas) |
| 3 | **Какие существуют альтернативы?** | Как пользователь решает эту проблему сегодня? Конкуренты? Workarounds? | Competitor analysis, user interviews |
| 4 | **Как будет измеряться успех?** | Конкретная метрика + target value + timeframe. Например: «Activation Rate с 5% до 12% за 30 дней» | PROJECT_OS.md §10 (KPI Dashboard) |
| 5 | **Какие риски имеются?** | Технические риски, риски пользовательского восприятия, риски приоритизации | Architecture review, domain knowledge |

### 8.3 Процесс

1. Z.ai заполняет Discovery-документ на основе существующих стратегических документов.
2. ChatGPT (Virtual CTO) проверяет полноту и ищет слепые spots.
3. Founder утверждает Discovery перед переходом к Requirements.
4. Если хотя бы один вопрос не имеет убедительного ответа — задача откладывается до проведения исследования.

### 8.4 Исключения

Discovery не требуется для:
- Hotfix'ов и баг-фиксов
- Рефакторинга внутри существующих модулей (без изменения функциональности)
- Операционных задач (документация, настройки CI/CD)
- Задач, явно помеченных Founder'ом как «skip discovery»

## 9. AI Collaboration Protocol

### 9.1 Передача контекста

Контекст между AI-агентами передаётся **только через артефакты**. Никаких «неявных» знаний.

| Механизм | Когда используется | Формат |
|----------|-------------------|--------|
| **PROJECT_OS.md** | Каждая новая сессия, любой агент | Структурированный .md |
| **worklog.md** | Понимание хронологии | Append-only лог |
| **DECISION_LOG.md** | Понимание принятых решений | Структурированные блоки |
| **TASK Specification** | Конкретная задача | В сообщении Founder'а |
| **Связанные документы** | Углублённый контекст | .md файлы по ссылкам из PROJECT_OS |
| **Код** | Технический контекст | TypeScript файлы через Grep/Glob/Read |

**Правило:** Если информация не зафиксирована в артефакте — она не существует для следующей сессии.

### 9.2 Управление конфликтами

При конфликте между агентами:

| Уровень конфликта | Пример | Разрешение |
|-------------------|--------|------------|
| **Фактологический** | Разные цифры, разные технические утверждения | Проверить по коду/документам. Артефакт — арбитр |
| **Приоритетный** | Z.ai хочет делать X, ChatGPT рекомендует Y | Сравнить по ROI (§6). Founder решает |
| **Архитектурный** | Разные подходы к решению одной проблемы | Формализовать как ADR. Founder утверждает |
| **Ролевой** | Два агента претендуют на одну зону ответственности | Обратиться к §1 (зоны ответственности). Founder уточняет |

**Эскалация:** Если конфликт не разрешён в течение одного цикла обсуждения — эскалировать на Founder'а с кратким описанием: позиция A, позиция B, рекомендация.

### 9.3 Процедура ревью

#### Cross-Agent Review (по запросу Founder'а)

1. Founder запрашивает multi-role review.
2. Z.ai последовательно переключается в требуемые роли.
3. Каждая роль даёт **независимую** оценку — без доступа к оценкам других ролей.
4. После всех оценок — Z.ai синтезирует общие выводы и противоречия.
5. Финальный документ содержит: оценки каждой роли + синтез + рекомендации.

#### Peer Review (между агентами, когда будут добавлены)

1. Автор создаёт результат.
2. Peer-ревьювер получает контекст через артефакты.
3. Peer-ревьювер возвращает: Approved / Needs Changes / Rejected + замечания.
4. Автор адресует замечания.
5. Если disagreement — эскалация на CTO (ChatGPT) или Founder.

### 9.4 Правила эскалации

```
Level 1: Z.ai → ChatGPT (консультация, review)
Level 2: Z.ai → Founder (решение по спецификации)
Level 3: ChatGPT + Z.ai → Founder (стратегическое решение)
```

**Правило:** Эскалация на Founder'а происходит только если:
- Конфликт не разрешён на Level 1-2
- Требуется решение, выходящее за рамки компетенции агентов (бюджет, найм, партнёрства)
- Задача требует изменения Core Principles (PROJECT_OS.md §3)

### 9.5 Журнал рекомендаций

Все рекомендации ChatGPT (Virtual CTO) фиксируются в worklog.md как часть записи TASK:

```
ChatGPT Recommendations:
- [Рекомендация 1] → [Принята / Отклонена / Отложена] — [причина]
- [Рекомендация 2] → [Принята / Отклонена / Отложена] — [причина]
```

Это позволяет отслеживать:
- Как часто рекомендации принимаются (показатель доверия)
- Какие типы рекомендаций отклоняются (области расхождения)
- Накопленный опыт принятия решений

### 9.6 Финальное решение Founder'а

Founder принимает финальное решение в следующих случаях:

1. **Стратегические решения** (BDR) — всегда Founder.
2. **Архитектурные решения с High/Breaking impact** — Founder утверждает после ADR.
3. **Конфликтные ситуации** — Founder как арбитр.
4. **Изменение Core Principles** — только Founder.
5. **Бюджет и ресурсы** — только Founder.

**Формат решения Founder'а:** Явное утверждение в ответе на предложение Z.ai. Неявное молчание ≠ утверждение (кроме случаев, когда Founder явно делегировал решение).

## 10. Governance

### 10.1 Кто может менять Project OS

| Изменение | Требует утверждения | Процесс |
|-----------|-------------------|---------|
| **Исправление опечаток, обновление ссылок** | Нет | Z.ai делает, фиксирует в worklog |
| **Обновление метрик, KPI, Current Stage** | Нет | Z.ai делает после каждого TASK |
| **Добавление нового раздела** | Founder | Z.ai предлагает, Founder утверждает |
| **Изменение Core Principles** | Founder | Явное решение Founder'а + DECISION_LOG |
| **Изменение Governance правил** | Founder | Явное решение Founder'а + DECISION_LOG |
| **Переписывание документа** | Founder | Явное решение + полный review |

### 10.2 Кто утверждает архитектурные решения

| Тип решения | Утверждающий | Процесс |
|-------------|-------------|---------|
| ADR (Low/Medium impact) | Z.ai (CTO role) | Z.ai записывает в DECISION_LOG |
| ADR (High impact) | ChatGPT (CTO review) → Founder | Z.ai proposes → ChatGPT reviews → Founder approves |
| ADR (Breaking) | Founder только | Z.ai формализует → ChatGPT reviews → Founder решает |
| SDR | Z.ai (CTO role) + Founder для Critical | Z.ai записывает. Critical SDR → Founder |

### 10.3 Кто создаёт новые документы

| Ситуация | Создатель | Процесс |
|----------|-----------|---------|
| TASK требует новый документ | Z.ai | По DOCUMENT_STANDARDS.md → добавить в DOCUMENT_AUDIT.md → обновить PROJECT_OS.md |
| Инициатива Z.ai | Z.ai (с разрешения) | Предложить Founder'у. Если одобрено — создать по стандарту |
| Инициатива Founder'а | Z.ai | Founder описывает → Z.ai создаёт по стандарту |

### 10.4 Когда документ архивируется

Документ архивируется когда:

1. Статус изменён на `Archived` в метаданных.
2. Документ перемещён в `docs/archive/` (или помечен как archived в DOCUMENT_AUDIT.md).
3. Все ссылки в активных документах обновлены — указывают на заменяющий документ.
4. DECISION_LOG содержит запись о причине архивации.

**Исключение:** DECISION_LOG.md и worklog.md **никогда не архивируются**.

### 10.5 Ежегодный аудит документации

Каждые 12 месяцев (или при значительном изменении проекта — pivot, new market, new tech stack):

1. Engineering Manager (Z.ai) проводит полный аудит всех активных документов.
2. Каждому документу присваивается актуальность (1-5) по критериям из DOCUMENT_AUDIT.md.
3. Документы с актуальностью ≤ 2 помечаются как Deprecated.
4. Документы с актуальностью = 1 архивируются.
5. Результаты аудита записываются в DOCUMENT_AUDIT.md с пометкой `[Annual Audit YYYY]`.
6. PROJECT_OS.md Document Index обновляется.

### 10.6 Как обновляются стандарты

Стандарты (этот документ, DOCUMENT_STANDARDS.md, DECISION_LOG.md шаблон) обновляются когда:

1. Обнаружен процесс, который не покрывается текущими стандартами.
2. Существующий стандарт создаёт friction (замедляет работу без добавления ценности).
3. Масштабирование проекта требует новых правил (новые агенты, новые роли).

**Процесс:**
1. Z.ai предлагает изменение в worklog.md (как recommendation).
2. Founder утверждает или отклоняет.
3. Если утверждено — стандарт обновлён, версия увеличена (Major или Minor).
4. DECISION_LOG содержит запись о причине изменения.

## 11. Operating Metrics

### 11.1 Каталог метрик

| Метрика | Определение | Способ измерения | Целевое значение | Частота |
|---------|-------------|------------------|------------------|---------|
| **Lead Time** | Время от Idea до Done (все стадии) | Разница timestamps в worklog | < 1 TASK-цикл для P0, < 3 для P1 | Каждый TASK |
| **Cycle Time** | Время от Approved до Done (реализация) | Разница timestamps в worklog | < 1 TASK-цикл | Каждый TASK |
| **Decision Time** | Время от Discussion до Approved | Разница timestamps в worklog | < 24 часа для P0, < 72 для P1 | Каждый TASK |
| **Documentation Freshness** | Доля активных документов, обновлённых за последние 3 месяца | `актуальные / всего активных` | > 80% | Ежемесячно |
| **Review Coverage** | Доля задач с CTO Review (из тех, где он обязателен) | `задач с review / задач требующих review` | 100% | Каждый TASK |
| **Technical Debt Trend** | Изменение объёма tech debt (в часах) по сравнению с предыдущим месяцем | Оценка Z.ai в PLATFORM_AUDIT.md | Снижение или стабилизация | Ежемесячно |
| **Business Debt Trend** | Изменение объёма business debt (gap между текущим и целевым MRR) | `(Target MRR - Actual MRR)` | Снижение | Ежемесячно |
| **Sprint Predictability** | Доля задач, завершённых в пределах оценки | `завершённых в срок / всего запланированных` | > 70% | Каждые 5 TASK |
| **DoR Pass Rate** | Доля задач, прошедших DoR с первого раза | `прошедших / всего` | > 80% | Каждые 5 TASK |
| **DoD Pass Rate** | Доля задач, прошедших DoD с первого раза | `прошедших / всего` | > 70% | Каждые 5 TASK |

### 11.2 Использование метрик

Метрики **не являются целью сами по себе**. Они служат инструментом выявления проблем:

| Сигнал | Возможная проблема | Действие |
|--------|-------------------|----------|
| Lead Time растёт | Задачи становятся сложнее или процессы замедляются | Анализ bottleneck'ов, упрощение процессов |
| Decision Time > 72h | Founder недоступен или спецификация неполна | Улучшить спецификации, добавить async-утверждение |
| Documentation Freshness < 60% | Документация устаревает | Запланировать документational TASK |
| Review Coverage < 100% | CTO Review пропускается | Усилить контроль, сделать обязательным в DoD |
| Business Debt не снижается | GTM не работает или продукт не привлекает пользователей | Pivot в стратегии, deeper user research |

### 11.3 Отчётность

- **Каждый TASK:** Lead Time, Cycle Time, Decision Time фиксируются в worklog.
- **Каждые 5 TASK:** Sprint Predictability, DoR/DoD Pass Rate рассчитываются и записываются в worklog.
- **Ежемесячно:** Documentation Freshness, Tech/Business Debt Trend — записываются в PROJECT_OS.md §10 (KPI Dashboard).
- **Ежеквартально:** Полный Operating Metrics Review с трендами и рекомендациями.

## 12. Continuous Improvement

### 12.1 Механизм

После каждой завершённой задачи (для P0/P1 — обязательно, для P2+ — каждые 3 задачи) проводится мини-ретроспектива.

### 12.2 Вопросы ретроспективы

| # | Вопрос | Цель |
|---|--------|------|
| 1 | **Что получилось хорошо?** | Зафиксировать практики, которые стоит продолжать |
| 2 | **Что можно улучшить?** | Выявить friction points и inefficiencies |
| 3 | **Какие правила перестали работать?** | Определить устаревшие процессы, которые создают friction |
| 4 | **Какие документы требуют обновления?** | Поддерживать документацию в актуальном состоянии |
| 5 | **Какие процессы необходимо изменить?** | Сформировать конкретные предложения по улучшению |

### 12.3 Процесс

1. Z.ai отвечает на 5 вопросов в конце каждого TASK (для P0/P1).
2. Ответы записываются в worklog.md как раздел `Retrospective` внутри записи TASK.
3. Если вопрос 3 или 5 выявляет проблему — Z.ai создаёт follow-up recommendation.
4. Если Founder согласен — создаётся отдельный TASK для улучшения процесса.
5. Если улучшение касается этого документа (AI_OPERATING_MODEL.md) — документ обновляется, версия увеличивается.

### 12.4 Пример формата

```markdown
### Retrospective (TASK-XXX)

1. **Что получилось хорошо:** Чёткая спецификация позволила выполнить задачу за один цикл.
   Context budget приоритеты сработали — прочитал только нужные документы.

2. **Что можно улучшить:** Architecture Review занял больше времени, чем реализация.
   Возможно, для Low-impact задач стоит упрощать процесс.

3. **Какие правила перестали работать:** Правило «каждый документ обновляется в конце TASK»
   создаёт friction для больших задач — лучше обновлять incrementally.

4. **Какие документы требуют обновления:** PROJECT_OS.md §7 — добавить информацию о новом
   компоненте. PLATFORM_API_ARCHITECTURE.md — обновить диаграмму.

5. **Какие процессы необходимо изменить:** Предлагаю добавить исключение в Architecture Review
   для задач с impact = None (чисто документальные).
```

## Приложение C: Доработки по результатам ревью

### C.1 Emergency Track (по замечанию Engineering Manager)

Для production incidents и критических hotfix'ов определён упрощённый 3-стадийный pipeline:

```
Emergency Requirements → Emergency Implementation → Emergency Verify
```

| Стадия | Действия | Время |
|--------|----------|-------|
| **Emergency Requirements** | Фиксация проблемы, минимальный scope (только исправление, без рефакторинга) | < 5 мин |
| **Emergency Implementation** | Минимальное исправление. Никаких «заодно» изменений | < 30 мин |
| **Emergency Verify** | Build + тесты + ручная проверка. Deploy | < 15 мин |

**Post-emergency:** В течение 24 часов — полный DoD (документация, DECISION_LOG, worklog) + Postmortem (если применимо).

### C.2 WASP привязка в Product Discovery (по замечанию PM)

В §8.2 вопрос 4 дополнен: каждый Discovery должен содержать оценку влияния на North Star Metric:

> «Как будет измеряться успех? Конкретная метрика + target value + timeframe. Обязательно указать ожидаемое влияние на WASP (Weekly Active Scanning Projects).»

### C.3 NRR анализ (по замечанию PM)

В §8.1 добавлено условие для обязательного Discovery:

> Затрагивает retention или churn (directly или indirectly через user experience)

### C.4 WIP Limit (по замечанию Agile Coach)

Добавлено правило в §14.1:

> **WIP Limit = 1.** В текущей стадии проекта (solo AI-agent, один Founder) одновременно может выполняться только одна задача. WIP увеличивается до 2-3 при появлении human developers или параллельных AI-агентов.
