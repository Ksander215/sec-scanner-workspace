## 1. Цель

Определить единый стандарт для всех документов проекта Sec Scanner. Стандарт обеспечивает:
- быструю ориентацию в любом документе
- однозначное определение владельца и ответственности
- отслеживание актуальности
- связность между документами

## 2. Обязательные метаданные

Каждый документ **должен** начинаться с блока метаданных:

```markdown
# [НАЗВАНИЕ] — Sec Scanner

> **Дата:** YYYY-MM-DD
> **Версия:** X.Y
> **Тип:** [Категория] — [Подкатегория]
> **Владелец:** [Роль]
> **Статус:** [Active | Draft | Deprecated | Archived]
> **Связанные документы:** [список через запятую]
```

### 2.1 Категории документов

| Категория | Описание | Примеры |
|-----------|----------|---------|
| Стратегия | Продуктовые и бизнес-решения | PMF Blueprint, Multi-Channel Strategy |
| Архитектура | Технические решения и дизайн | Platform API Architecture, Domain Model |
| Инженерия | Спецификации реализованных модулей | Security State Engine, Explainability Layer |
| Продукт | UX, пользовательские сценарии | User Flow, First Value Experience |
| Операционный | Процессы, стандарты, логи | Project OS, Decision Log, Document Standards |
| Временный | Рабочие артефакты (жизнь < 1 TASK) | Исследования, черновики |

### 2.2 Роли-владельцы

| Роль | Зона ответственности |
|------|---------------------|
| CEO | Стратегия, монетизация, GTM, investors |
| CTO | Архитектура, инженерия, безопасность, инфраструктура |
| CPO | Продукт, UX, roadmap, метрики, ICP |
| VP Marketing | Контент, бренд, каналы привлечения |
| Head of Sales | Ценообразование, продажи, ICP |
| Customer Success | Онбординг, retention, support |
| Product Designer | UX/UI, визуальный дизайн, interaction design |
| Engineering Manager | Процессы, стандарты, документация, операционная эффективность |

### 2.3 Статусы документов

| Статус | Значение | Действия |
|--------|----------|----------|
| **Active** | Текущий, используется для принятия решений | Обновляется при изменениях |
| **Draft** | Черновик, требует ревью или доработки | Должен перейти в Active или Archived в течение 2 TASK |
| **Deprecated** | Устарел, но ещё может быть полезен для контекста | Не обновляется, ссылки на более новый документ |
| **Archived** | Исторический, не используется | Перемещён в `docs/archive/` |

## 4. Правила обновления

### 4.1 Когда обновлять

| Событие | Обязательное действие |
|---------|----------------------|
| TASK затронул документ | Обновить затронутые разделы, увеличить версию (patch) |
| Новое архитектурное решение | Добавить в DECISION_LOG.md, обновить связанные документы |
| Изменился связанный документ | Проверить актуальность ссылок, обновить если нужно |
| Документ устарел | Изменить статус на Deprecated или Archived |
| Новый TASK создаёт документ | Создать по стандарту, добавить в DOCUMENT_AUDIT.md |

### 4.2 Версионирование

- **Major (X.0):** переработка структуры или изменение ключевых решений
- **Minor (0.Y):** добавление новых разделов или значительное обновление содержания
- **Patch (0.0.Z):** исправления, уточнения, обновление ссылок

### 4.3 Язык

- **Основной язык:** Русский (для стратегических и операционных документов)
- **Технические термины:** На английском (DAST, SAST, Clean Architecture, и т.д.)
- **Код и идентификаторы:** На английском
- Допускается английский для документов, целевой аудиторией которых является международное сообщество (open-source, technical blog posts)

## 5. Критерии актуальности

Документ считается **актуальным**, если:
1. Все технические утверждения соответствуют текущему коду
2. Все ссылки на другие документы валидны
3. Статус = Active
4. Дата последнего обновления < 6 месяцев
5. Решения, описанные в документе, не были пересмотрены

Документ считается **требующим ревью**, если:
1. Дата последнего обновления > 3 месяцев
2. Связанный документ был обновлён с момента последней ревью данного документа
3. Код, описанный в документе, был изменён
