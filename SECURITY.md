# Политика безопасности SIP

Данный документ описывает политику безопасности проекта **SIP — Security Intelligence Platform** и процедуры ответственного раскрытия уязвимостей. Безопасность наших пользователей и их данных является абсолютным приоритетом, и мы призываем сообщество помогать нам в обеспечении защиты платформы.

---

## Поддерживаемые версии

Мы обеспечиваем исправления безопасности для следующих версий проекта:

| Версия  | Поддержка                           | Статус       |
|---------|--------------------------------------|--------------|
| v0.4.x  | ✅ Активная поддержка                | Stable       |
| v0.3.x  | ✅ Исправления безопасности          | Maintenance  |
| v0.2.x  | ⚠️ Только критические уязвимости    | End-of-life  |
| v0.1.x  | ❌ Не поддерживается                 | Deprecated   |
| < v0.1  | ❌ Не поддерживается                 | Deprecated   |

Мы рекомендуем всем пользователям обновляться до последней стабильной версии. Каждая новая версия содержит улучшения безопасности, исправления известных уязвимостей и актуальные зависимости.

## Как сообщить об уязвимости

### Ответственное раскрытие

Мы придерживаемся принципов ответственного раскрытия (Responsible Disclosure) и призываем исследователей безопасности сообщать об обнаруженных уязвимостей конфиденциально, не раскрывая информацию публично до выпуска исправления.

### Каналы связи

1. **Приоритетный канал**: отправьте отчёт на email **security@sip-platform.dev** с темой `[SECURITY] Краткое описание уязвимости`. Мы гарантируем ответ в течение 24 часов.
2. **GitHub Security Advisory**: создайте приватный Security Advisory через [GitHub Vulnerability Reporting](https://github.com/sip-platform/sip/security/advisories/new).
3. **PGP-шифрование**: для особо чувствительных отчётов используйте наш публичный PGP-ключ (доступен на странице безопасности проекта).

### Что включить в отчёт

Для максимально быстрой обработки уязвимости, пожалуйста, предоставьте:

- **Тип уязвимости**: XSS, SQL-инъекция, SSRF, утечка данных, повышение привилегий и т. д.
- **Компонент**: конкретный модуль, эндпоинт или файл, в котором обнаружена уязвимость.
- **Версия**: версия SIP, на которой воспроизводится проблема.
- **Условия воспроизведения**: пошаговая инструкция, включая необходимые привилегии, конфигурацию и входные данные.
- **Доказательство концепции (PoC)**: минимальный пример кода или HTTP-запрос, демонстрирующий уязвимость.
- **Влияние**: оценка потенциального ущерба (утечка данных, отказ в обслуживании, выполнение произвольного кода).
- **Возможные меры снижения**: если вы видите способ временной защиты до выпуска исправления.

### Чего не следует делать

- ❌ Не эксплуатируйте уязвимость для доступа к данным других пользователей.
- ❌ Не публикуйте информацию об уязвимости в публичных Issue, форумах или социальных сетях до выпуска исправления.
- ❌ Не используйте автоматизированные сканеры, которые могут нарушить доступность сервиса для других пользователей.
- ❌ Не пытайтесь получить доступ к системам, не относящимся к проекту SIP.

## График реагирования

Мы стремимся к максимально оперативной обработке сообщений о уязвимостях:

| Этап                               | Срок                           |
|-------------------------------------|--------------------------------|
| Подтверждение получения отчёта     | 24 часа                        |
| Первичная оценка и классификация    | 3 рабочих дня                  |
| Разработка исправления (Critical)   | 7 рабочих дней                 |
| Разработка исправления (High)       | 14 рабочих дней                |
| Разработка исправления (Medium)     | 30 рабочих дней                |
| Разработка исправления (Low)        | Следующий плановый релиз       |
| Выпуск патча и уведомление          | В течение 48 часов после готовности исправления |

### Классификация серьёзности

Мы используем стандарт **CVSS v3.1** для оценки серьёзности уязвимостей:

- **Critical** (9.0–10.0): удалённое выполнение кода, массовая утечка данных, полный обход аутентификации.
- **High** (7.0–8.9): повышение привилегий, утечка конфиденциальных данных, SSRF с доступом к внутренним ресурсам.
- **Medium** (4.0–6.9): отражённый XSS, утечка метаданных, ограниченный отказ в обслуживании.
- **Low** (0.1–3.9): хранимый XSS с ограниченным влиянием, утечка некритичной информации.

### Уведомление пользователей

После выпуска исправления мы:

1. Публикуем Security Advisory на GitHub с описанием уязвимости и инструкции по обновлению.
2. Отправляем уведомление через канал безопасности в Discord/Slack сообщества.
3. Обновляем CHANGELOG.md с пометкой `[SECURITY]`.
4. При критических уязвимостях — отправляем email-уведомление зарегистрированным пользователям.

## Практики безопасности для контрибьюторов

### Общие принципы

- **Принцип наименьших привилегий**: код должен запрашивать минимально необходимые права для выполнения своих функций.
- **Защита в глубину (Defense in Depth)**: каждый уровень системы должен иметь собственные механизмы защиты, не полагаясь исключительно на внешний периметр.
- **Не доверяйте входящим данным**: все данные от клиента, внешних API и плагинов должны валидироваться и санитизироваться перед обработкой.

### Специфические требования

- **Аутентификация и авторизация**: каждый API-эндпоинт должен проверять права доступа. Используйте middleware `requireAuth()` и `requireRole()`. Никогда не доверяйте данным из JWT без верификации подписи.
- **SQL/NoSQL-инъекции**: используйте параметризованные запросы. Запрещена конкатенация строк для формирования запросов к базе данных.
- **XSS**: все пользовательские данные, отображаемые в UI, должны экранироваться автоматически (React экранирует по умолчанию). При использовании `dangerouslySetInnerHTML` обязательна предварительная санитизация через DOMPurify.
- **Секреты и учётные данные**: никогда не коммитьте секреты, API-ключи, токены и пароли. Используйте переменные окружения и `.env`-файлы (добавленные в `.gitignore`). В CI/CD используйте зашифрованные секреты.
- **Зависимости**: перед добавлением новой зависимости проверьте её репутацию, количество звёзд, активность поддержки и наличие известных уязвимостей через `npm audit`.
- **Плагины**: плагины выполняются в изолированной среде (sandbox) с ограниченным доступом к файловой системе и сети. Не расширяйте права плагинов без веской причины и ревью безопасности.

### Проверка безопасности в CI

Каждый PR автоматически проходит:

- Анализ зависимостей на известные уязвимости (`npm audit`, Snyk).
- Статический анализ безопасности (CodeQL, Semgrep).
- Проверку на утечку секретов (git-secrets, TruffleHog).
- Сканирование Docker-образов на уязвимости (Trivy).
