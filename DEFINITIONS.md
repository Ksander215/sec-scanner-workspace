## Цель

Единая точка ссылки на Definition of Ready и Definition of Done для всех типов артефактов. **Не дублирует** полные определения из AI_OPERATING_MODEL.md, а дополняет их специфическими критериями для Issue, ADR, Sprint и Review.

Полные DoR (10 чекпоинтов) и DoD (10 чекпоинтов) для задач: см. [AI_OPERATING_MODEL.md, Sections 3-4](docs/project_os/AI_OPERATING_MODEL.md).

## 1. Issue — Definition of Ready

Issue готов к работе, если:

| # | Критерий | Проверка |
|---|----------|----------|
| 1 | **Заголовок** описывает суть (не «bug» а «Pricing page shows $49 instead of $29») | Читаемость |
| 2 | **Тип** выбран (Feature / Bug / ADR / Product Improvement / Tech Debt / Research) | Шаблон заполнен |
| 3 | **Приоритет** установлен (P0-P4) | По приоритизационному фреймворку |
| 4 | **Критерии успеха** измеримы | Содержат числа, проценты, yes/no |
| 5 | **Контекст** достаточен для начала работы | Связанные документы указаны |
| 6 | **Связь со стратегией** | Указано влияние на WASP, Success Gate, или KPI |
| 7 | **Нет дубликатов** | Поиск по Issues не находит аналогичных |

## 2. Issue — Definition of Done

Issue закрыт, если:

| # | Критерий |
|---|----------|
| 1 | Все критерии успеха из Issue выполнены |
| 2 | Связанные документы обновлены |
| 3 | Затронутые тесты проходят (если код) |
| 4 | DECISION_LOG обновлён (если принято решение) |
| 5 | worklog обновлён |
| 6 | PR merged в main (если код) |

## 3. ADR — Definition of Ready

Архитектурное решение готово к записи, если:

| # | Критерий |
|---|----------|
| 1 | Проблема сформулирована (не симптом, а причина) |
| 2 | Минимум 2 альтернативы рассмотрены |
| 3 | Выбранная альтернатива имеет измеримые Benefits |
| 4 | Trade-offs и Risks явно описаны |
| 5 | Review Date установлен (через 3-6 месяцев) |
| 6 | Success Metrics определены (Primary + Secondary) |

## 4. ADR — Definition of Done

ADR записан, если:

| # | Критерий |
|---|----------|
| 1 | Запись следует стандартному шаблону (DECISION_MANAGEMENT_FRAMEWORK) |
| 2 | Все поля заполнены (Проблема, Варианты, Выбор, Обоснование, Trade-offs, Risks) |
| 3 | Связанные решения указаны (Related Decisions) |
| 4 | Затронутые документы обновлены |
| 5 | Затронутый код соответствует решению (если применимо) |

## 5. Sprint — Definition of Ready

Спринт готов к старту, если:

| # | Критерий |
|---|----------|
| 1 | Sprint Goal сформулирован (1-2 предложения) |
| 2 | Все задачи в спринте имеют статус Ready (DoR пройден) |
| 3 | Зависимости между задачами известны и разрешимы |
| 4 | Capacity оценена (дней/часов на спринт) |
| 5 | Sprint связан с Success Gate (какой Gate продвигаем) |
| 6 | KPI targets для спринта определены |

## 6. Sprint — Definition of Done

Спринт завершён, если:

| # | Критерий |
|---|----------|
| 1 | Все задачи спринта: Done или перемещены в backlog с причиной |
| 2 | Sprint Goal достигнут (или documented reason почему нет) |
| 3 | KPI замерены (WASP, Activation, TTFV и т.д.) |
| 4 | FOUNDER_DASHBOARD обновлён |
| 5 | Блокеры задокументированы или устранены |
| 6 | Извлечённые уроки записаны (Retrospective) |
| 7 | Следующий спринт спланирован (задачи готовы в backlog) |

## 7. Review — Definition of Ready

Review готов к проведению, если:

| # | Критерий |
|---|----------|
| 1 | Данные собраны (dashboard, метрики, feedback) |
| 2 | Предыдущий review проведён (weekly: 7 дней назад, monthly: 30 дней) |
| 3 | Формат соответствует шаблону (WEEKLY_REVIEW_TEMPLATE или MONTHLY_BUSINESS_REVIEW) |

## 8. Review — Definition of Done

Review завершён, если:

| # | Критерий |
|---|----------|
| 1 | Все разделы шаблона заполнены |
| 2 | Key Questions содержат конкретные ответы (не «нормально») |
| 3 | Action Items имеют ответственных и сроки |
| 4 | Review сохранён в `docs/reviews/` |
| 5 | FOUNDER_DASHBOARD обновлён (если есть изменения) |
| 6 | Решения, требующие фиксации, добавлены в DECISION_LOG |

## 9. Roadmap — Definition of Ready

Roadmap-элемент готов к добавлению, если:

| # | Критерий |
|---|----------|
| 1 | Элемент имеет оценку ROI (BV / EC) |
| 2 | Связан с North Star (WASP) или Success Gate |
| 3 | Не дублирует существующий элемент в EXECUTION_BACKLOG |
| 4 | Имеет приоритет (P0-P4) |

## 10. Roadmap — Definition of Done

Roadmap-элемент выполнен, если:

| # | Критерий |
|---|----------|
| 1 | Связанная задача в EXECUTION_BACKLOG имеет статус Done |
| 2 | Метрики, связанные с элементом, замерены |
| 3 | MILESTONES обновлён (если элемент привязан к milestone) |
| 4 | FOUNDER_DASHBOARD отражает изменение |
