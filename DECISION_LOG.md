## Назначение

Этот файл — **центральный каталог решений, принятых в отношении самого Workspace** (инфраструктуры управления проектом), а не продукта Sec Scanner.

**Разделение ответственности:**
- `docs/decisions/DECISION_LOG.md` — что мы строим и почему (продукт)
- `DECISION_LOG.md` (этот файл) — как мы управляем и почему (Workspace/процессы)

## Решения

### WS-001: Нумерованная структура директорий docs/

- **Дата:** 2026-07-15
- **Задача:** REPO-001
- **Краткое описание:** Использовать числовые префиксы `00_`—`08_` для каталогов документации вместо алфавитного порядка.
- **Причина:** Новый участник должен понимать порядок чтения без отдельного руководства. Алфавитный порядок ставит `archive` рядом с `architecture`.
- **Рассмотренные альтернативы:**
  1. Алфавитный порядок (status quo)
  2. По типу владельца (CEO docs, CTO docs)
  3. Числовые префиксы, отражающие порядок чтения
- **Принятое решение:** Вариант 3 — `00_project_os` → `01_strategy` → `02_architecture` → `03_product` → `04_execution` → `05_growth` → `06_decisions` → `07_reviews` → `08_draft` + `archive/` без номера
- **Последствия:** Новые каталоги требуют согласования (нельзя просто создать `new_folder/`). Масштабирование — добавить `09_*/`, `10_*/` и т.д.
- **Статус:** Accepted
- **Связь с ADR:** DECISION-012 (Project OS вместо набора документов)

### WS-002: Шаблоны Issue с привязкой к стратегии

- **Дата:** 2026-07-15
- **Задача:** REPO-001
- **Краткое описание:** 6 шаблонов Issue (Feature, Bug, ADR, Product Improvement, Tech Debt, Research), каждый с обязательной привязкой к North Star / Success Gate / KPI.
- **Причина:** Стандартные GitHub Issue Templates не обеспечивают связь задачи со стратегией. Без привязки — риск работать на неважных задачах.
- **Рассмотренные альтернативы:**
  1. Стандартные GitHub Issue Templates (без кастомизации)
  2. Единый шаблон с чекбоксами
  3. 6 специализированных шаблонов с обязательными полями стратегии
- **Принятое решение:** Вариант 3 — 6 шаблонов в `.github/ISSUE_TEMPLATE/`
- **Последствия:** Каждая задача требует понимания North Star. Небольшой overhead при создании Issue, но предотвращает стратегическое漂移.
- **Статус:** Accepted
- **Связь с ADR:** DECISION-EXEC-001 (Master Execution Plan)

### WS-003: 7-колоночный Kanban вместо стандартного 5-колоночного

- **Дата:** 2026-07-15
- **Задача:** REPO-001
- **Краткое описание:** Использовать 7 колонок (Icebox, Backlog, Ready, In Progress, Review, Testing, Done) для GitHub Projects вместо стандартного 5-колоночного Kanban.
- **Причина:** WIP=1 требует чёткого разделения Ready и Backlog. Тестирование отделено от Review, потому что у нас определённые критерии тестирования (build, unit tests, manual verification).
- **Рассмотренные альтернативы:**
  1. Стандартный GitHub Kanban (5 колонок)
  2. Упрощённый 5-колоночный (Backlog, Ready, In Progress, Review, Done)
  3. 7 колонок с WIP limits
- **Принятое решение:** Вариант 3 — 7 колонок с WIP limits (In Progress = 1, Review = 2, Testing = 2)
- **Последствия:** Больше колонок для навигации, но каждая имеет чёткие критерии входа/выхода.
- **Статус:** Accepted

### WS-004: Corporate-level README вместо стандартного

- **Дата:** 2026-07-15
- **Задача:** REPO-001
- **Краткое описание:** README.md как корпоративный entry point — не технический how-to, а обзор проекта для любого участника.
- **Причина:** Этот репозиторий — SSOT (Single Source of Truth). README — первое, что видит любой участник. Должен отвечать «что это и зачем» за 2 минуты.
- **Рассмотренные альтернативы:**
  1. Минимальный README (только ссылки)
  2. Технический README (setup instructions)
  3. Корпоративный README с обзором проекта, архитектурой, ролями, quick links
- **Принятое решение:** Вариант 3
- **Последствия:** README требует обновления при изменении стадии проекта, Team, или структуры.
- **Статус:** Accepted

### WS-005: DoR/DoD в отдельном DEFINITIONS.md, не разбросаны

- **Дата:** 2026-07-15
- **Задача:** REPO-001
- **Краткое описание:** Единый файл `DEFINITIONS.md` с DoR/DoD для 5 артефактов (Issue, ADR, Sprint, Review, Roadmap) вместо распределения по нескольким документам.
- **Причина:** DoR/DoD для задач уже есть в AI_OPERATING_MODEL.md. Для других артефактов (Sprint, Review, Roadmap) — не были формализованы. DEFINITIONS.md дополняет, не дублирует.
- **Рассмотренные альтернативы:**
  1. Все DoR/DoD в AI_OPERATING_MODEL.md (один документ > 1200 строк)
  2. DoR/DoD в каждом документе (Contributing, Governance, Sprint template)
  3. Единый DEFINITIONS.md с перекрёстными ссылками
- **Принятое решение:** Вариант 3 — DEFINITIONS.md как единая точка ссылки
- **Последствия:** При изменении DoR/DoD нужно обновить один файл.
- **Статус:** Accepted
- **Связь с ADR:** DECISION-OS-002 (AI Operating Model v2)

### WS-006: Milestones привязаны к Success Gates, не к Sprint

- **Дата:** 2026-07-15
- **Задача:** REPO-001
- **Краткое описание:** GitHub Milestones отражают стратегические контрольные точки (M1: Alpha, M2: Beta Ready, ..., M6: PMF Signal), а не спринты.
- **Причина:** Milestones в GitHub — долгосрочные цели. Sprint — недельные итерации. Смешение создаёт путаницу.
- **Рассмотренные альтернативы:**
  1. Milestone = Sprint (1 неделя)
  2. Milestone = Strategic Gate (1-6 месяцев)
  3. Не использовать Milestones (только Labels)
- **Принятое решение:** Вариант 2 — 6 Milestones от Alpha до PMF Signal
- **Последствия:** Sprint трекинг через Labels (`sprint: 01`), не через Milestones.
- **Статус:** Accepted

### WS-007: INDEX.md как навигационный центр

- **Дата:** 2026-07-15
- **Задача:** REPO-002
- **Краткое описание:** Создать INDEX.md в корне Workspace — единая точка входа в документацию с описанием каждого раздела, связями и рекомендуемым порядком чтения.
- **Причина:** README.md содержит overview проекта, но не заменяет навигацию по документации. У нового участника нет единого места, где описана вся структура документов и их взаимосвязи.
- **Рассмотренные альтернативы:**
  1. Расширить README.md разделом «All Documents»
  2. Индекс в PROJECT_OS.md (Document Index)
  3. Отдельный INDEX.md как навигационный хаб
- **Принятое решение:** Вариант 3 — INDEX.md как отдельный документ. README остаётся overview проекта. PROJECT_OS остаётся конституцией. INDEX связывает всё вместе.
- **Последствия:** Три entry point (README → INDEX → конкретный документ). Чёткое разделение ответственности между ними.
- **Статус:** Accepted

### WS-008: 6-стадийный Document Lifecycle вместо 4-стадийного

- **Дата:** 2026-07-15
- **Задача:** REPO-002
- **Краткое описание:** Расширить жизненный цикл документов с 4 стадий (Draft→Active→Deprecated→Archived) до 6 стадий (Draft→Review→Approved→Active→Archived→Obsolete) с формальными критериями переходов.
- **Причина:** REPOSITORY_STANDARDS.md (REPO-001) определял только 4 стадии без критериев переходов. Стадия Review нужна для стратегических документов. Стадия Approved нужна между Review и Active. Стадия Obsolete отличается от Archived (Archived = сохранён для истории, Obsolete = больше не актуален, может быть удалён).
- **Рассмотренные альтернативы:**
  1. Оставить 4 стадии из REPOSITORY_STANDARDS.md
  2. 5 стадий (добавить Review)
  3. 6 стадий (Draft→Review→Approved→Active→Archived→Obsolete)
- **Принятое решение:** Вариант 3 — полная модель lifecycle
- **Последствия:** REPOSITORY_STANDARDS.md Section 5 нужно будет обновить при MIG-001 (Documentation Migration).
- **Статус:** Accepted

### WS-009: Workspace CHANGELOG отдельно от продукта

- **Дата:** 2026-07-15
- **Задача:** REPO-002
- **Краткое описание:** Создать CHANGELOG.md для истории развития Workspace (инфраструктуры управления), а не продукта.
- **Причина:** Workspace — это тоже продукт (внутренний). Его эволюция (добавление шаблонов, изменение процессов, обновление стандартов) должна отслеживаться. Это поможет при onboarding новых участников и при аудите процессов.
- **Рассмотренные альтернативы:**
  1. Не вести CHANGELOG (Git log достаточно)
  2. CHANGELOG как раздел в GOVERNANCE.md
  3. Отдельный CHANGELOG.md с версионированием Workspace
- **Принятое решение:** Вариант 3 — отдельный CHANGELOG.md. Git log показывает «что изменили в файлах», CHANGELOG показывает «почему изменили процессы».
- **Последствия:** Каждое значимое изменение Workspace требует записи в CHANGELOG. Обновляется при REPO-* и MIG-* задачах.
- **Статус:** Accepted

### WS-010: Repository Map с Mermaid-диаграммами

- **Дата:** 2026-07-15
- **Задача:** REPO-002
- **Краткое описание:** Создать визуальное описание структуры репозитория с Mermaid-диаграммами (иерархия каталогов, поток информации, связи между разделами).
- **Причина:** Текстовое описание структуры (REPOSITORY_STANDARDS.md Section 2) не передаёт потоки информации и связи между разделами. Визуальная карта сокращает время ориентации.
- **Рассмотренные альтернативы:**
  1. Только текстовое описание
  2. ASCII-art диаграммы
  3. Mermaid-диаграммы в Markdown (поддерживается GitHub)
- **Принятое решение:** Вариант 3 — Mermaid встроен в Markdown, рендерится на GitHub
- **Последствия:** Зависимость от Mermaid-поддержки GitHub (доступна нативно).
- **Статус:** Accepted
