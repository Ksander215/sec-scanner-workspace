    12→
    13→## Overview
    14→
    15→Security Intelligence Platform is not another DAST scanner. It is an **intelligence layer** that ingests raw findings from any scanner, correlates them into a unified Security Knowledge Graph, computes contextual risk scores, traces multi-step attack paths, and produces actionable, explainable remediation plans.
    16→
    17→The core pipeline:
    18→
    19→```
    20→Knowledge → Intelligence → Explainability → Action
    21→```
    22→
    23→**Key capabilities:**
    24→
    25→- **Correlation Engine** — deduplicate and link findings across scanners by vulnerability type, asset, and data-flow proximity
    26→- **Security Knowledge Graph** — represent assets, vulnerabilities, and their relationships as a queryable graph
    27→- **Risk Engine** — score findings with configurable weights for severity, confidence, exposure, impact, exploitability, and correlation multiplier
    28→- **Attack Path Builder** — trace multi-step exploit chains through the knowledge graph with depth and score thresholds
    29→- **Impact Engine** — assess business impact of attack paths (data exposure, service disruption, compliance violation)
    30→- **Recommendation Engine** — generate prioritized, contextual remediation guidance per finding and attack path
    31→- **Explainability Engine** — produce human-readable explanations of why each risk score and attack path exists
    32→- **AI Layer** — 6 LLM providers, Risk Assistant, Remediation Advisor, Threat Hunting, SOC Copilot
    33→
    34→**Access modes:** CLI (`si`), REST API (Fastify), SDK (TypeScript / Python / Go)

   112→## Feature Matrix
   113→
   114→| Module | ID | Components | Description |
   115→|--------|----|------------|-------------|
   116→| **Normalization** | INT-001 | NormalizationEngine | Normalize findings from heterogeneous scanners into a unified schema |
   117→| **Correlation** | INT-002 | CorrelationEngine | Deduplicate, link, and score relationships between findings |
   118→| **Knowledge Graph** | INT-003 | KnowledgeGraphBuilder | Build a queryable Security Knowledge Graph of assets and vulnerabilities |
   119→| **Risk Scoring** | INT-004 | RiskEngine | Contextual risk scoring with configurable weights (severity, confidence, exposure, impact, exploitability, correlation) |
   120→| **Attack Paths** | INT-005 | AttackPathBuilder | Trace multi-step exploit chains through the knowledge graph |
   121→| **Impact Analysis** | INT-005.5 | ImpactEngine | Assess business impact: data exposure, service disruption, compliance violations |
   122→| **Recommendations** | INT-006 | RecommendationEngine | Generate prioritized, contextual remediation guidance |
   123→| **Explainability** | INT-006.5 | ExplainabilityEngine | Produce human-readable explanations for risk scores and attack paths |
   124→| **Orchestrator** | INT-006.5 | SecurityIntelligenceEngine, Builder | 9-stage pipeline coordinator with progress tracking |
   125→| **Persistence** | INT-007 | PersistenceEngine, JsonPersistenceProvider | 8 repository interfaces (Report, Finding, Risk, Correlation, AttackPath, Recommendation, Explainability, Snapshot) |
   126→| **API** | INT-008 | Fastify Server | 28 routes across 12 route groups with Zod validation and DTO mapping |
   127→| **CLI** | INT-009 | Commander | 10 command groups, 6 output formats (table/json/yaml/csv/jsonl/markdown) |
   128→| **Event Bus** | INT-010 | EventBus | In-process publish/subscribe with typed events |
   129→| **Message Brokers** | INT-011 | Kafka, NATS, RabbitMQ, Redis providers | Pluggable message broker abstraction |
   130→| **Distributed Pipeline** | INT-012 | DistributedPipelineEngine | Partition analysis workloads across nodes |
   131→| **Saga** | INT-013 | SagaEngine | Orchestrate long-running distributed transactions with compensation |
   132→| **Scheduler** | INT-014 | SchedulerEngine | Cron-based recurring analysis and maintenance jobs |
   133→| **Cluster** | INT-015 | ClusterCoordinator | Leader election, node discovery, work distribution |
   134→| **Multi-tenancy** | INT-016 | MultiTenancyEngine | Tenant isolation, per-tenant configuration and quotas |
   135→| **Streaming** | INT-017 | StreamingEngine | Real-time finding ingestion via WebSocket and SSE |
   136→| **AI Layer** | INT-018 | 6 LLM Providers, RiskAssistant, Remediation, ThreatHunting, SOCCopilot | AI-powered analysis, remediation, and co-pilot capabilities |
   137→| **Enterprise** | INT-019 | SSO, SIEM, Ticketing, Notification, CMDB, Secrets | Enterprise integration layer |
   138→| **Threat Intel** | INT-020 | ThreatIntelEngine | Ingest and correlate external threat intelligence feeds |
   139→| **Detection Engineering** | INT-020 | DetectionEngine | Custom detection rule authoring and management |
   140→| **Attack Simulation** | INT-020 | AttackSimEngine | Safe attack chain simulation for validation |

   143→| **Cloud** | INT-020 | CloudEngine | AWS, Azure, GCP, Kubernetes integration |

   392→### Authentication
   393→
   394→All `/api/v1/*` endpoints support three authentication modes:
   395→
   396→| Provider | Header | Configuration |
   397→|----------|--------|---------------|
   398→| JWT Bearer | `Authorization: Bearer <token>` | `SI_AUTH_PROVIDER=jwt`, `SI_JWT_SECRET` |
   399→| API Key | `X-API-Key: <key>` | `SI_AUTH_PROVIDER=api-key` |
   400→| OAuth2 | `Authorization: Bearer <token>` | `SI_AUTH_PROVIDER=oauth2` |
   401→
   402→Set `SI_AUTH_ENABLED=false` to disable authentication (default for local development).

   408→### TypeScript (20 methods)
   409→
   410→```bash
   411→npm install security-intelligence-platform
   412→```
   413→
   414→```typescript
   415→import { SecurityIntelligenceClient, SiApiError } from 'security-intelligence-platform/sdk';
   416→
   417→const client = new SecurityIntelligenceClient({
   418→  baseUrl: 'http://localhost:8080',
   419→  authToken: process.env.SI_AUTH_TOKEN,   // or apiKey: '...'
   420→  timeout: 30000,
   421→});
   422→
   423→// Run synchronous analysis
   424→const report = await client.analyzeSync({
   425→  findings: [
   426→    {
   427→      id: 'f-001',
   428→      title: 'SQL Injection in login endpoint',
   429→      severity: 'critical',
   430→      confidence: 0.95,
   431→      asset: { id: 'a-1', name: 'web-app', type: 'application' },
   432→      location: { file: 'auth.ts', line: 42 },
   433→    },
   434→  ],
   435→});
   436→
   437→console.log(`Risk score: ${report.riskScore}`);
   438→console.log(`Attack paths: ${report.attackPaths.length}`);
   439→
   440→// List and query
   441→const reports = await client.listReports(50, 0);
   442→const risks = await client.listRisks(report.id);
   443→const topRisks = await client.getTopRisks(10);
   444→const attackPaths = await client.listAttackPaths(report.id);
   445→const attackGraph = await client.getAttackGraph(report.id);
   446→const recommendations = await client.listRecommendations(report.id);
   447→const explanations = await client.listExplanations(report.id);
   448→const plan = await client.createRemediationPlan(report.id);
   449→
   450→// Snapshots
   451→const { snapshotId } = await client.createSnapshot(report.id, 'pre-remediation baseline');
   452→const restored = await client.restoreSnapshot(snapshotId);
   453→
   454→// Health check
   455→const health = await client.getHealth();
   456→```

   469→# Synchronous analysis
   470→report = client.analyze_sync(findings=[
   471→    {
   472→        "id": "f-001",
   473→        "title": "XSS in search endpoint",
   474→        "severity": "high",
   475→        "confidence": 0.9,
   476→        "asset": {"id": "a-1", "name": "web-app", "type": "application"},
   477→    },
   478→])
   479→
   480→# Query results
   481→reports = client.list_reports(limit=50)
   482→risks = client.list_risks(report_id=report["id"])
   483→top_risks = client.get_top_risks(limit=10)
   484→attack_paths = client.list_attack_paths(report_id=report["id"])
   485→recommendations = client.list_recommendations(report_id=report["id"])
   486→explanations = client.list_explanations(report_id=report["id"])
   487→plan = client.create_remediation_plan(report_id=report["id"])
   488→
   489→# Snapshots
   490→snapshot = client.create_snapshot(report_id=report["id"], description="baseline")
   491→restored = client.restore_snapshot(snapshot_id=snapshot["snapshotId"])
   492→
   493→# Health
   494→health = client.get_health()
   495→```

   545→## Configuration
   546→
   547→Configuration is loaded with the following priority (lowest to highest):
   548→
   549→1. **Defaults** — built-in sensible defaults
   550→2. **Config file** — `si-config.json` or path from `SI_CONFIG_FILE`
   551→3. **Local override** — `.si-config.local.json`
   552→4. **Environment variables** — `SI_*` prefix (e.g., `SI_SERVER_PORT`, `SI_DB_HOST`)
   553→5. **CLI flags** — `--port`, `--format`, etc.
   554→

   628→## Plugins
   629→
   630→The platform provides a plugin engine with 8 extension points:
   631→
   632→| Extension Point | Interface | Description |
   633→|----------------|-----------|-------------|
   634→| `correlation-rule` | `CorrelationRuleExtension` | Custom finding correlation logic |
   635→| `risk-factor` | `RiskFactorExtension` | Additional risk scoring factors |
   636→| `recommendation-rule` | `RecommendationRuleExtension` | Custom recommendation generators |
   637→| `cli-command` | `CliCommandExtension` | Additional CLI commands |
   638→| `rest-endpoint` | `RestEndpointExtension` | Additional REST endpoints |
   639→| `persistence-provider` | `PersistenceProviderExtension` | New storage backends |
   640→| `analysis-stage` | `AnalysisStageExtension` | Custom pipeline stages |
   641→| `output-formatter` | `OutputFormatterExtension` | New output formats |
   642→
   643→### Plugin Structure
   644→
   645→```typescript
   646→import type { SiPlugin, PluginContext, PluginManifest } from './types.js';
   647→
   648→export class MyCustomPlugin implements SiPlugin {
   649→  readonly manifest: PluginManifest = {
   650→    name: 'my-custom-plugin',
   651→    version: '1.0.0',
   652→    description: 'Custom correlation rules for my environment',
   653→    author: 'security-team',
   654→    entryPoint: __filename,
   655→    extensions: ['correlation-rule', 'risk-factor'],
   656→  };
   657→
   658→  async initialize(context: PluginContext): Promise<void> {
   659→    context.registerCorrelationRule({
   660→      id: 'custom-sqli-correlation',
   661→      name: 'SQLi + Auth Bypass Chain',
   662→      condition: (a, b) => a['type'] === 'sqli' && b['type'] === 'auth-bypass',
   663→      scoreCalculator: (a, b) => 0.95,
   664→    });
   665→
   666→    context.registerRiskFactor({
   667→      id: 'public-facing',
   668→      name: 'Public-Facing Asset',
   669→      weight: 0.25,
   670→      calculate: (finding) => finding['exposure'] === 'public' ? 1.0 : 0.0,
   671→    });
   672→  }
   673→
   674→  async destroy(): Promise<void> {}
   675→}
   676→```

   757→## Security
   758→
   759→The platform is built with a security-first approach across all layers:
   760→
   761→### Transport & Headers
   762→
   763→- **Security headers** applied to every response: `X-Content-Type-Options`, `X-Frame-Options`, `Strict-Transport-Security`, `Content-Security-Policy`, `Referrer-Policy`, `Permissions-Policy`, `Cache-Control: no-store`
   764→- **CORS** with configurable origins, methods, and credentials
   765→- **Rate limiting** via `@fastify/rate-limit` with configurable window and max requests
   766→- **Payload size validation** — rejects oversized request bodies with `413`
   767→- **Input sanitization** — strips script tags and HTML from string inputs
   768→- **Request ID validation** — enforces `^[a-zA-Z0-9\-]{1,64}$` format on `X-Request-ID`
   769→
   770→### Authentication & Authorization
   771→
   772→- **4 auth providers**: None, JWT (HS256/HS384/HS512/RS256/RS384/RS512), API Key, OAuth2
   773→- **RBAC** with 4 roles and 15 fine-grained permissions:
   774→
   775→| Role | Permissions |
   776→|------|-------------|
   777→| `viewer` | Read reports, findings, risks, attack paths, recommendations, explanations |
   778→| `operator` | Viewer + write reports, create/restore snapshots |
   779→| `security-analyst` | Operator + delete findings, write recommendations |
   780→| `administrator` | Full access including user/role management and system config |
   781→
   782→### Container Security
   783→
   784→- Multi-stage Docker build minimizes attack surface
   785→- Runs as non-root user (`si-user`, UID 1001)
   786→- `node:20-alpine` base image
   787→- No dev dependencies in production image
   788→- Built-in health check with `wget`
   789→
   790→### Audit
   791→
   792→All configuration changes and administrative actions are recorded in the audit log:
   793→
   794→| Setting | Values | Env |
   795→|---------|--------|-----|
   796→| Enabled | `true` \| `false` | `SI_AUDIT_ENABLED` |
   797→| Backend | `json` \| `database` | `SI_AUDIT_BACKEND` |
   798→| Retention | Days | `SI_AUDIT_RETENTION` |
