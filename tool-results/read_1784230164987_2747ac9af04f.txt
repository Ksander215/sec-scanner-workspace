    40→## 1. Executive Summary
    41→
    42→Security Intelligence Engine (SIE) -- центральный аналитический модуль платформы, превращающий структурированные данные Security Knowledge Graph в осмысленные выводы о безопасности приложения. SIE не выполняет сканирование и не обращается к сканирующим движкам напрямую. Он работает исключительно через Graph Query API, анализируя уже построенный граф знаний.
    43→
    44→Архитектура SIE строится на принципах детерминированности, неизменяемости, объяснимости и расширяемости. Каждый компонент engine получает данные из графа, производит вычисления и возвращает типизированный результат с полной трассировкой происхождения. Одинаковый граф всегда даёт одинаковый результат -- это фундаментальное отличие от недетерминированных AI/LLM-систем.
    45→
    46→SIE состоит из 11 компонентов: Finding Normalizer, Finding Correlator, Identity Resolution, Evidence Aggregator, Risk Engine, Confidence Engine, Attack Path Builder, Recommendation Engine, Knowledge Graph Adapter, Explainability Adapter и Intelligence API. Каждый компонент имеет чёткую ответственность, определённые входы и выходы, и взаимодействует с другими через внутренние контракты.

    50→## 2. Problem Statement
    51→
    52→Текущая платформа успешно собирает данные из множества источников (Discovery, Browser Intelligence, HTTP Intelligence, Nuclei), но данные остаются изолированными. Каждая находка существует сама по себе, без связи с другими. Это создаёт ряд критических проблем:
    53→
    54→**Изоляция находок.** Nuclei обнаруживает SQL Injection на эндпоинте `/api/users`, Browser Intelligence видит, что этот эндпоинт доступен без аутентификации, а HTTP Intelligence фиксирует отсутствие rate limiting. Каждая находка верна, но их комбинация означает критический риск, который не виден при изолированном анализе.
    55→
    56→**Дублирование данных.** Один и тот же эндпоинт может быть обнаружен Discovery Engine, просканирован Nuclei и проанализирован Browser Intelligence. Без нормализации и дедупликации пользователь видит три записи вместо одной, что снижает доверие к результатам и затрудняет приоритизацию.
    57→
    58→**Отсутствие оценки риска.** CVSS-скоринг отдельных уязвимостей не учитывает контекст приложения. Уязвимость с CVSS 7.5 на административном эндпоинте за аутентификацией может быть менее критичной, чем CVSS 5.3 на публичном API, обрабатывающем чувствительные данные.
    59→
    60→**Нет цепочек атак.** Отдельные уязвимости могут комбинироваться в цепочки эксплуатации, ведущие от публичной точки входа к критическому активу. Без построения Attack Paths такая комбинационная угроза остаётся невидимой.
    61→
    62→**Нет рекомендаций.** Пользователь получает список находок, но не получает конкретных, приоритизированных рекомендаций по их устранению с учётом контекста приложения.

    66→## 3. Goals
    67→
    68→1. **Агрегация и нормализация** -- объединение результатов всех движков в единую модель Finding с устранением дубликатов и разрешением идентичности
    69→2. **Корреляция** -- выявление связей между находками, активами и доказательствами, которые невозможно обнаружить при изолированном анализе
    70→3. **Оценка риска** -- детерминированный расчёт агрегированного риска с учётом severity, confidence, exploitability, exposure, reachability и бизнес-критичности
    71→4. **Построение Attack Paths** -- автоматическое формирование цепочек эксплуатации от точки входа до критического актива
    72→5. **Генерация рекомендаций** -- формирование приоритизированных, контекстных рекомендаций по устранению выявленных рисков
    73→6. **Формирование Security Story** -- связное описание состояния безопасности приложения, объединяющее находки, корреляции, риски и рекомендации в единый нарратив
    74→7. **Explainability Contract** -- обеспечение полной трассируемости каждого вывода для будущего Explainability Engine
    75→8. **Zero Coupling** -- SIE не зависит от конкретных сканирующих движков и не требует изменений в TASK-201, TASK-202 и существующих API
    76→
    77→---
    78→
    79→## 4. Non Goals
    80→
    81→1. **Не выполнение сканирования** -- SIE не инициирует и не управляет сканированием
    82→2. **Не LLM/ML-модуль** -- все аналитические алгоритмы детерминированы и воспроизводимы
    83→3. **Не замена Knowledge Graph** -- SIE использует KG как источник данных, а не заменяет его
    84→4. **Не визуализация** -- Dashboard и UI не входят в зону ответственности SIE
    85→5. **Не Explainability Engine** -- SIE формирует данные для Explainability, но не генерирует пользовательские объяснения
    86→6. **Не修改 существующих модулей** -- TASK-201, TASK-202, Pipeline Executor, Scan Platform Foundation остаются неизменными
    87→7. **Не real-time processing** -- SIE работает в batch/incremental режиме, а не в streaming
    88→8. **Не внешний API** -- Intelligence API предназначен для внутренних потребителей платформы

   184→
   185→SIE представляет собой пайплайн обработки данных, в котором информация проходит последовательную трансформацию: от сырых узлов графа через нормализацию, корреляцию, оценку риска и построение атак к рекомендациям и Security Story. Каждый этап добавляет новый уровень абстракции и ценности.
   186→
   187→**Принцип неизменяемости:** Результат каждого компонента -- immutable-объект. Finding Normalizer создаёт NormalizedFinding, который не может быть изменён последующими компонентами. Correlator создаёт Correlation, Risk Engine создаёт RiskAssessment и т.д. Если результат нужно «обновить», создаётся новая версия с новой меткой времени.
   188→
   189→**Принцип детерминированности:** Все алгоритмы SIE -- чистые функции от входного графа. Один и тот же граф всегда даёт один и тот же результат. Никаких случайных чисел, никаких эвристик с порогами, зависящими от времени суток. Исключение: порядок обработки может быть недетерминирован при параллельном выполнении, но итоговый результат -- всегда детерминирован благодаря коммутативности операций агрегации.

   193→## 7. Component Overview
   194→
   195→### 7.1 Finding Normalizer
   196→
   197→| Атрибут | Значение |
   198→|---------|----------|
   199→| **Назначение** | Приведение сырых находок из разных источников к единой модели NormalizedFinding |
   200→| **Ответственность** | Нормализация severity, маппинг CWE/CVE, унификация форматов, обогащение метаданными из графа |
   201→| **Входные данные** | Сырые Findings из Knowledge Graph (nodes типа Finding) |
   202→| **Выходные данные** | NormalizedFinding[] -- массив унифицированных находок |
   203→| **Взаимодействие** | Knowledge Graph Adapter (чтение), Finding Correlator (запись) |
   204→| **Ограничения** | Не изменяет исходные данные в графе. Не удаляет дубликаты (это делает Correlator). |
   205→
   206→Finding Normalizer решает проблему гетерогенности данных. Каждый сканирующий движок формирует находки в собственном формате: Nuclei использует свой severity-маппинг, Browser Intelligence оперирует DOM-специфичными терминами, HTTP Intelligence работает с заголовками и TLS-параметрами. Normalizer приводит все находки к единой модели, которая становится основой для последующего анализа.
   207→
   208→Процесс нормализации включает: маппинг severity на единую шкалу (Critical/High/Medium/Low/Info), присвоение CWE-классификации при отсутствии, унификацию формата URL, извлечение и стандартизацию Evidence, обогащение метаданными из связанных узлов графа (Host, Endpoint, Technology), формирование единого идентификатора Finding ID на основе детерминированного хеша.
   209→
   210→### 7.2 Finding Correlator
   211→
   212→| Атрибут | Значение |
   213→|---------|----------|
   214→| **Назначение** | Выявление связей между находками и устранение дубликатов |
   215→| **Ответственность** | Группировка находок по корреляционным правилам, дедупликация, формирование Correlation-объектов |
   216→| **Входные данные** | NormalizedFinding[] из Finding Normalizer |
   217→| **Выходные данные** | Correlation[] -- массив выявленных связей, DeduplicationResult -- информация об устранённых дубликатах |
   218→| **Взаимодействие** | Finding Normalizer (чтение), Identity Resolution (чтение), Risk Engine (запись), Confidence Engine (запись) |
   219→| **Ограничения** | Не изменяет NormalizedFinding. Создаёт новые Correlation-объекты. |
   220→
   221→Finding Correlator -- ключевой компонент, который превращает набор изолированных находок в связную модель. Корреляция выполняется по множеству измерений: URL, Host, Endpoint, CWE, CVE, Technology, Shared Evidence, Shared Identity, Shared Authentication. Правила корреляции настраиваемые и расширяемые через CorrelationRule Registry.
   222→
   223→### 7.3 Identity Resolution
   224→
   225→| Атрибут | Значение |
   226→|---------|----------|
   227→| **Назначение** | Разрешение идентичности -- определение, относятся ли разные находки к одному и тому же объекту |
   228→| **Ответственность** | Слияние дублирующихся сущностей, формирование Identity Map, отслеживание Provenance |
   229→| **Входные данные** | NormalizedFinding[], Knowledge Graph nodes |
   230→| **Выходные данные** | IdentityMap -- маппинг Finding → Canonical Entity, MergedEntity[] |
   231→| **Взаимодействие** | Finding Normalizer (чтение), Finding Correlator (чтение/запись) |
   232→| **Ограничения** | Не удаляет данные. Создаёт маппинги и объединённые сущности. |
   233→
   234→Identity Resolution решает фундаментальную проблему: один и тот же эндпоинт может быть обнаружен тремя разными источниками с разными URL-форматами (`https://example.com/api`, `https://example.com:443/api`, `example.com/api/`). Identity Resolution определяет, что все три записи относятся к одному объекту, и создаёт единую каноническую сущность.
   235→
   236→### 7.4 Evidence Aggregator
   237→
   238→| Атрибут | Значение |
   239→|---------|----------|
   240→| **Назначение** | Агрегация доказательств из разных источников для каждой находки |
   241→| **Ответственность** | Объединение Evidence, оценка качества доказательств, формирование EvidenceBundle |
   242→| **Входные данные** | NormalizedFinding[], Knowledge Graph edges и связанные nodes |
   243→| **Выходные данные** | EvidenceBundle[] -- агрегированные наборы доказательств с оценкой качества |
   244→| **Взаимодействие** | Finding Normalizer (чтение), Risk Engine (запись), Confidence Engine (запись) |
   245→| **Ограничения** | Не оценивает риск -- только собирает и классифицирует доказательства. |
   246→
   247→Evidence Aggregator собирает все доказательства, относящиеся к конкретной находке или группе коррелированных находок, из разных источников. Для SQL Injection на `/api/users` агрегатор соберёт: HTTP-запрос/ответ от Nuclei, DOM-снимок от Browser Intelligence, информацию об аутентификации от HTTP Intelligence, данные о технологии от Discovery.
   248→
   249→### 7.5 Risk Engine
   250→
   251→| Атрибут | Значение |
   252→|---------|----------|
   253→| **Назначение** | Детерминированный расчёт агрегированного риска |
   254→| **Ответственность** | Вычисление Risk Score на основе множества параметров, формирование RiskAssessment |
   255→| **Входные данные** | Correlation[], EvidenceBundle[], NormalizedFinding[] |
   256→| **Выходные данные** | RiskAssessment[] -- оценка риска для каждой находки/группы |
   257→| **Взаимодействие** | Finding Correlator (чтение), Evidence Aggregator (чтение), Confidence Engine (запись), Attack Path Builder (запись) |
   258→| **Ограничения** | Не изменяет Finding. Создаёт независимую оценку RiskAssessment. |
   259→
   260→Risk Engine не заменяет CVSS -- он дополняет его контекстной информацией из графа знаний. Математическая модель риска подробно описана в разделе 12.
   261→
   262→### 7.6 Confidence Engine
   263→
   264→| Атрибут | Значение |
   265→|---------|----------|
   266→| **Назначение** | Расчёт достоверности каждого вывода |
   267→| **Ответственность** | Вычисление Confidence Score на основе качества и разнообразия доказательств |
   268→| **Входные данные** | RiskAssessment[], EvidenceBundle[], Correlation[] |
   269→| **Выходные данные** | ConfidenceScore[] -- оценка достоверности для каждого вывода |
   270→| **Взаимодействие** | Risk Engine (чтение), Evidence Aggregator (чтение), Recommendation Engine (запись) |
   271→| **Ограничения** | Не влияет на Risk Score -- только оценивает уверенность в корректности вывода. |
   272→
   273→Confidence Engine реализует multi-factor credibility scoring: количество доказательств (Evidence Count), разнообразие источников (Source Diversity), свежесть данных (Data Freshness), согласованность доказательств (Cross-Validation), полнота покрытия (Coverage Score).
   274→
   275→### 7.7 Attack Path Builder
   276→
   277→| Атрибут | Значение |
   278→|---------|----------|
   279→| **Назначение** | Построение цепочек эксплуатации от точки входа до критического актива |
   280→| **Ответственность** | Поиск путей в графе, оценка осуществимости, ранжирование по риску |
   281→| **Входные данные** | RiskAssessment[], Knowledge Graph (nodes + edges) |
   282→| **Выходные данные** | AttackPath[] -- массив цепочек эксплуатации с оценками |
   283→| **Взаимодействие** | Risk Engine (чтение), Knowledge Graph Adapter (чтение), Recommendation Engine (запись) |
   284→| **Ограничения** | Максимальная глубина поиска -- 10 шагов. Не моделирует атаки -- только анализирует существующие связи. |
   285→
   286→Attack Path Builder использует модифицированный алгоритм DFS/BFS для поиска путей в графе знаний. Подробное описание алгоритма -- в разделе 13.
   287→

   289→
   290→| Атрибут | Значение |
   291→|---------|----------|

   293→| **Ответственность** | Формирование рекомендаций, дедупликация, группировка, категоризация, связь с Attack Paths |
   294→| **Входные данные** | RiskAssessment[], AttackPath[], ConfidenceScore[] |
   295→| **Выходные данные** | Recommendation[] -- массив рекомендаций с приоритетами |
   296→| **Взаимодействие** | Risk Engine (чтение), Attack Path Builder (чтение), Confidence Engine (чтение), Explainability Adapter (запись) |
   297→| **Ограничения** | Работает после Correlation. Не предлагает нерелевантные рекомендации. |

   301→| Атрибут | Значение |
   302→|---------|----------|

   658→### 10.3 Правила формирования графа
   659→
   660→1. **Provenance First:** Каждый узел и связь содержат Provenance -- откуда получены данные, какой движок их обнаружил, когда они были обновлены
   661→2. **Confidence Weighting:** Вес связи умножается на Confidence источника
   662→3. **Identity Merging:** Узлы, отождествлённые Identity Resolution, сливаются с сохранением всех свойств и связей
   663→4. **Incremental Updates:** Граф обновляется инкрементально при получении новых артефактов, без полного перестроения
   664→5. **No Orphan Nodes:** Каждый узел должен иметь хотя бы одну связь. Изолированные узлы помечаются как Unconnected

   668→## 11. Correlation Engine
   669→
   670→### 11.1 Correlation Rules
   671→
   672→| # | Правило | Описание | Вес | Приоритет |
   673→|---|---------|----------|-----|-----------|
   674→| 1 | Same URL | Находки на одном URL | 1.0 | P0 |
   675→| 2 | Same Host | Находки на одном хосте | 0.8 | P1 |
   676→| 3 | Same Endpoint | Находки на одном эндпоинте (с учётом метода) | 0.95 | P0 |
   677→| 4 | Same CVE | Находки с одинаковым CVE | 0.9 | P0 |
   678→| 5 | Same CWE | Находки с одинаковым CWE | 0.7 | P2 |
   679→| 6 | Same Technology | Находки в одной технологии | 0.6 | P2 |
   680→| 7 | Same Component | Находки в одном компоненте/сервисе | 0.7 | P1 |
   681→| 8 | Shared Evidence | Находки с общими доказательствами | 0.85 | P1 |
   682→| 9 | Shared Identity | Находки с общей идентичностью | 0.75 | P1 |
   683→| 10 | Shared Authentication | Находки с общим механизмом аутентификации | 0.8 | P1 |
   684→
   685→### 11.2 Алгоритм корреляции
   686→
   687→```
   688→FUNCTION correlate(findings: NormalizedFinding[]) → Correlation[]:
   689→    correlations = []
   690→    
   691→    # Step 1: Группировка по правилам
   692→    FOR EACH rule IN correlation_rules (ORDERED BY priority DESC):
   693→        groups = GROUP findings BY rule.key_extractor
   694→        FOR EACH group IN groups WHERE len(group) > 1:
   695→            correlation = CREATE Correlation(
   696→                type = rule.type,
   697→                entity_ids = [f.id FOR f IN group],
   698→                strength = rule.weight * compute_group_cohesion(group),
   699→                rule_id = rule.id,
   700→                confidence = min(f.confidence FOR f IN group)
   701→            )
   702→            correlations.APPEND(correlation)
   703→    
   704→    # Step 2: Транзитивная корреляция
   705→    # Если A коррелирует с B, а B с C, то A косвенно коррелирует с C
   706→    transitive = COMPUTE_TRANSITIVE_CLOSURE(correlations, max_depth=3)
   707→    correlations.EXTEND(transitive)
   708→    
   709→    # Step 3: Устранение ложных совпадений
   710→    correlations = FILTER(correlations, c => c.strength >= MIN_CORRELATION_THRESHOLD)
   711→    correlations = DEDUPLICATE(correlations)
   712→    
   713→    RETURN correlations
   714→```
   715→
   716→### 11.3 Устранение ложных совпадений
   717→
   718→1. **Минимальный порог силы:** Корреляции со strength < 0.3 отбрасываются
   719→2. **Cross-validation:** Корреляция подтверждается, если она выявлена минимум двумя независимыми правилами
   720→3. **Контекстная фильтрация:** Корреляции между находками на разных хостах с разной технологией помечаются как WeakCorrelation
   721→4. **Evidence-based verification:** Корреляции без подтверждающих Evidence получают пониженный Confidence

   731→| Severity | 0.0–1.0 | 0.25 | Базовая критичность уязвимости |
   732→| Confidence | 0.0–1.0 | 0.15 | Достоверность находки |
   733→| Exploitability | 0.0–1.0 | 0.20 | Осуществимость эксплуатации |
   734→| Exposure | 0.0–1.0 | 0.15 | Степень раскрытия (публичный/внутренний) |
   735→| Reachability | 0.0–1.0 | 0.10 | Достижимость от точки входа |
   736→| Business Criticality | 0.0–1.0 | 0.10 | Бизнес-критичность затронутого актива |
   737→| Evidence Quality | 0.0–1.0 | 0.05 | Качество доказательств |
   738→
   739→### 12.2 Математическая модель
   740→
   741→```
   742→RiskScore = Σ(w_i x p_i) x AdjustmentFactors
   743→
   744→где:
   745→  w_i -- вес параметра i
   746→  p_i -- нормализованное значение параметра i
   747→  AdjustmentFactors = f(correlation_count, attack_path_count)
   748→
   749→Детализация:
   750→
   751→  BaseScore = 0.25xS + 0.15xC + 0.20xE + 0.15xX + 0.10xR + 0.10xB + 0.05xQ
   752→
   753→  CorrelationBoost = 1.0 + min(0.3, 0.1 x log(1 + correlation_count))
   754→  AttackPathBoost = 1.0 + min(0.4, 0.15 x attack_path_count)
   755→
   756→  RiskScore = clamp(BaseScore x CorrelationBoost x AttackPathBoost, 0.0, 1.0)
   757→
   758→  RiskLevel:
   759→    Critical  -- RiskScore >= 0.80
   760→    High      -- RiskScore >= 0.60
   761→    Medium    -- RiskScore >= 0.40
   762→    Low       -- RiskScore >= 0.20
   763→    Info      -- RiskScore <  0.20
   764→```
   765→
   766→### 12.3 Расчёт параметров
   767→
   768→**Exploitability** определяется на основе: доступности публичного эксплойта (CVE database), сложности эксплуатации (Access Vector, Attack Complexity из CVSS), требований к аутентификации (None → 1.0, Single → 0.6, Multi → 0.2), взаимодействия с пользователем (None → 1.0, Required → 0.5).
   769→
   770→**Exposure** рассчитывается как: публичный эндпоинт = 1.0, внутренний с аутентификацией = 0.4, localhost-only = 0.1.
   771→
   772→**Reachability** определяется через анализ графа: существует ли путь от публичного узла до данной находки. Reachability = 1.0 если публично достижимо, 0.5 если достижимо через аутентификацию, 0.1 если изолировано.
   773→
   774→**Business Criticality** берётся из метаданных актива: если актив помечен как Critical → 1.0, High → 0.7, Medium → 0.4, Low → 0.1.
   775→

   778→## 13. Attack Path Builder
   779→
   780→### 13.1 Алгоритм
   781→
   782→```mermaid
   783→graph TD
   784→    START[Старт: Публичные эндпоинты] --> DISCOVER[Обнаружить исходящие связи]
   785→    DISCOVER --> FILTER{Фильтр: только эксплуатируемые связи?}
   786→    FILTER -->|Да| TRAVERSE[Обход: DFS/BFS]
   787→    FILTER -->|Нет| PRUNE[Отсечение низковероятных путей]
   788→    PRUNE --> TRAVERSE
   789→    TRAVERSE --> CHECK{Целевой актив достигнут?}
   790→    CHECK -->|Да| RECORD[Записать Attack Path]
   791→    CHECK -->|Нет| DEPTH{Глубина < 10?}
   792→    DEPTH -->|Да| DISCOVER
   793→    DEPTH -->|Нет| ABORT[Прекратить обход]
   794→    RECORD --> RANK[Ранжировать по Risk Score]
   795→    RANK --> OUTPUT[AttackPath[]]
   796→```
   797→
   798→### 13.2 Формальные правила
   799→
   800→1. **Точка входа** -- любой публичный узел (Exposure >= 0.8) или узел с Finding
   801→2. **Целевой актив** -- узел с Business Criticality >= 0.7 или содержащий Secret
   802→3. **Допустимые рёбра** -- LEADS_TO, EXPOSES, AFFECTS, USES, CALLS, DEPENDS_ON
   803→4. **Максимальная глубина** -- 10 шагов
   804→5. **Минимальный Confidence пути** -- средний Confidence всех рёбер >= 0.3
   805→6. **Обнаружение циклов** -- если узел уже посещён в текущем пути, обход прекращается
   806→
   807→### 13.3 Приоритизация
   808→
   809→```
   810→PathPriority = PathRiskScore x PathFeasibility / PathLength
   811→
   812→где:
   813→  PathRiskScore = max(step.risk_score FOR step IN path)
   814→  PathFeasibility = geometric_mean(step.feasibility FOR step IN path)
   815→  PathLength = len(path.steps)
   816→```
   817→
   818→Короткие пути с высоким риском и высокой осуществимостью получают приоритет. Длинные пути с низким риском -- низкий приоритет.
   819→
   820→### 13.4 Альтернативные сценарии
   821→
   822→Для каждой пары (entry_point, target) может существовать несколько альтернативных путей. Attack Path Builder сохраняет все найденные пути, но ранжирует их по приоритету. Альтернативные пути группируются и предоставляются как AlternativeAttackPaths в SecurityStory.
   823→

   826→## 14. Recommendation Engine
   827→
   828→### 14.1 Типы рекомендаций
   829→
   830→| Тип | Описание | Критерий |
   831→|-----|----------|----------|
   832→| **Fix** | Непосредственное исправление | Для Finding с Confidence >= 0.7 |
   833→| **Mitigate** | Смягчение последствий | Для Attack Path с Feasibility >= 0.5 |
   834→| **Monitor** | Мониторинг изменений | Для Finding с Confidence < 0.5 |
   835→| **Accept** | Принятие риска | Для Risk Score < 0.2 |
   836→
   837→### 14.2 Генерация рекомендаций
   838→
   839→Рекомендации генерируются на основе трёх источников: CWE Remedy Database (стандартные рекомендации для каждого CWE), Attack Path Break (разрыв цепочки атаки -- какая рекомендация разорвёт самый критический путь), и Correlation Insight (рекомендации на основе корреляций -- исправление одного корня устраняет несколько находок).
   840→
   841→### 14.3 Дедупликация
   842→
   843→Две рекомендации считаются дубликатами, если: они относятся к одному CWE, затрагивают один и тот же актив и предлагают одно и то же действие. Дубликаты сливаются с увеличением priority на один уровень.
   844→
   845→### 14.4 Группировка
   846→
   847→Рекомендации группируются по: категории (Auth, Config, Patch, Network, Data), затронутому активу, Attack Path (рекомендации, разрывающие один и тот же путь, группируются вместе), приоритету.
   848→
   849→### 14.5 Категоризация и приоритет
   850→
   851→```
   852→Priority = P0  IF RiskScore >= 0.8 AND Confidence >= 0.7  → Immediate action
   853→Priority = P1  IF RiskScore >= 0.6 AND Confidence >= 0.5  → Fix within sprint
   854→Priority = P2  IF RiskScore >= 0.4 AND Confidence >= 0.4  → Plan for next sprint
   855→Priority = P3  IF RiskScore < 0.4                          → Backlog
   856→```

  1056→## 18. Failure Handling
  1057→
  1058→### 18.1 Стратегия обработки ошибок
  1059→
  1060→| Сценарий | Стратегия | Поведение |
  1061→|----------|-----------|-----------|
  1062→| KG Adapter не может загрузить данные | Retry with backoff | 3 попытки с экспоненциальной задержкой, затем AnalysisFailed |
  1063→| Finding Normalizer встречает неизвестный формат | Skip and log | Пропуск находки с логированием, анализ продолжается |
  1064→| Correlation Engine timeout | Partial results | Возврат корреляций, найденных до timeout |
  1065→| Risk Engine получает неполные данные | Degraded calculation | Расчёт с доступными параметрами, пониженный Confidence |
  1066→| Attack Path Builder: циклы в графе | Cycle detection | Обнаружение и пропуск циклов при обходе |
  1067→| Recommendation Engine: нет CWE | Generic recommendations | Использование generic-рекомендаций вместо CWE-specific |
  1068→| Полный сбой компонента | Circuit breaker | После 3 последовательных сбоев -- Circuit Open, fallback на cached results |
  1069→
  1070→### 18.2 Гарантии
  1071→
  1072→1. **No Data Loss:** Сбой одного компонента не приводит к потере данных предыдущих этапов
  1073→2. **Partial Results:** При невозможности полного анализа возвращаются частичные результаты с пометкой CompletenessLevel
  1074→3. **Idempotency:** Повторный вызов analyze() с теми же входными данными даёт тот же результат
  1075→4. **Auditability:** Каждый сбой логируется с полным контекстом для диагностики
  1076→

  1079→## 19. Scalability
  1080→
  1081→### 19.1 Горизонтальное масштабирование
  1082→
  1083→SIE проектируется с учётом горизонтального масштабирования. Каждый компонент является stateless (состояние хранится в Knowledge Graph), что позволяет запускать несколько экземпляров одного компонента параллельно.
  1084→
  1085→**Finding Normalizer:** Может обрабатывать находки параллельно (каждая находка независима). Параллелизм: N workers на N finding batches.
  1086→
  1087→**Finding Correlator:** Требует доступа ко всем находкам для корреляции. Стратегия: Map-Reduce -- map-фаза распределяет находки по группам, reduce-фаза объединяет корреляции.
  1088→
  1089→**Risk Engine:** Расчёт риска для разных находок независим. Параллелизм: N workers.
  1090→
  1091→**Attack Path Builder:** Каждый путь строится независимо. Параллелизм: N workers для разных entry points.
  1092→
  1093→### 19.2 Инкрементальный анализ
  1094→
  1095→При обновлении графа (новые артефакты) SIE не выполняет полный повторный анализ. Вместо этого:
  1096→
  1097→1. Определяются затронутые подграфы (affected subgraphs)
  1098→2. Перерасчёт выполняется только для затронутых сущностей
  1099→3. Результаты, не затронутые изменениями, используются из кэша
  1100→
  1101→Это критично для Continuous Security, где анализ может выполняться ежедневно или при каждом коммите.
  1102→

  1105→## 20. Security Considerations
  1106→
  1107→1. **Read-Only Access:** SIE имеет read-only доступ к Knowledge Graph через Knowledge Graph Adapter. Никакие изменения в графе не производятся.
  1108→2. **No Secret Exposure:** SIE не логирует содержимое Evidence, которое может содержать секреты (JWT, cookies, API keys). Все логирование -- по ID и типу.
  1109→3. **Deterministic Results:** Отсутствие недетерминированности исключает возможность манипуляции результатами через входные данные.
  1110→4. **Input Validation:** Все входные данные валидируются через Pydantic-модели перед обработкой.
  1111→5. **Audit Trail:** Каждое вычисление содержит Provenance, позволяя отследить происхождение любого вывода.

  1115→## 21. Performance Considerations
  1116→
  1117→### 21.1 Целевые показатели
  1118→
  1119→| Метрика | Целевое значение |
  1120→|---------|-----------------|
  1121→| Полный анализ (1000 findings) | < 30 секунд |
  1122→| Инкрементальный анализ (10 новых findings) | < 5 секунд |
  1123→| Attack Path Builder (граф 10K узлов) | < 10 секунд |
  1124→| Risk Calculation (1000 findings) | < 5 секунд |
  1125→| API Response Time (cached) | < 100 ms |
  1126→
  1127→### 21.2 Оптимизации
  1128→
  1129→1. **Кэширование:** Результаты нормализации и корреляции кэшируются. При повторном анализе используются cached results для неизменённых данных.
  1130→2. **Batch Processing:** Finding Normalizer обрабатывает находки батчами для снижения overhead.
  1131→3. **Lazy Loading:** Knowledge Graph Adapter загружает подграфы по требованию, а не весь граф.
  1132→4. **Parallel Execution:** Компоненты, не зависящие друг от друга, выполняются параллельно (например, Evidence Aggregator и Identity Resolution).
  1133→5. **Index-Based Lookup:** Graph Query API использует индексы для быстрого поиска узлов по типу, свойствам и связям.

  1139→### ADR-INT-001: Knowledge Graph как центральная модель данных
  1140→
  1141→**Контекст:** SIE может читать данные напрямую из Artifact Bus или через Knowledge Graph.  
  1142→**Решение:** SIE читает данные исключительно через Knowledge Graph.  
  1143→**Обоснование:** Единый источник истины (CTO Decision #3). KG обеспечивает нормализацию, Identity Resolution и Provenance, которые критичны для корректного анализа. Прямое чтение из Artifact Bus привело бы к дублированию логики нормализации и потенциальной рассинхронизации данных.  
  1144→**Последствия:** SIE зависит от доступности KG. Если KG недоступен, анализ невозможен.  
  1145→**Статус:** ACCEPTED
  1146→

  1149→### ADR-INT-002: Intelligence Engine -- immutable
  1150→
  1151→**Контекст:** Результаты анализа могут обновляться при получении новых данных.  
  1152→**Решение:** Все результаты SIE -- immutable. При обновлении создаётся новая версия с новой меткой времени.  
  1153→**Обоснование:** Immutable Architecture гарантирует аудитоспособность, воспроизводимость и отсутствие побочных эффектов (CTO Decision #7). Позволяет сравнивать результаты разных прогонов анализа.  
  1154→**Последствия:** Увеличение объёма хранимых данных. Требуется стратегия garbage collection для устаревших версий.  
  1155→**Статус:** ACCEPTED
  1156→
  1157→---
  1158→
  1159→### ADR-INT-003: Explainability вынесен в отдельный Adapter
  1160→
  1161→**Контекст:** Объяснимость результатов может быть встроена в каждый компонент SIE или вынесена отдельно.  
  1162→**Решение:** Explainability Adapter -- отдельный компонент, который собирает данные от всех компонентов SIE и формирует ExplainabilityOutput.  
  1163→**Обоснование:** Разделение ответственности (Single Responsibility Principle). Компоненты SIE фокусируются на анализе, а Explainability Adapter -- на структурировании данных для объяснения. Это позволяет заменять или модернизировать Explainability Engine независимо от SIE.  
  1164→**Последствия:** Компоненты SIE должны явно предоставлять трассировочные данные через Provenance и TraceStep.  
  1165→**Статус:** ACCEPTED
  1166→
  1167→---
  1168→
  1169→### ADR-INT-004: Recommendation Engine работает после Correlation и Risk
  1170→
  1171→**Контекст:** Рекомендации могут генерироваться на разных этапах анализа.  
  1172→**Решение:** Recommendation Engine получает данные только после завершения Correlation, Risk Calculation и Attack Path Building.  
  1173→**Обоснование:** Качественные рекомендации требуют полного контекста: какие находки коррелируют, каков их реальный риск, какие Attack Paths существуют. Генерация рекомендаций до корреляции привела бы к дублированию и неточностям.  
  1174→**Последствия:** Recommendation Engine -- финальный этап пайплайна. Сбой на ранних этапах задерживает генерацию рекомендаций.  
  1175→**Статус:** ACCEPTED
  1176→

  1179→### ADR-INT-005: Risk Engine не изменяет Finding
  1180→
  1181→**Контекст:** Risk Engine может обновлять severity находок на основе контекста.  
  1182→**Решение:** Risk Engine создаёт независимый RiskAssessment, не модифицируя исходные Finding.  
  1183→**Обоснование:** Immutable Models (CTO Decision #7). Исходная находка сохраняет свою целостность -- пользователь видит и оригинальную severity от сканера, и контекстную оценку риска от SIE. Это предотвращает потерю информации и обеспечивает прозрачность.  
  1184→**Последствия:** Потребители API должны использовать RiskAssessment.risk_level, а не Finding.severity для приоритизации.  
  1185→**Статус:** ACCEPTED

  1189→### ADR-INT-006: Knowledge Graph Adapter как Anti-Corruption Layer
  1190→
  1191→**Контекст:** SIE может напрямую использовать Graph Query API или через адаптер.  
  1192→**Решение:** Knowledge Graph Adapter инкапсулирует все обращения к Graph Query API, предоставляя стабильный интерфейс для компонентов SIE.  
  1193→**Обоснование:** При смене бэкенда графа (NetworkX → Neo4j) изменения коснутся только адаптера, а не всех компонентов SIE. Это реализует паттерн Anti-Corruption Layer из DDD.  
  1194→**Последствия:** Адаптер добавляет тонкий слой абстракции, но изолирует SIE от инфраструктурных изменений.  
  1195→**Статус:** ACCEPTED

  1199→### ADR-INT-007: Детерминированный расчёт Confidence
  1200→
  1201→**Контекст:** Confidence может рассчитываться через статистические модели или детерминированные алгоритмы.  
  1202→**Решение:** Confidence Engine использует детерминированный multi-factor scoring без вероятностных моделей.  
  1203→**Обоснование:** CTO Decision #6 -- все аналитические компоненты детерминированы. Статистические модели (Bayesian, Monte Carlo) вносят недетерминированность, нарушая воспроизводимость результатов.  
  1204→**Последствия:** Confidence Score -- взвешенная сумма наблюдаемых факторов, а не вероятность. Пользователи должны понимать разницу.  
  1205→**Статус:** ACCEPTED

  1209→## 23. Open Questions
  1210→
  1211→1. **Версионирование графа:** Как управлять версиями Knowledge Graph при инкрементальных обновлениях? Полное перестроение vs. snapshot-изоляция?
  1212→2. **Корреляция между приложениями:** Должен ли SIE поддерживать корреляцию находок между разными приложениями (multi-tenant)?
  1213→3. **Настраиваемые веса Risk Engine:** Должны ли пользователи иметь возможность переопределять веса параметров Risk Score? Если да, как обеспечить воспроизводимость при разных конфигурациях?
  1214→4. **Максимальный размер графа:** Какова верхняя граница размера Knowledge Graph, при которой SIE сохраняет приемлемую производительность? Когда нужен переход на распределённый граф (Neo4j cluster)?
  1215→5. **Cache Invalidation:** Когда и как инвалидируются кэши SIE при обновлении графа? Стратегия TTL vs. event-driven invalidation?
  1216→6. **External Rule Registry:** Должны ли правила корреляции и правила риска храниться в коде или во внешнем хранилище (конфигурация, БД)?
  1217→7. **Attack Path Visualization:** Как визуализировать альтернативные Attack Paths в Dashboard? Treemap vs. Sankey vs. Graph?
