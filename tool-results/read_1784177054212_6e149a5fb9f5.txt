    36→## 1. Executive Summary
    37→
    38→Knowledge Graph (KG) — независимая подсистема платформы, выступающая единым источником знаний о безопасности приложения. В отличие от традиционных хранилищ находок, KG хранит не отдельные результаты сканирования, а **отношения между объектами безопасности** — от эндпоинтов и хостов до уязвимостей, цепочек атак и рекомендаций. Это фундаментальное различие превращает KG из простого хранилища данных в активную модель знаний, на которой строятся все аналитические возможности платформы.
    39→
    40→RFC-001 (Security Intelligence Engine) утвердил, что SIE будет использовать Knowledge Graph как центральную модель данных. Однако архитектура самого графа до сих пор не определена — существует лишь концептуальное описание 12 Node Types и 9 Edge Types в рамках SIE-спецификации. Данный RFC закрывает этот пробел, определяя полноценную архитектуру Graph Platform: от внутренней модели хранения и обхода до версионирования, синхронизации и API.
    41→
    42→Архитектура KG строится на восьми фундаментальных принципах: Immutable Data Model (неизменяемость узлов и рёбер), Event Driven Updates (обновления только через события), Incremental Synchronization (инкрементальное обновление без полного перестроения), Versioned Snapshots (снимки состояния графа), Graph-first Architecture (все данные моделируются как граф), Engine Agnostic (независимость от конкретной графовой СУБД), Explainability Ready (готовность к объяснению любого вывода) и Horizontal Scalability (горизонтальное масштабирование). Эти принципы гарантируют, что KG станет надёжным фундаментом для Security Intelligence Engine, Risk Engine, Attack Path Builder, Recommendation Engine и будущих AI-модулей.

    46→## 2. Problem Statement
    47→
    48→Текущая платформа успешно собирает данные из множества источников (Discovery Engine, Browser Intelligence, HTTP Intelligence, Nuclei), но данные остаются разрозненными. Каждая находка существует как изолированный артефакт, без связи с другими объектами безопасности. RFC-001 определил, как SIE будет анализировать данные графа, но сам граф ещё не спроектирован. Это создаёт ряд критических проблем.
    49→
    50→**Отсутствие модели хранения.** Нет определения, как именно узлы и рёбра хранятся, индексируются и извлекаются. Без чёткой модели хранения невозможно оценить производительность и масштабируемость решения при росте объёма данных. При сканировании среднего приложения генерируется от 500 до 5000 находок, каждая из которых связана с 3-10 объектами — это значит, что граф типичного приложения содержит от 2000 до 50000 узлов и от 5000 до 150000 рёбер.
    51→
    52→**Отсутствие механизма обхода.** SIE требует обхода графа для построения Attack Paths, расчёта Reachability и выполнения корреляций. Без специализированного Traversal Engine каждый потребитель будет реализовывать обход самостоятельно, что приведёт к дублированию кода, несогласованным результатам и проблемам с производительностью. Алгоритм BFS для графа с 50000 узлов и 150000 рёбрами может выполняться от нескольких миллисекунд до нескольких секунд в зависимости от реализации — разница в 1000 раз.
    53→
    54→**Отсутствие версионирования.** Граф обновляется при каждом сканировании. Без механизма версионирования невозможно отследить, как изменилась поверхность атаки, воспроизвести результаты предыдущего анализа или откатиться к предыдущему состоянию при ошибке. Инкрементальное обновление без версионирования — это односторонний процесс без точки возврата.
    55→
    56→**Отсутствие синхронизации.** Несколько потребителей (SIE, Explainability Engine, Dashboard) нуждаются в актуальных данных графа. Без чёткой модели синхронизации каждый потребитель будет опрашивать граф самостоятельно, создавая избыточную нагрузку и рассинхронизацию данных. Событийная модель, уже принятая платформой (Artifact Bus), должна быть расширена на Graph Events.
    57→
    58→**Зависимость от конкретной СУБД.** Без Storage Agnostic Architecture выбор конкретной графовой базы данных (Neo4j, ArangoDB, NetworkX) станет архитектурным решением, которое невозможно изменить без переписывания всего аналитического слоя. Это противоречит CTO Decision #4 (Plugin Architecture) и Engine Agnostic принципу.

    62→## 3. Goals
    63→
    64→1. **Единый источник знаний** — KG становится SSOT (Single Source of Truth) для всех данных о безопасности: Finding, Asset, Evidence, Relationship, Attack Path и Recommendation не существуют вне графа
    65→2. **Immutable Graph Model** — узлы и рёбра неизменяемы после создания; обновление создаёт новую версию, а не модифицирует существующую
    66→3. **Engine Agnostic Storage** — архитектура не зависит от конкретной графовой СУБД; замена бэкенда (NetworkX -> Neo4j -> ArangoDB) не требует изменений в SIE, API или потребителях
    67→4. **Versioned Snapshots** — каждое значимое изменение графа фиксируется как Snapshot; поддерживается Delta, Rollback, Replay и Incremental Update
    68→5. **Deterministic Traversal** — обход графа (BFS, DFS, Shortest Path) всегда даёт детерминированный результат для одного и того же графа и одинаковых параметров
    69→6. **Event Driven Synchronization** — обновления графа публикуются как события через Graph Event Bus; потребители подписываются на нужные типы событий
    70→7. **Explainability Ready** — каждый узел, ребро и путь содержит Provenance; любой вывод может быть трассирован до исходных данных
    71→8. **Zero Coupling** — KG не зависит от существующих модулей (TASK-201, TASK-202, Pipeline Executor, SIE); изменения в KG не требуют модификации других подсистем
    72→9. **Performance by Design** — архитектура закладывает кэширование, индексирование, lazy loading и graph partitioning для обеспечения производительности при масштабировании
    73→

    76→## 4. Non Goals
    77→
    78→1. **Не реализация** — данный RFC определяет архитектуру, а не код; реализация будет выполнена в INT-001
    79→2. **Не визуализация** — рендеринг и визуализация графа (Dashboard) не входят в зону ответственности KG Platform
    80→3. **Не аналитика** — KG хранит данные и обеспечивает обход, но не выполняет корреляцию, оценку рисков или построение Attack Paths (это ответственность SIE)
    81→4. **Не выбор конкретной СУБД** — архитектура остаётся storage-agnostic; конкретный бэкенд будет выбран при реализации INT-001
    82→5. **Не modification существующих модулей** — TASK-201, TASK-202, Pipeline Executor, Scan Platform Foundation, RFC-001, Explainability Engine остаются неизменными
    83→6. **Не real-time streaming** — KG работает в batch/incremental режиме; streaming-обновления поддерживаются через Event Bus, но не как primary pattern
    84→7. **Не распределённая согласованность** — на данном этапе не рассматривается multi-datacenter репликация; горизонтальное масштабирование закладывается через partitioning, а не через distributed consensus

   245→Архитектура Knowledge Graph Platform состоит из трёх основных слоёв: Ingestion, Graph Platform и Storage. Каждый слой имеет чёткую ответственность и взаимодействует с соседними через определённые контракты.
   246→
   247→**Ingestion Layer** отвечает за приём данных из Artifact Bus, нормализацию артефактов и построение узлов/рёбер графа. Normalizer Filter преобразует артефакты разных форматов (Nuclei JSON, Browser Intel, HTTP Intel) в единый внутренний формат GraphBuilder. Graph Builder создаёт и обновляет узлы и рёбра, обновляет индексы и публикует Graph Events.
   248→
   249→**Graph Platform Layer** предоставляет все операции над графом: запросы (Graph Query API), обход (Traversal Engine), версионирование (Version Manager), синхронизацию (Sync Manager) и события (Graph Event Bus). Это единственный слой, доступный потребителям.
   250→
   251→**Storage Layer** обеспечивает физическое хранение узлов, рёбер, индексов, кэша, снимков и дельт. Storage Layer абстрагирован через интерфейсы, что позволяет заменять конкретную реализацию (NetworkX, Neo4j, ArangoDB) без изменения верхних слоёв.

   375→        +sort: List~Sort~
   376→    }
   377→
   378→    class GraphTransaction {
   379→        +transaction_id: str
   380→        +operations: List~Operation~
   381→        +status: TransactionStatus
   382→        +started_at: datetime
   383→        +committed_at: datetime
   384→    }
   385→
   386→    class GraphEvent {
   387→        +event_id: str
   388→        +event_type: GraphEventType
   389→        +payload: Dict
   390→        +timestamp: datetime
   391→        +source: str
   392→        +version: int
   393→    }
   394→
   395→    GraphNode --> NodeIdentity
   396→    GraphNode --> NodeMetadata
   397→    GraphEdge --> NodeIdentity : source
   398→    GraphEdge --> NodeIdentity : target
   399→    GraphEdge --> Relationship
   400→    Relationship --> RelationshipType
   401→    Traversal --> Subgraph
   402→    GraphSnapshot --> GraphVersion
   403→    GraphTransaction --> GraphEvent
   404→    Query --> GraphNode : returns
   405→```
   406→
   407→### 7.2 Детальное описание сущностей
   408→
   409→#### 7.2.1 GraphNode
   410→
   411→| Атрибут | Описание |
   412→|---------|----------|
   413→| **Назначение** | Представляет любой объект безопасности в графе: приложение, хост, эндпоинт, уязвимость, доказательство и т.д. |
   414→| **Поля** | `node_id: NodeIdentity` — уникальный идентификатор узла; `node_type: str` — тип узла (Application, Host, Endpoint и др.); `metadata: NodeMetadata` — метаданные узла; `version: int` — номер версии; `created_at: datetime` — время создания; `provenance: Provenance` — происхождение данных |
   415→| **Жизненный цикл** | Created -> Active -> Deprecated -> Archived. Узел создаётся Graph Builder при получении артефакта. Deprecated при обнаружении более актуальной версии. Archived при удалении (soft delete). |
   416→| **Ответственность** | Инкапсуляция данных об объекте безопасности. Предоставление доступа к метаданным и идентичности. Поддержка версионирования. |
   417→| **Связи** | Содержит NodeIdentity и NodeMetadata. Является source/target для GraphEdge. Входит в Subgraph. Версионируется через GraphVersion. |
   418→
   419→#### 7.2.2 GraphEdge
   420→
   421→| Атрибут | Описание |
   422→|---------|----------|
   423→| **Назначение** | Представляет направленную связь между двумя узлами графа с семантикой, весом и ограничениями |
   424→| **Поля** | `edge_id: str` — уникальный идентификатор ребра; `source_id: NodeIdentity` — исходный узел; `target_id: NodeIdentity` — целевой узел; `relationship: Relationship` — тип и семантика связи; `weight: float` — вес связи [0.0, 1.0]; `version: int` — номер версии; `created_at: datetime` — время создания; `provenance: Provenance` — происхождение |
   425→| **Жизненный цикл** | Created -> Active -> Superseded -> Archived. Ребро создаётся при обнаружении связи. Superseded при обнаружении более точной связи. Archived при soft delete. |
   426→| **Ответственность** | Определение направленной связи между узлами. Предоставление семантики и веса связи. Обеспечение ограничений целостности. |
   427→| **Связи** | Связывает два GraphNode (source, target). Содержит Relationship. Входит в Subgraph. Индексируется через GraphIndex. |
   428→
   429→#### 7.2.3 NodeIdentity
   430→
   431→| Атрибут | Описание |
   432→|---------|----------|
   433→| **Назначение** | Уникальная идентичность узла, поддерживающая слияние (merge) при разрешении конфликтов. Identity определяет, являются ли два узла одним и тем же объектом. |
   434→| **Поля** | `identity_type: str` — тип идентичности (url, hostname, cve, cwe, ip); `value: str` — значение идентичности; `namespace: str` — пространство имён (application scope); `hash: str` — детерминированный хеш для быстрого сравнения |
   435→| **Жизненный цикл** | Created -> Resolved -> Merged. Created при первом появлении. Resolved при успешной привязке к узлу. Merged при слиянии с другой идентичностью. |
   436→| **Ответственность** | Уникальная идентификация узла. Поддержка слияния идентичностей. Детерминированное сравнение. |
   437→| **Связи** | Принадлежит GraphNode. Используется в GraphEdge для source/target. Используется в Query для фильтрации. |
   438→
   439→#### 7.2.4 NodeMetadata
   440→
   441→| Атрибут | Описание |
   442→|---------|----------|
   443→| **Назначение** | Метаданные узла: свойства, теги, confidence и источник. Поддерживает инкрементальное обогащение через merge. |
   444→| **Поля** | `properties: Dict` — словарь свойств узла (зависит от node_type); `tags: Set` — множество тегов; `confidence: float` — достоверность данных [0.0, 1.0]; `source: str` — источник данных (nuclei, browser_intel, http_intel, discovery); `updated_at: datetime` — время последнего обновления |
   445→| **Жизненный цикл** | Created -> Enriched -> Frozen. Created при первом заполнении. Enriched при получении дополнительных данных от других источников. Frozen при версионировании (immutable). |
   446→| **Ответственность** | Хранение свойств узла. Поддержка обогащения. Предоставление confidence и provenance. |
   447→| **Связи** | Принадлежит GraphNode. Используется в Traversal для фильтрации. Влияет на Edge weight. |
   448→
   449→#### 7.2.5 Relationship
   450→
   451→| Атрибут | Описание |
   452→|---------|----------|
   453→| **Назначение** | Определяет семантику связи между узлами: тип, направление, ограничения. Relationship — value object, не сущность. |
   454→| **Поля** | `relationship_type: RelationshipType` — тип связи из перечисления; `direction: Direction` — направление (OUTGOING, INCOMING, BIDIRECTIONAL); `semantics: str` — текстовое описание семантики; `constraints: Constraints` — ограничения на допустимые source/target node types |
   455→| **Жизненный цикл** | Value object — не имеет собственного жизненного цикла. Создаётся вместе с GraphEdge. |
   456→| **Ответственность** | Определение семантики связи. Обеспечение типобезопасности (ограничения на node types). |
   457→| **Связи** | Принадлежит GraphEdge. Ссылается на RelationshipType. |
   458→
   459→#### 7.2.6 RelationshipType
   460→
   461→| Атрибут | Описание |
   462→|---------|----------|
   463→| **Назначение** | Перечисление всех допустимых типов связей в графе. Открытое перечисление — новые типы добавляются без изменения существующих. |
   464→| **Значения** | USES, OWNS, CALLS, DEPENDS_ON, HOSTS, CONNECTED_TO, LEADS_TO, DISCOVERED_BY, EXPOSES, AUTHENTICATES, TRUSTS, CONTAINS, RELATED_TO, MITIGATED_BY |
   465→| **Ответственность** | Контроль допустимых типов связей. Предотвращение опечаток и некорректных типов. |
   466→
   467→#### 7.2.7 Traversal
   468→
   469→| Атрибут | Описание |
   470→|---------|----------|
   471→| **Назначение** | Описание операции обхода графа: алгоритм, стартовые узлы, параметры, результат. Traversal — это команда, которая выполняется Traversal Engine. |
   472→| **Поля** | `traversal_id: str` — уникальный идентификатор обхода; `algorithm: TraversalAlgorithm` — выбранный алгоритм; `start_nodes: List[NodeIdentity]` — стартовые узлы; `parameters: Dict` — параметры алгоритма (max_depth, weight_threshold и др.); `result: Subgraph` — результирующий подграф; `execution_time_ms: float` — время выполнения |
   473→| **Жизненный цикл** | Created -> Executing -> Completed / Failed / Cancelled. Created при инициации обхода. Executing во время выполнения. Completed при успешном завершении. |
   474→| **Ответственность** | Инкапсуляция параметров обхода. Хранение результата. Учёт времени выполнения. |
   475→| **Связи** | Выполняется Traversal Engine. Возвращает Subgraph. Использует GraphIndex для оптимизации. |
   476→
   477→#### 7.2.8 Subgraph
   478→
   479→| Атрибут | Описание |
   480→|---------|----------|
   481→| **Назначение** | Подграф — подмножество узлов и рёбер полного графа. Результат Traversal, Query или Snapshot. |
   482→| **Поля** | `nodes: List[GraphNode]` — узлы подграфа; `edges: List[GraphEdge]` — рёбра подграфа; `root_id: NodeIdentity` — корневой узел; `depth: int` — глубина обхода; `node_count: int` — количество узлов; `edge_count: int` — количество рёбер |
   483→| **Жизненный цикл** | Value object — не имеет собственного жизненного цикла. Создаётся как результат операции. |
   484→| **Ответственность** | Инкапсуляция подмножества графа. Предоставление статистики. |
   485→| **Связи** | Содержит GraphNode и GraphEdge. Возвращается Traversal, Query, Snapshot. |
   486→
   487→#### 7.2.9 GraphSnapshot
   488→
   489→| Атрибут | Описание |
   490→|---------|----------|
   491→| **Назначение** | Неизменяемый снимок состояния графа на определённый момент времени. Используется для воспроизведения анализа, отката и аудита. |
   492→| **Поля** | `snapshot_id: str` — уникальный идентификатор снимка; `version: GraphVersion` — версия графа; `timestamp: datetime` — время создания снимка; `node_count: int` — количество узлов; `edge_count: int` — количество рёбер; `checksum: str` — контрольная сумма для целостности; `delta_from: str` — ID предыдущего снимка (для delta snapshots) |
   493→| **Жизненный цикл** | Created -> Verified -> Archived. Created при создании снимка. Verified после проверки checksum. Archived при очистке старых снимков. |
   494→| **Ответственность** | Фиксация состояния графа. Обеспечение воспроизводимости. Поддержка delta snapshots. |
   495→| **Связи** | Ссылается на GraphVersion. Связан с Delta Store. |
   496→
   497→#### 7.2.10 GraphVersion
   498→
   499→| Атрибут | Описание |
   500→|---------|----------|
   501→| **Назначение** | Метаданные версии графа: автор, операция, описание. Поддерживает линейную историю версий. |
   502→| **Поля** | `version_id: str` — уникальный идентификатор версии; `parent_id: str` — ID родительской версии; `timestamp: datetime` — время создания; `operation: str` — тип операции (create_node, update_node, create_edge и др.); `author: str` — инициатор (system, pipeline, manual); `description: str` — описание изменения |
   503→| **Жизненный цикл** | Created -> Active -> Superseded. Created при фиксации версии. Active при использовании. Superseded при появлении новой версии. |
   504→| **Ответственность** | Отслеживание истории изменений. Поддержка rollback. Обеспечение audit trail. |
   505→| **Связи** | Используется в GraphSnapshot. Связан с Delta Store. |
   506→
   507→#### 7.2.11 GraphIndex
   508→
   509→| Атрибут | Описание |
   510→|---------|----------|
   511→| **Назначение** | Индекс графа для ускорения запросов. Поддерживает несколько типов индексов: adjacency list, reverse index, property index, temporal index. |
   512→| **Поля** | `index_id: str` — уникальный идентификатор индекса; `index_type: IndexType` — тип индекса (ADJACENCY, REVERSE, PROPERTY, TEMPORAL); `field: str` — индексируемое поле; `cardinality: int` — количество уникальных значений; `last_updated: datetime` — время последнего обновления |
   513→| **Жизненный цикл** | Created -> Building -> Active -> Stale -> Rebuilding. Created при создании. Building при первоначальной индексации. Active при использовании. Stale при устаревании. Rebuilding при перестроении. |
   514→| **Ответственность** | Ускорение запросов. Поддержка актуальности. Предоставление статистики. |
   515→| **Связи** | Используется в Traversal Engine. Обновляется Graph Builder. |
   516→
   517→#### 7.2.12 Query
   518→
   519→| Атрибут | Описание |
   520→|---------|----------|
   521→| **Назначение** | Описание поискового запроса к графу: паттерн, фильтры, сортировка, пагинация. Query — это спецификация, а не результат. |
   522→| **Поля** | `query_id: str` — уникальный идентификатор запроса; `pattern: QueryPattern` — паттерн поиска (node_type, edge_type, property match); `filters: List[Filter]` — список фильтров; `limit: int` — ограничение количества результатов; `offset: int` — смещение для пагинации; `sort: List[Sort]` — правила сортировки |
   523→| **Жизненный цикл** | Created -> Executing -> Completed / Failed. Created при создании запроса. Executing во время выполнения. Completed при получении результатов. |
   524→| **Ответственность** | Инкапсуляция параметров поиска. Поддержка пагинации и сортировки. |
   525→| **Связи** | Выполняется через Graph Query API. Возвращает List[GraphNode] или Subgraph. |
   526→
   527→#### 7.2.13 GraphTransaction
   528→
   529→| Атрибут | Описание |
   530→|---------|----------|
   531→| **Назначение** | Транзакция над графом: набор операций (create/update/delete узлов и рёбер), выполняемых атомарно. |
   532→| **Поля** | `transaction_id: str` — уникальный идентификатор транзакции; `operations: List[Operation]` — список операций; `status: TransactionStatus` — статус (PENDING, COMMITTED, ROLLED_BACK); `started_at: datetime` — время начала; `committed_at: datetime` — время фиксации |
   533→| **Жизненный цикл** | Pending -> Committed / RolledBack. Pending при создании. Committed при успешной фиксации. RolledBack при ошибке или отмене. |
   534→| **Ответственность** | Атомарность операций. Поддержка отката. Обеспечение целостности графа. |
   535→| **Связи** | Генерирует GraphEvent при фиксации. Влияет на GraphVersion. |
   536→
   537→#### 7.2.14 GraphEvent
   538→
   539→| Атрибут | Описание |
   540→|---------|----------|
   541→| **Назначение** | Событие изменения графа. Публикуется через Graph Event Bus для уведомления потребителей. |
   542→| **Поля** | `event_id: str` — уникальный идентификатор события; `event_type: GraphEventType` — тип события (NODE_CREATED, NODE_UPDATED, EDGE_CREATED, EDGE_REMOVED, SNAPSHOT_CREATED, GRAPH_VERSION_INCREMENTED); `payload: Dict` — данные события; `timestamp: datetime` — время события; `source: str` — источник (pipeline, manual, system); `version: int` — версия графа на момент события |
   543→| **Жизненный цикл** | Created -> Published -> Delivered. Created при генерации. Published при отправке в Event Bus. Delivered при получении потребителем. |
   544→| **Ответственность** | Уведомление потребителей об изменениях. Поддержка eventual consistency. Обеспечение audit trail. |
   545→| **Связи** | Генерируется GraphTransaction. Публикуется через Graph Event Bus. Потребляется Sync Manager. |

   603→### 8.2 Полный перечень Node Types
   604→
   605→| # | Node Type | Свойства | Описание |
   606→|---|-----------|----------|----------|
   607→| 1 | **Application** | name, url, tech_stack, business_criticality, environment, owner | Корневой узел приложения. Все объекты безопасности принадлежат Application. |
   608→| 2 | **Host** | hostname, ip, os, ports, services, cloud_provider, region | Сервер, виртуальная машина или контейнер-хост. Физическая или логическая единица инфраструктуры. |
   609→| 3 | **Endpoint** | path, method, params, auth_required, content_type, rate_limit, response_codes | HTTP-эндпоинт. Включает путь, метод, параметры и требования аутентификации. |
   610→| 4 | **API** | path, method, params, auth, rate_limit, schema, version, deprecation_status | API-эндпоинт. Отличается от Endpoint наличием формальной схемы и версионирования. |
   611→| 5 | **Technology** | name, version, category, cpe, known_cves, end_of_life, license | Используемая технология (framework, library, runtime). Включает CPE для корреляции с CVE. |
   612→| 6 | **Finding** | title, severity, cwe, cve, evidence, source, confidence, cvss_vector | Обнаруженная уязвимость или проблема безопасности. Core-сущность графа. |
   613→| 7 | **Evidence** | type, content, quality, source, timestamp, raw_data_hash | Доказательство, подтверждающее Finding. Может быть HTTP-запросом, скриншотом, логом. |
   614→| 8 | **Identity** | type, value, context, auth_mechanism, scope, expiry | Идентичность: пользователь, токен, cookie, API key. Используется для анализа аутентификации. |
   615→| 9 | **Secret** | type, location, exposure, rotation_policy, entropy_score, pattern_match | Обнаруженный секрет: API key, пароль, сертификат. Включает оценку энтропии. |
   616→| 10 | **Credential** | type, storage, access_pattern, privilege_level, mfa_enabled | Учётные данные: пара логин/пароль, SSH key, OAuth token. Связан с Identity. |
   617→| 11 | **AttackStep** | type, description, severity, prerequisites, exploit_available, complexity | Шаг в цепочке атаки. Связывается с другими AttackStep через LEADS_TO. |
   618→| 12 | **Recommendation** | title, type, priority, effort, impact, references, auto_fix_available | Рекомендация по устранению уязвимости. Связана с Finding через MITIGATED_BY. |

   630→### 9.1 Полный перечень Edge Types
   631→
   632→| # | Edge Type | Source -> Target | Направление | Вес | Семантика | Ограничения |

   634→| 1 | **USES** | Endpoint/API/Service -> Technology | OUTGOING | 0.8 | Объект использует технологию | Source: {Endpoint, API, Service, Container}; Target: Technology |
   635→| 2 | **OWNS** | Asset -> Application; Application -> Host/Service/Repository | OUTGOING | 1.0 | Владение ресурсом | Source: {Asset, Application}; Target: {Application, Host, Service, Repository} |

   649→### 9.2 Правила формирования рёбер
   650→
   651→1. **Provenance First:** Каждое ребро содержит Provenance — источник, время обнаружения, confidence обнаружившего движка
   652→2. **Weight = Confidence x SemanticWeight:** Вес ребра = confidence источника x семантический вес по умолчанию (из таблицы выше)
   653→3. **Direction Enforcement:** OUTGOING рёбра обходятся только в одном направлении; BIDIRECTIONAL — в обоих
   654→4. **Constraint Validation:** Перед созданием ребра проверяются ограничения на допустимые source/target node types; некорректные рёбра отбрасываются
   655→5. **Deduplication:** Два узла не могут иметь более одного ребра одного типа в одном направлении; при конфликте побеждает ребро с большим confidence
   656→6. **No Self-Loops:** Ребро не может связывать узел с самим собой (source_id != target_id)

   719→### 10.2 Node Storage
   720→
   721→Узлы хранятся в виде key-value отображения, где ключ — `NodeIdentity.hash`, значение — сериализованный `GraphNode`. Дополнительные индексы обеспечивают быстрый поиск по типу и свойствам.
   722→
   723→**Primary Storage:**
   724→- Формат: `Dict[str, GraphNode]` (in-memory) или column-family store (persistent)
   725→- Ключ: `NodeIdentity.hash` (SHA-256 от identity_type + value + namespace)
   726→- Значение: Полный GraphNode с metadata и provenance
   727→- Сложность чтения: O(1)
   728→- Сложность записи: O(1) + O(k) для обновления индексов (k — количество индексов)
   729→
   730→**Type Index:**
   731→- Формат: `Dict[str, Set[str]]` — node_type -> set of node_id hashes
   732→- Назначение: Быстрый поиск всех узлов определённого типа (например, все Finding)
   733→- Сложность: O(1) lookup + O(n) для перечисления всех узлов типа
   734→
   735→**Property Index:**
   736→- Формат: `Dict[str, Dict[Any, Set[str]]]` — property_name -> property_value -> set of node_id hashes
   737→- Назначение: Быстрый поиск узлов по значению свойства (например, все Finding с severity=critical)
   738→- Сложность: O(1) lookup
   739→- Ограничение: Индексируются только часто запрашиваемые свойства (severity, cwe, cve, hostname, path)
   740→
   741→### 10.3 Edge Storage
   742→
   743→Рёбра хранятся в трёх параллельных структурах для обеспечения быстрого доступа из разных контекстов.
   744→
   745→**Primary Storage:**
   746→- Формат: `Dict[str, GraphEdge]` — edge_id -> GraphEdge
   747→- Ключ: `edge_id` = SHA-256(source_id + target_id + relationship_type)
   748→- Сложность: O(1)
   749→
   750→**Adjacency List:**
   751→- Формат: `Dict[str, List[GraphEdge]]` — source_node_id -> list of outgoing edges
   752→- Назначение: Обход графа от узла к его соседям (ключевая операция для Traversal Engine)
   753→- Сложность: O(1) для получения всех исходящих рёбер узла
   754→- Оптимизация: Рёбра в adjacency list отсортированы по весу (убывание) для приоритизации обхода
   755→
   756→**Reverse Index:**
   757→- Формат: `Dict[str, List[GraphEdge]]` — target_node_id -> list of incoming edges
   758→- Назначение: Обратный обход (кто ссылается на данный узел). Необходим для построения Attack Paths в обратном направлении.
   759→- Сложность: O(1) для получения всех входящих рёбер узла
   760→
   761→**Edge Type Index:**
   762→- Формат: `Dict[RelationshipType, Set[str]]` — relationship_type -> set of edge_ids
   763→- Назначение: Быстрый поиск всех рёбер определённого типа (например, все LEADS_TO для Attack Path Builder)
   764→- Сложность: O(1) lookup
   765→
   766→### 10.4 Specialized Indexes
   767→
   768→**Temporal Index:**
   769→- Формат: `SortedDict[datetime, Set[str]]` — timestamp -> set of node/edge_ids
   770→- Назначение: Поиск объектов по временному диапазону (что изменилось за последний час?)
   771→- Сложность: O(log n) для range query
   772→
   773→**Confidence Index:**
   774→- Формат: `SortedDict[float, Set[str]]` — confidence -> set of node_ids
   775→- Назначение: Фильтрация по confidence (например, только находки с confidence >= 0.7)
   776→- Сложность: O(log n) для range query
   777→
   778→**Provenance Index:**
   779→- Формат: `Dict[str, Set[str]]` — source_name -> set of node/edge_ids
   780→- Назначение: Поиск всех объектов, полученных от конкретного источника (например, все находки от Nuclei)
   781→- Сложность: O(1) lookup

   834→### 11.2 Алгоритмы и их сложность
   835→
   836→| # | Алгоритм | Описание | Сложность (время) | Сложность (память) | Оптимальный сценарий |
   837→|---|----------|----------|--------------------|--------------------|---------------------|
   838→| 1 | **BFS** | Обход в ширину — посещает все узлы на расстоянии k перед переходом к k+1 | O(V + E) | O(V) | Поиск кратчайшего пути в невзвешенном графе; обнаружение всех узлов на фиксированной глубине |
   839→| 2 | **DFS** | Обход в глубину — идёт до максимальной глубины перед возвратом | O(V + E) | O(d), d = max depth | Поиск всех путей; построение Attack Paths; обнаружение циклов |
   840→| 3 | **Bidirectional Search** | Двунаправленный поиск — запускает BFS от source и target одновременно | O(V/2 + E/2) | O(V/2) | Поиск кратчайшего пути между двумя известными узлами; ускоряет поиск в больших графах |
   841→| 4 | **Shortest Path** | Поиск кратчайшего пути (Dijkstra для взвешенного, BFS для невзвешенного) | O((V+E) log V) | O(V) | Определение минимального количества шагов атаки от публичного эндпоинта до критического актива |
   842→| 5 | **Multi Path** | Поиск всех путей между двумя узлами с ограничениями | O(V + E) per path | O(V x P), P = number of paths | Обнаружение всех возможных цепочек атаки; ранжирование по Risk Score |
   843→| 6 | **Cycle Detection** | Обнаружение циклов в графе (DFS с colour marking) | O(V + E) | O(V) | Предотвращение бесконечного обхода; обнаружение циркулярных зависимостей в сервисах |
   844→| 7 | **Reachability** | Проверка достижимости одного узла от другого (BFS/DFS с ранним завершением) | O(V + E) worst, O(k) average | O(V) | Определение Reachability для Risk Engine; публичная достижимость уязвимости |
   845→
   846→**Обозначения:** V — количество узлов, E — количество рёбер, d — максимальная глубина, P — количество найденных путей, k — среднее расстояние между узлами.
   847→

   903→### 12.2 Snapshot
   904→
   905→Snapshot — полный снимок графа на определённый момент времени. Включает все узлы, рёбра, индексы и метаданные. Snapshot создаётся:
   906→- При завершении каждого сканирования (post-pipeline)
   907→- По расписанию (ежедневно, если были изменения)
   908→- По требованию (manual trigger через API)
   909→
   910→Snapshot содержит:
   911→- Полный набор узлов и рёбер (или delta от предыдущего snapshot)
   912→- Контрольную сумму (SHA-256 от сериализованного графа) для верификации целостности
   913→- Метаданные версии (GraphVersion)
   914→- Статистику (node_count, edge_count)
   915→
   916→### 12.3 Delta
   917→
   918→Delta — разница между двумя последовательными версиями графа. Включает:
   919→- Список добавленных узлов (with full data)
   920→- Список удалённых узлов (by node_id)
   921→- Список обновлённых узлов (with diff: old_value -> new_value)
   922→- Список добавленных/удалённых/обновлённых рёбер
   923→
   924→Delta хранится компактно: вместо полного графа — только изменения. При восстановлении: базовый snapshot + последовательность delta = текущее состояние.
   925→
   926→### 12.4 Rollback
   927→
   928→Rollback возвращает граф к предыдущей версии. Механизм:
   929→1. Определить целевую версию (target_version_id)
   930→2. Найти ближайший snapshot раньше target_version
   931→3. Применить snapshot
   932→4. Применить последовательность delta от snapshot до target_version
   933→5. Создать RollbackVersion (новая версия, указывающая на причину отката)
   934→6. Опубликовать GraphRolledBack event
   935→
   936→**Гарантии Rollback:**
   937→- Atomicity: откат выполняется как единая транзакция
   938→- No Data Loss: откат не удаляет данные, а создаёт новую версию
   939→- Audit Trail: каждый откат зафиксирован в истории версий
   940→- Reversible: откат может быть сам откачен (rollback of rollback)
   941→
   942→### 12.5 Replay
   943→
   944→Replay воспроизводит последовательность операций от одной версии к другой. Используется для:
   945→- Воспроизведения анализа на исторических данных
   946→- Тестирования новых правил корреляции на прошлых версиях графа
   947→- A/B тестирования аналитических алгоритмов
   948→
   949→Механизм: аналогичен Rollback, но целевая версия может быть любой, а не только предыдущей.
   950→
   951→### 12.6 Incremental Update
   952→
   953→Incremental Update добавляет новые данные в граф без полного перестроения. Механизм:
   954→1. Pipeline публикует артефакты через Artifact Bus
   955→2. Normalizer Filter преобразует артефакты в Graph Operations
   956→3. Graph Builder выполняет операции в транзакции
   957→4. Создаётся Delta (разница с предыдущей версией)
   958→5. Публикуется GraphIncrementalUpdate event
   959→
   960→**Преимущества инкрементального обновления:**
   961→- Нет простоя: граф доступен во время обновления
   962→- Минимальная нагрузка: обрабатываются только изменения
   963→- Детерминированность: один и тот же набор артефактов всегда даёт один и тот же результат
   964→
   965→### 12.7 Merge и Conflict Resolution
   966→
   967→Merge объединяет изменения из двух параллельных веток (например, два одновременных сканирования разных модулей). Conflict Resolution определяет, как обрабатывать конфликтующие изменения.
   968→
   969→**Стратегии разрешения конфликтов:**
   970→
   971→| Конфликт | Стратегия | Обоснование |
   972→|----------|-----------|-------------|
   973→| Одно свойство обновлено двумя источниками | Выбрать значение с большим confidence | Более достоверный источник побеждает |
   974→| Узел создан двумя источниками | Merge: создать один узел с объединёнными свойствами | Один и тот же объект безопасности |
   975→| Ребро создано дважды с разным весом | Выбрать ребро с большим весом | Более сильная связь побеждает |
   976→| Узел создан в одной ветке и удалён в другой | Сохранить узел (create wins over delete) | Безопаснее сохранить данные, чем потерять |
   977→| Свойство изменено в одной ветке и удалено в другой | Сохранить изменение (update wins over delete) | Аналогично: сохранение данных приоритетнее |

   982→
   983→### 13.1 API Methods
   984→
   985→| # | Метод | Сигнатура | Описание | Возвращает |
   986→|---|-------|-----------|----------|-----------|
   987→| 1 | **createNode** | `createNode(node_type: str, identity: NodeIdentity, metadata: NodeMetadata) -> GraphNode` | Создаёт новый узел в графе. Если узел с такой identity уже существует, обновляет metadata (merge). | GraphNode |

   993→| 7 | **traverse** | `traverse(start: NodeIdentity, algorithm: TraversalAlgorithm, params: Dict) -> Subgraph` | Обход графа от стартового узла с заданным алгоритмом и параметрами. | Subgraph |
   994→| 8 | **shortestPath** | `shortestPath(source: NodeIdentity, target: NodeIdentity, weight_attribute: str = "weight") -> List[GraphEdge]` | Поиск кратчайшего пути между двумя узлами. Использует Dijkstra для взвешенного графа. | List[GraphEdge] |

   996→| 10 | **snapshot** | `snapshot(description: str = "") -> GraphSnapshot` | Создание снимка текущего состояния графа. | GraphSnapshot |
   997→| 11 | **rollback** | `rollback(target_version_id: str) -> GraphVersion` | Откат графа к указанной версии. Создаёт новую версию с типом ROLLBACK. | GraphVersion |

   999→### 13.2 Query Flow
  1000→
  1001→```mermaid
  1002→sequenceDiagram
  1003→    participant C as Consumer
  1004→    participant API as Graph Query API
  1005→    participant CACHE as Cache Layer
  1006→    participant IDX as Index Manager
  1007→    participant NS as Node Store
  1008→    participant ES as Edge Store
  1009→
  1010→    C->>API: query(pattern, filters, limit, offset)
  1011→    API->>CACHE: check cache(pattern_hash)
  1012→    alt Cache Hit
  1013→        CACHE-->>API: cached result
  1014→        API-->>C: List[GraphNode]
  1015→    else Cache Miss
  1016→        API->>IDX: lookup indexes(filters)
  1017→        IDX-->>API: candidate node_ids
  1018→        API->>NS: fetch nodes(candidate_ids)
  1019→        NS-->>API: List[GraphNode]
  1020→        API->>API: apply remaining filters
  1021→        API->>CACHE: store result(pattern_hash, result)
  1022→        API-->>C: List[GraphNode]
  1023→    end
  1024→
  1025→    C->>API: traverse(start, BFS, {max_depth: 5})
  1026→    API->>CACHE: check cache(traversal_hash)
  1027→    alt Cache Hit
  1028→        CACHE-->>API: cached Subgraph
  1029→        API-->>C: Subgraph
  1030→    else Cache Miss
  1031→        API->>ES: get_adjacent(start)
  1032→        ES-->>API: List[GraphEdge]
  1033→        loop For each edge
  1034→            API->>NS: fetch node(edge.target)
  1035→            NS-->>API: GraphNode
  1036→            API->>ES: get_adjacent(node)
  1037→            ES-->>API: List[GraphEdge]
  1038→        end
  1039→        API->>API: build Subgraph
  1040→        API->>CACHE: store result
  1041→        API-->>C: Subgraph
  1042→    end
  1043→```
  1044→
  1045→### 13.3 Контрактные гарантии API
  1046→
  1047→| Гарантия | Описание |
  1048→|----------|----------|
  1049→| **Idempotency** | createNode с одинаковой identity — idempotent (merge metadata). createEdge с одинаковым source/target/type — idempotent (update weight). |
  1050→| **Atomicity** | Все операции внутри Transaction — атомарны. Либо все выполняются, либо ни одна. |
  1051→| **Consistency** | После commit транзакции граф находится в согласованном состоянии (все индексы обновлены). |
  1052→| **Isolation** | Параллельные транзакции не видят промежуточных результатов друг друга. |
  1053→| **Durability** | После commit транзакции результат сохранён в Snapshot/Delta и переживает перезапуск. |
  1054→| **Determinism** | Один и тот же запрос на одном и том же графе всегда даёт один и тот же результат. |

  1110→### 14.2 Pipeline -> Graph
  1111→
  1112→Артефакты из Scan Pipeline поступают через Artifact Bus и преобразуются в Graph Operations:
  1113→
  1114→1. **ArtifactPublished** событие из Artifact Bus
  1115→2. **Normalizer Filter** определяет тип артефакта и извлекает данные для создания узлов/рёбер
  1116→3. **Graph Builder** создаёт GraphTransaction с набором операций
  1117→4. **Commit** транзакции: выполняются все операции, обновляются индексы, создаётся GraphVersion
  1118→5. **Delta** вычисляется и сохраняется
  1119→6. **Graph Events** публикуются в Graph Event Bus
  1120→
  1121→### 14.3 Graph -> Intelligence
  1122→
  1123→SIE подписывается на Graph Events для инкрементального анализа:
  1124→
  1125→| Событие | Действие SIE |
  1126→|---------|-------------|
  1127→| NodeCreated (type=Finding) | Найти корреляции с существующими находками |
  1128→| NodeUpdated (type=Finding) | Пересчитать Risk Score для обновлённой находки |
  1129→| EdgeCreated (type=LEADS_TO) | Перестроить Attack Path с новым ребром |
  1130→| EdgeCreated (type=EXPOSES) | Пересчитать Exposure для затронутого эндпоинта |
  1131→| SnapshotCreated | Запустить полный анализ (batch mode) |
  1132→
  1133→### 14.4 Graph -> Explainability
  1134→
  1135→Explainability Engine подписывается на Graph Events для обновления объяснений:
  1136→
  1137→| Событие | Действие EE |
  1138→|---------|------------|
  1139→| NodeCreated (type=Finding) | Создать начальное объяснение для находки |
  1140→| EdgeCreated (type=LEADS_TO) | Обновить объяснение Attack Path |
  1141→| SnapshotCreated | Пересоздать объяснения для всего графа |
  1142→| GraphRolledBack | Откатить объяснения к предыдущей версии |

  1144→### 14.5 Event Bus Details
  1145→
  1146→**Graph Event Bus** — внутренняя событийная шина Graph Platform. Отличается от Artifact Bus:
  1147→- Artifact Bus — шина артефактов сканирования (между Pipeline компонентами)
  1148→- Graph Event Bus — ша графовых событий (между Graph Platform и потребителями)
  1149→
  1150→**Типы событий:**
  1151→
  1152→| Тип | Payload | Частота |
  1153→|-----|---------|---------|
  1154→| NodeCreated | node_id, node_type, version | Высокая |
  1155→| NodeUpdated | node_id, changed_properties, old_values, new_values | Средняя |
  1156→| NodeRemoved | node_id, node_type | Низкая |
  1157→| EdgeCreated | edge_id, source_id, target_id, relationship_type | Высокая |
  1158→| EdgeRemoved | edge_id | Низкая |
  1159→| SnapshotCreated | snapshot_id, version, node_count, edge_count | Низкая |
  1160→| GraphVersionChanged | version_id, operation, description | Средняя |
  1161→| GraphRolledBack | version_id, target_version, reason | Редкая |
  1162→
  1163→### 14.6 Incremental Updates
  1164→
  1165→Incremental Updates — ключевой механизм синхронизации. Вместо полного перестроения графа при каждом сканировании, обновляются только изменённые узлы и рёбра.
  1166→
  1167→**Алгоритм Incremental Update:**
  1168→
  1169→1. Получить артефакты из Pipeline
  1170→2. Для каждого артефакта определить затронутые NodeIdentities
  1171→3. Запросить существующие узлы по identity (Graph Query API)
  1172→4. Если узел существует — вычислить diff и создать updateNode операцию
  1173→5. Если узел не существует — создать createNode операцию
  1174→6. Аналогично для рёбер
  1175→7. Выполнить все операции в одной транзакции
  1176→8. Опубликовать GraphEvents
  1177→
  1178→**Преимущества:**
  1179→- Минимальная нагрузка на Storage (только изменения)
  1180→- Потребители получают только релевантные события
  1181→- Детерминированность: результат не зависит от порядка операций (благодаря merge semantics)

  1185→## 15. Performance Strategy
  1186→
  1187→### 15.1 Caching
  1188→
  1189→**Стратегия кэширования:** Multi-level cache с event-driven invalidation.
  1190→
  1191→| Уровень | Содержимое | TTL | Invalidation |
  1192→|---------|-----------|-----|-------------|
  1193→| L1: Query Cache | Результаты query() по pattern_hash | 5 мин | GraphEvent (NodeCreated, NodeUpdated, EdgeCreated, EdgeRemoved) |
  1194→| L2: Traversal Cache | Результаты traverse() по traversal_hash | 10 мин | GraphEvent (любое изменение в затронутом подграфе) |
  1195→| L3: Node Cache | Отдельные GraphNode по node_id | 30 мин | GraphEvent (NodeUpdated, NodeRemoved) |
  1196→| L4: Snapshot Cache | GraphSnapshot по snapshot_id | Без TTL | Никогда не инвалидируется (snapshots immutable) |
  1197→
  1198→**Политика инвалидации:** Event-driven (приоритет) + TTL (fallback). При получении GraphEvent определяются затронутые cache keys и инвалидируются. TTL — страховка на случай пропущенных событий.
  1199→
  1200→### 15.2 Indexing
  1201→
  1202→Стратегия индексирования определена в разделе Graph Storage Model (раздел 10). Дополнительные принципы:
  1203→
  1204→- **Lazy Index Building:** Индексы строятся при первом обращении, а не при старте
  1205→- **Incremental Index Update:** Индексы обновляются инкрементально при каждой транзакции
  1206→- **Index Selection:** Query Planner выбирает оптимальный индекс на основе паттерна запроса
  1207→- **Index Compression:** Для больших индексов применяется compression (run-length encoding для sorted lists)
  1208→
  1209→### 15.3 Lazy Loading
  1210→
  1211→Узлы и рёбра загружаются по требованию, а не целиком:
  1212→
  1213→- **Traversal:** Загружаются только узлы и рёбра, посещённые алгоритмом обхода
  1214→- **Query:** Загружаются только узлы, удовлетворяющие фильтрам (через индекс)
  1215→- **Subgraph:** Формируется из уже загруженных данных; при необходимости загружаются недостающие узлы
  1216→
  1217→Lazy Loading реализуется через proxy-объекты: NodeProxy содержит только node_id и загружает полные данные при первом обращении к metadata.
  1218→
  1219→### 15.4 Graph Partitioning
  1220→
  1221→Для горизонтального масштабирования граф разбивается на партиции:
  1222→
  1223→| Стратегия | Ключ партиции | Описание |
  1224→|-----------|--------------|----------|
  1225→| Application-scoped | Application node_id | Каждое приложение — отдельная партиция |
  1226→| Host-scoped | Host node_id | Все объекты одного хоста в одной партиции |
  1227→| Time-scoped | Timestamp range | Партиции по временным диапазонам |
  1228→
  1229→**Рекомендуемая стратегия:** Application-scoped (по умолчанию). Большинство запросов работают в контексте одного приложения, поэтому партиционирование по Application минимизирует cross-partition queries.
  1230→
  1231→### 15.5 Memory Limits
  1232→
  1233→| Параметр | Значение | Описание |
  1234→|----------|---------|----------|
  1235→| Max Graph Size (in-memory) | 500K nodes, 1.5M edges | При превышении — spill to disk |
  1236→| Max Node Size | 10 KB | Serialized GraphNode без raw evidence |
  1237→| Max Edge Size | 1 KB | Serialized GraphEdge |
  1238→| Max Traversal Depth | 20 | Защита от stack overflow |
  1239→| Max Query Results | 10,000 | Защита от memory exhaustion |
  1240→| Max Snapshot Size | 100 MB | При превышении — delta-only snapshots |
  1241→
  1242→### 15.6 Large Graph Strategy
  1243→
  1244→При превышении лимитов in-memory хранения:
  1245→
  1246→1. **Hot/Cold Split:** Узлы, посещённые за последние N запросов — hot (in-memory); остальные — cold (on-disk)
  1247→2. **Subgraph Materialization:** Для часто используемых подграфов (Attack Paths, Asset graphs) создаются материализованные подграфы
  1248→3. **Incremental Traversal:** Обход выполняется партициями, а не целиком; результаты объединяются
  1249→4. **Approximate Algorithms:** Для графов > 100K узлов — переключение на приближённые алгоритмы (sampling-based BFS/DFS)

  1253→## 16. Security
  1254→
  1255→### 16.1 Immutable Graph
  1256→
  1257→Все узлы и рёбра неизменяемы после создания (immutable). Обновление создаёт новую версию, а не модифицирует существующую. Это обеспечивает:
  1258→
  1259→- **Audit Trail:** Каждое изменение зафиксировано и может быть отслежено
  1260→- **Reproducibility:** Любая версия графа может быть восстановлена для воспроизведения анализа
  1261→- **Thread Safety:** Immutable объекты безопасны для параллельного чтения
  1262→- **Integrity:** Невозможно подделать историю изменений
  1263→
  1264→### 16.2 Audit Trail
  1265→
  1266→Каждая операция над графом логируется:
  1267→
  1268→| Поле лога | Описание |
  1269→|-----------|----------|
  1270→| timestamp | Время операции |
  1271→| operation_type | create_node, update_node, create_edge и т.д. |
  1272→| actor | Кто выполнил операцию (system, pipeline:nuclei, manual:user_id) |
  1273→| target_id | ID затронутого объекта |
  1274→| before | Состояние до операции (для update) |
  1275→| after | Состояние после операции |
  1276→| version_id | Версия графа |
  1277→| transaction_id | ID транзакции |
  1278→
  1279→Audit Log неизменяем и хранится отдельно от графа (append-only).
  1280→
  1281→### 16.3 Graph Integrity
  1282→
  1283→**Целостность графа обеспечивается следующими инвариантами:**
  1284→
  1285→1. **Referential Integrity:** Каждое ребро ссылается на существующие узлы (source и target)
  1286→2. **No Orphan Nodes:** Каждый узел имеет хотя бы одно ребро (кроме корневого Application)
  1287→3. **No Duplicate Edges:** Не может быть двух рёбер одного типа между одной парой узлов в одном направлении
  1288→4. **No Self-Loops:** Ребро не может связывать узел с самим собой
  1289→5. **Type Constraints:** Source и target узлы ребра соответствуют ограничениям RelationshipType
  1290→6. **Weight Range:** Вес ребра в диапазоне [0.0, 1.0]
  1291→7. **Confidence Range:** Confidence узла и ребра в диапазоне [0.0, 1.0]
  1292→
  1293→При нарушении любого инварианта транзакция отклоняется (rollback).
  1294→
  1295→### 16.4 Node Validation
  1296→
  1297→Перед созданием или обновлением узла выполняется валидация:
  1298→
  1299→- **Type Validation:** node_type входит в разрешённый перечень (18 типов)
  1300→- **Identity Validation:** NodeIdentity содержит обязательные поля (identity_type, value, namespace)
  1301→- **Metadata Validation:** Metadata содержит обязательные свойства для данного node_type
  1302→- **Confidence Validation:** confidence в диапазоне [0.0, 1.0]
  1303→- **Provenance Validation:** Provenance содержит источник и timestamp
  1304→
  1305→### 16.5 Edge Validation
  1306→
  1307→Перед созданием ребра выполняется валидация:
  1308→
  1309→- **Existence Validation:** Source и target узлы существуют в графе
  1310→- **Type Compatibility:** Source и target node types соответствуют constraints RelationshipType
  1311→- **Uniqueness Validation:** Ребро с таким source/target/type не существует (или будет обновлено)
  1312→- **Direction Validation:** Направление ребра соответствует semantics RelationshipType
  1313→- **Weight Validation:** Вес в диапазоне [0.0, 1.0]
  1314→- **No Self-Loop:** source_id != target_id
  1315→
  1316→### 16.6 Serialization Safety
  1317→
  1318→Сериализация графа (для экспорта, снапшотов, передачи по сети) должна быть безопасной:
  1319→
  1320→- **No Raw Secrets:** При сериализации Secret и Credential узлов значения маскируются
  1321→- **No Raw Evidence:** Поле raw_data в Evidence исключается из сериализации (используется hash)
  1322→- **Deterministic Serialization:** Порядок полей детерминирован (alphabetical key order) для воспроизводимости checksum
  1323→- **Size Limits:** Максимальный размер сериализованного узла — 10 KB; при превышении — truncation с предупреждением
  1324→- **Format Validation:** При десериализации проверяется соответствие формату (schema validation)

  1328→## 17. Failure Recovery
  1329→
  1330→### 17.1 Failure Scenarios
  1331→
  1332→| # | Сценарий | Обнаружение | Восстановление |
  1333→|---|----------|------------|----------------|
  1334→| 1 | **Partial Transaction Failure** | Операция в транзакции выбросила исключение | Rollback всей транзакции; все partial changes отменены |
  1335→| 2 | **Index Corruption** | Checksum mismatch при чтении индекса | Rebuild index из primary storage; log warning |
  1336→| 3 | **Snapshot Corruption** | Checksum mismatch при загрузке snapshot | Восстановление из предыдущего snapshot + delta chain |
  1337→| 4 | **Cache Stale** | TTL expired или event missed | Event-driven invalidation + TTL fallback; при сомнении — cache miss |
  1338→| 5 | **Storage Full** | Write operation возвращает StorageFull error | Switch to delta-only mode (не создавать full snapshots); alert ops |
  1339→| 6 | **OOM during Traversal** | Memory limit exceeded | Incremental traversal с пагинацией; spill to disk |
  1340→| 7 | **Concurrent Write Conflict** | Два Graph Builder одновременно обновляют один узел | Optimistic locking (version check); при конфликте — merge по правилам из 12.7 |
  1341→| 8 | **Consumer Lag** | Потребитель не обрабатывает GraphEvents вовремя | Event buffer (in-memory ring buffer, 10K events); при переполнении — snapshot-based resync |
  1342→
  1343→### 17.2 Recovery Guarantees
  1344→
  1345→| Гарантия | Уровень |
  1346→|----------|---------|
  1347→| **No Data Loss** | Все зафиксированные транзакции сохранены в Snapshot/Delta |
  1348→| **No Silent Corruption** | Checksum verification при загрузке snapshot; integrity checks при каждой транзакции |
  1349→| **Recoverability** | Любая версия графа может быть восстановлена из snapshot + delta chain |
  1350→| **Consistency** | После recovery граф находится в согласованном состоянии (все индексы актуальны) |
  1351→| **Availability** | Чтение доступно во время recovery (через кэш или последний snapshot) |

  1383→## 19. Architecture Decision Records
  1384→
  1385→### ADR-KG-001: Почему выбран Property Graph
  1386→
  1387→**Контекст:** Необходимо выбрать модель данных для Knowledge Graph. Основные варианты: Property Graph, RDF/SPARQL, Hypergraph.
  1388→
  1389→**Решение:** Использовать Property Graph модель.
  1390→
  1391→**Обоснование:**
  1392→- Property Graph — наиболее интуитивная модель для предметной области безопасности: узлы (Application, Host, Finding) с properties (severity, confidence) и направленные рёбра (EXPOSES, LEADS_TO) с весами
  1393→- RDF/SPARQL избыточен для наших задач: не нужна семантическая веб-совместимость, а SPARQL сложнее в реализации и оптимизации
  1394→- Hypergraph (рёбра связывают более двух узлов) добавляет сложность без очевидной пользы: все наши связи бинарные (один объект воздействует на другой)
  1395→- Property Graph поддерживается всеми основными графовыми СУБД (Neo4j, ArangoDB, Amazon Neptune), обеспечивая portability
  1396→- Серийная сериализация Property Graph проще и эффективнее, чем RDF triples
  1397→
  1398→**Последствия:** Все узлы и рёбра имеют properties (key-value pairs). Рёбра направленные и взвешенные. Модель не поддерживает n-арные связи напрямую (решается через промежуточные узлы).
  1399→
  1400→---

  1402→### ADR-KG-002: Почему Graph Immutable
  1403→
  1404→**Контекст:** Необходимо определить, могут ли узлы и рёбра изменяться после создания.
  1405→
  1406→**Решение:** Все узлы и рёбра immutable. Обновление создаёт новую версию.
  1407→
  1408→**Обоснование:**
  1409→- CTO Decision #7 (Immutable Domain Models) прямо требует неизменяемости всех доменных моделей
  1410→- Immutable graph обеспечивает полную воспроизводимость анализа: один и тот же граф всегда даёт один и тот же результат
  1411→- Audit trail естественным образом формируется из истории версий — не нужен отдельный механизм логирования изменений
  1412→- Thread safety: immutable объекты безопасны для параллельного чтения без блокировок
  1413→- Immutable graph упрощает кэширование: нет необходимости в cache invalidation при чтении — данные не меняются
  1414→- Rollback и Replay тривиальны: достаточно восстановить нужную версию
  1415→
  1416→**Последствия:** Рост объёма данных (каждое обновление создаёт новую версию). Необходима Retention Policy для очистки старых версий. Delta snapshots минимизируют overhead.
  1417→

  1420→### ADR-KG-003: Почему Snapshot-based Versioning
  1421→
  1422→**Контекст:** Необходимо выбрать стратегию версионирования графа. Варианты: Event Sourcing (воспроизведение всех событий), Snapshot-based (периодические снимки), Hybrid.
  1423→
  1424→**Решение:** Hybrid: Snapshot + Delta. Периодические полные снимки + инкрементальные дельты между снимками.
  1425→
  1426→**Обоснование:**
  1427→- Event Sourcing чист, но медленный при восстановлении: для графа с 10K изменений воспроизведение всех событий занимает неприемлемо долго
  1428→- Snapshot-only быстро для восстановления, но создаёт избыточную нагрузку при сохранении полного графа при каждом изменении
  1429→- Hybrid (Snapshot + Delta) оптимален: полный snapshot создаётся редко (post-pipeline), а delta — при каждом инкрементальном обновлении
  1430→- Восстановление: загрузить ближайший snapshot + применить delta chain = текущее состояние
  1431→- Сложность восстановления: O(snapshot_size + sum(delta_sizes)), что значительно меньше O(all_events) при Event Sourcing
  1432→
  1433→**Последствия:** Необходимость управления lifecycle snapshots (retention, cleanup). Delta chain не должен быть слишком длинным (макс. 50 delta перед обязательным snapshot). Контрольная сумма snapshot для верификации целостности.
  1434→

  1437→### ADR-KG-004: Почему Traversal Engine отделён от Storage
  1438→
  1439→**Контекст:** Traversal может быть реализован как часть Storage Layer (native graph DB traversal) или как отдельный компонент.
  1440→
  1441→**Решение:** Traversal Engine — отдельный компонент, не зависящий от Storage.
  1442→
  1443→**Обоснование:**
  1444→- Storage Agnostic: если Traversal привязан к конкретной СУБД (например, Cypher для Neo4j), замена бэкенда потребует переписывания всего аналитического слоя
  1445→- Determinism: собственный Traversal Engine гарантирует детерминированность результатов независимо от оптимизаций конкретной СУБД
  1446→- Testability: Traversal Engine можно тестировать изолированно, с mock-графами
  1447→- Extensibility: новые алгоритмы обхода добавляются в Traversal Engine без изменения Storage
  1448→- Performance Control: собственный кэш на уровне Traversal Engine, независимый от Storage caching
  1449→
  1450→**Последствия:** Traversal Engine использует Graph Query API для доступа к данным, что добавляет один уровень абстракции. Native traversal СУБД может быть быстрее, но мы приоритизируем portability и determinism над raw performance.
  1451→

  1454→### ADR-KG-005: Почему Query API не зависит от Storage
  1455→
  1456→**Контекст:** Graph Query API может напрямую обращаться к конкретной СУБД или быть абстрагирован через интерфейс.
  1457→
  1458→**Решение:** Graph Query API определён через интерфейс (AbstractGraphStorage), конкретная реализация предоставляется через dependency injection.
  1459→
  1460→**Обоснование:**
  1461→- CTO Decision #4 (Plugin Architecture) требует, чтобы ядро не зависело от конкретных реализаций
  1462→- Замена NetworkX на Neo4j должна быть прозрачной для SIE, Explainability Engine и других потребителей
  1463→- Testing: можно создать InMemoryGraphStorage для unit-тестов без необходимости поднимать СУБД
  1464→- Migration path: при необходимости смены бэкенда — реализация нового Storage adapter, без изменения API
  1465→- Different deployment scenarios: in-memory для small graphs, Neo4j для production, ArangoDB для multi-model needs
  1466→
  1467→**Последствия:** Все операции графа проходят через абстрактный интерфейс. Невозможно использовать специфичные возможности конкретной СУБД (например, Cypher-запросы). Performance overhead от абстракции минимален (thin wrapper).

  1471→### ADR-KG-006: Почему Graph Event Bus
  1472→
  1473→**Контекст:** Потребители графа (SIE, EE, Dashboard) нуждаются в уведомлениях об изменениях. Варианты: polling, shared callback, event bus.
  1474→
  1475→**Решение:** Отдельная Graph Event Bus для графовых событий, интегрированная с Artifact Bus на уровне Platform.
  1476→
  1477→**Обоснование:**
  1478→- Polling создаёт избыточную нагрузку и задержку (SIE опрашивает граф каждые N секунд)
  1479→- Shared callback нарушает Loose Coupling (Graph знает о каждом потребителе)
  1480→- Event Bus — стандартный паттерн для decoupled communication, уже принятый платформой (CTO Decision #5)
  1481→- Graph Event Bus изолирована от Artifact Bus: разные типы событий, разные потребители, разные гарантии доставки
  1482→- Поддержка Incremental Synchronization: потребитель получает только релевантные события, а не весь граф
  1483→- Eventual Consistency: потребители обновляются асинхронно, без блокировки Graph Platform
  1484→
  1485→**Последствия:** Потребители должны обрабатывать события идемпотентно (событие может быть доставлено более одного раза). Необходим buffer для consumer lag. При потере событий — snapshot-based resync.
  1486→

  1489→### ADR-KG-007: Почему Delta Synchronization
  1490→
  1491→**Контекст:** При обновлении графа потребители могут получить полный граф (full sync) или только изменения (delta sync).
  1492→
  1493→**Решение:** Delta Synchronization по умолчанию; Full Sync только при SnapshotCreated или по запросу.
  1494→
  1495→**Обоснование:**
  1496→- Полный граф типичного приложения содержит 50K+ узлов и 150K+ рёбер. Передача полного графа при каждом обновлении — недопустимый overhead
  1497→- Delta sync передаёт только изменения:新增/изменённые/удалённые узлы и рёбра. Типичный delta — 10-100 объектов, а не 50K+
  1498→- SIE работает в incremental mode: при появлении новой находки пересчитать Risk Score только для неё, а не для всего графа
  1499→- Delta sync естественно ложится на Graph Event Bus: каждое событие — это delta
  1500→- Full sync необходим только при: (1) первом подключении потребителя, (2) SnapshotCreated, (3) потере событий
  1501→
  1502→**Последствия:** Потребитель должен поддерживать локальное состояние (cache графа) и применять delta к нему. При расхождении — запросить full sync через snapshot().
  1503→

  1505→
  1506→### ADR-KG-008: Почему Explainability использует Graph Snapshot
  1507→
  1508→**Контекст:** Explainability Engine нуждается в стабильных данных для формирования объяснений. Если граф изменится в процессе формирования объяснения, результат будет неконсистентным.
  1509→
  1510→**Решение:** Explainability Engine работает с Graph Snapshot, а не с live graph.
  1511→
  1512→**Обоснование:**
  1513→- Snapshot immutable — данные не изменятся в процессе формирования объяснения
  1514→- Explainability требует воспроизводимости: одно и то же объяснение должно быть идентичным при повторном запросе
  1515→- Snapshot привязан к конкретной версии — объяснение можно соотнести с версией графа
  1516→- Live graph может измениться между началом и концом формирования объяснения, что приведёт к неконсистентному результату
  1517→- Snapshot-based объяснения поддерживают Audit: можно пересоздать объяснение для любой исторической версии графа
  1518→
  1519→**Последствия:** Explainability Engine может работать с немного устаревшими данными (snapshot создаётся post-pipeline, а не real-time). Задержка: от нескольких секунд до нескольких минут. Это допустимо, т.к. объяснения не требуют real-time обновления.

  1523→## 20. Open Questions
  1524→
  1525→1. **Retention Policy:** Сколько версий графа хранить? Предложение: последние 30 snapshots + все delta за последние 7 дней. Требует валидации с реальными данными.
  1526→
  1527→2. **Snapshot Frequency:** Как часто создавать полные snapshot? Предложение: после каждого pipeline run. Альтернатива: по расписанию (ежечасно). Требует load testing.
  1528→
  1529→3. **Max Graph Size:** При каком размере графа необходимо переключаться на distributed storage? Предложение: 500K узлов (in-memory threshold). Требует benchmarking.
  1530→
  1531→4. **Event Delivery Guarantees:** At-least-once или exactly-once? Предложение: at-least-once с idempotent consumers. Exactly-once требует distributed transactions и значительно усложняет архитектуру.
  1532→
  1533→5. **Cross-Application Queries:** Должен ли Graph Query API поддерживать запросы, объединяющие узлы из разных приложений? Пока нет — но архитектура закладывает extension point.
  1534→
  1535→6. **Graph Schema Evolution:** Как добавлять новые Node Types и Edge Types без нарушения существующих данных? Предложение: schema-less (каждый узел содержит свой тип), но с обязательной валидацией при создании.
  1536→
  1537→7. **Storage Backend Selection:** NetworkX (in-memory, Python-native) или Neo4j (persistent, Cypher) для INT-001? Зависит от performance testing и deployment requirements.
  1538→
  1539→8. **Concurrent Write Handling:** Optimistic locking (version check) или Pessimistic locking (write lock)? Предложение: Optimistic — ожидается низкая конкуренция (pipeline writes sequentially).
